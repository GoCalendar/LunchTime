# PRD-01. LunchTime MVP

> 상태: **정본 초안**
>
> 대상 플랫폼: macOS
>
> 기준 논의: [10. 결정 및 미결정 목록](../product-definition/10_decision_backlog.md)과 2026-07-24 제품 회의
>
> 역할: 첫 MVP에서 제공할 제품 경험과 구현 수용 조건을 정의합니다.

이 문서는 회의에서 확정한 제품 동작을 우선 반영합니다. 기존 `product-definition/` 문서와 충돌하는 경우 이 PRD의 확정 요구사항을 기준으로 하며, 세부 동작은 [POL-01 일일·방 생명주기](../policies/01_daily_room_lifecycle.md), [POL-02 복제·정합성·보존](../policies/02_replication_consistency_retention.md), [POL-03 보안·신뢰 경계](../policies/03_security_and_trust.md), [POL-04 macOS 화면·채팅](../policies/04_surfaces_and_chat.md)에서 구체화합니다.

## 1. 제품 목표

LunchTime은 회사 구성원이 점심 가게별 모집을 발견하고, 참여하고, 자신의 메뉴를 빠짐없이 전달하며, 주문 상태를 확인할 수 있게 하는 macOS 앱입니다.

자유로운 대화는 유지하되 주문에 필요한 가게, 참여자, 개인 메뉴, 마감과 주문 상태는 구조화합니다. 별도 중앙 서버를 운영하지 않고 회사 내부망의 Peer들이 각자 보유한 데이터를 교환해 현재 상태를 복구합니다.

### 가장 중요한 실패

참여자가 메뉴를 제출했는데 다른 참여자, 특히 실제 주문하는 사람이 이를 인식하지 못해 주문에서 누락되는 상황을 최우선으로 방지합니다.

완전한 중앙 정본이 없는 구조에서는 모든 네트워크 단절 상황의 무손실을 보장할 수 없습니다. 따라서 MVP는 다음 원칙을 지켜야 합니다.

1. 메뉴 변경을 로컬에 반영했다는 사실과 다른 Peer에 전달됐다는 사실을 구분합니다.
2. 아직 공유되지 않았거나 최신 여부를 확인할 수 없는 메뉴를 성공한 것처럼 표시하지 않습니다.
3. 주문 전에 참여자와 메뉴를 다시 동기화하고, 누락 가능성이 있는 상태를 명확히 알립니다.
4. 잠자기·네트워크 단절에서 돌아오면 자동으로 차이를 수집하고 수동 새로고침도 제공합니다.

### 제품 가치 신호

- Slack을 다시 탐색하지 않고 오늘의 모집과 주문 상태를 파악할 수 있습니다.
- 참여자가 제출한 개인 메뉴의 누락과 재확인 횟수가 줄어듭니다.
- 생성부터 참여, 메뉴 제출, 주문 상태 확인과 14:30 종료까지의 흐름이 완주됩니다.
- 과거 가게 기록을 이용해 반복 모집을 더 빠르게 시작합니다.
- 지원 네트워크 안에서 동기화 실패를 인지하고 복구할 수 있습니다.

anti-entropy의 `마지막 성공 대조 후 다음 대조 시작 간격 30초 이하` 안전 목표를 제외한 정량 목표와 파일럿 규모는 기술 스파이크 및 현재 업무 기준선 측정 후 확정합니다.

## 3. MVP에서 제외하는 것

- Windows 클라이언트
- CLI 인터페이스(D-23, C-06, F-60)
- 가게 후보 목록, 투표와 투표를 통한 최종 가게 확정(F-29~F-32)
- 가게명과 별개인 사용자 정의 방 제목·모집 메뉴·메모(F-10)
- QR 이외의 AirDrop, 시스템 공유 시트 등 추가 휴대폰 전달 방식(F-27)
- 배민의 비공개 API 또는 스크래핑을 이용한 메뉴 조회(D-12)
- LunchTime 안에서의 실제 주문, 결제와 배달 상태 추적
- 주문 완료 이후의 `출발` 상태
- 새 방, 주문 마감과 주문 완료 시스템 알림
- 11:00 전사 점심 선택 알림(D-25, F-74)
- MenuBar의 라운지 바로가기, 라운지 읽지 않음과 메시지 알림(D-24, F-48)
- Mac이 잠자기 또는 전원 꺼짐 상태일 때의 실시간 통신·알림 보장
- 중앙 서버를 전제로 한 계정, 푸시 알림과 영구 백업
- 같은 사람의 여러 Mac을 하나의 사용자 권한으로 연결하거나 기기 사이에 권한을 이전하는 계정 기능
- 배민 링크 형식·도메인 검증과 잘못된 링크 경고
- 지원 사내 Wi-Fi 내부의 악의적인 LunchTime Peer 탐지·격리
- 취소됐거나 최종화가 불완전한 Room을 성공 가게 히스토리에 반영하는 동작
- 채팅의 영구 보존, 무손실 전달, 읽음 확인과 모든 Peer 수신 보장
- Peer 수동 승인·신뢰 관리 UI와 연결 화면의 변경별 ACK 상세

## 4. 용어와 핵심 모델

| 용어 | 정의 |
|------|------|
| 운영일 | `Asia/Seoul` 기준 한 날짜의 11:00~14:30 점심 운영 구간. 쓰기는 14:30에 종료합니다. |
| 점심방(Room) | 가게 하나와 그 배민 링크를 중심으로 참여자, 개인 메뉴, 주문 상태와 채팅을 묶는 공간 |
| 참여 | 사용자가 해당 가게의 공동 주문에 포함되겠다는 의사 표시. 요청 기기가 마감 전에 유효한 참여 수락 ACK를 받아야 확정됩니다. |
| 주문 마감 시각 | 방 생성 시 11:00 이상 14:30 미만에서 정하는 신규 참여 종료 시각 |
| 주문 완료 | 실제 배민 주문이 접수됐음을 참여자가 표시한 가역 상태 |
| 쓰기 종료 | 14:30에 일어나며 신규 이벤트 작성을 막는 비가역 일일 경계 |
| 최종화 | 활성 기기에서는 14:30부터 14:32까지, 경계에 잠든 기기에서는 다음 실행·깨우기부터 최대 120초 동안 14:30 이전 이벤트를 대조해 종료 스냅샷을 만드는 과정 |
| 취소된 Room | 생성자 취소 또는 마지막 참여자 철회로 종료되어 최근 Room 상세와 가게 히스토리에서 제외되는 Room |
| 라운지 | 특정 점심방과 무관한 잡담을 위한 별도 채팅 공간 |
| 운영 데이터 | 방, 참여, 메뉴, 상태와 이를 재구성하는 Peer 동기화 데이터 |
| 가게 히스토리 | 반복 모집을 위해 로컬에 장기 보존하는 가게명, 배민 링크와 최근 이용 정보 |
| 최근 Room 상세 | 취소되지 않은 Room의 최종 또는 `종료 · 동기화 불완전` 스냅샷을 자동 신뢰된 사내망 Peer가 최대 14일 동안 보는 열람 전용 기록 |
| 종료 · 동기화 불완전 | 최대 120초 finalization 안에 필요한 대조를 끝내지 못한 사용자 표시 상태. 열람과 후속 제한 동기화는 허용하지만 성공 히스토리는 만들지 않습니다. |
| 인식된 Peer | 지원 사내 Wi-Fi에서 발견되어 자동 신뢰되고 데이터를 읽고 교환할 수 있는 LunchTime Mac 클라이언트 |
| 정상 응답 Peer | 현재 발견 가능하며 제한 시간 안에 유효한 동기화 응답을 반환하는 인식된 Peer |
| 사용자 ID | 설치 시 생성해 기기 키에 결합하는 불투명 권한 식별자. MVP에서는 설치·기기 하나와 1:1이며 닉네임과 구분합니다. |
| 닉네임 | Peer 목록과 참여자 화면에 보이는 이름. 중복을 허용하고 권한 판정에는 사용하지 않습니다. |

참여자가 한 명뿐인 방도 유효하며 주문 완료, 14:30 쓰기 종료와 최종화까지 정상적으로 진행할 수 있어야 합니다. `한 사용자`와 본인 소유권은 닉네임이 아니라 사용자 ID로 판정합니다. 한 사용자 ID는 같은 운영일에 하나의 방에만 참여할 수 있으며, 주문 상태가 `진행 중`일 때 현재 방에서 참여를 철회한 뒤 다른 방으로 이동하거나 새 방을 만들 수 있습니다.

## 5. 사용자 경험

### 5.1 MenuBar

MenuBar는 오늘 상황을 빠르게 확인하고 앱 창을 여는 진입점입니다.

MVP에서 각 활성 방에 다음 정보를 표시합니다.

- 가게명
- 참여자 수
- 참여 모집 중, 주문 마감과 주문 완료를 구분하는 상태 아이콘
- 선택한 방을 별도 앱 창에서 여는 동작
- 새 방을 만들거나 전체 앱 창을 여는 동작
- 마지막 동기화 시각과 주의가 필요한 연결 상태

채팅, 긴 메뉴 입력과 히스토리 관리는 MenuBar 안에서 수행하지 않습니다. 당일 Room은 14:30에 MenuBar에서 제거합니다.

### 5.2 앱 창

별도 앱 창은 다음 영역을 제공합니다.

- 오늘의 점심방 목록과 선택한 점심방 상세
- 새 점심방 생성
- 참여자와 개인 메뉴 목록
- 방별 채팅
- 배민 링크 QR
- 주문 상태 변경
- 히스토리와 재모집
- 최근 14일 Room 상세 열람
- 라운지
- 연결·동기화 상태 및 수동 새로고침
- 연결된 Peer 목록

### 5.3 QR 표시와 휴대폰 스캔

입력 링크가 등록된 방은 QR 코드를 표시합니다. 사용자는 Mac 화면의 QR을 휴대폰으로 스캔할 수 있습니다. LunchTime은 입력 문자열의 QR 생성·표시까지만 책임지며, 스캔 뒤 배민 앱이나 페이지가 실제로 열리는지는 보장하지 않습니다.

- QR에는 사용자가 입력한 배민 링크 문자열을 변환·정규화하지 않고 그대로 담습니다.
- MVP는 링크 형식이나 도메인을 검증하지 않으며 링크가 잘못됐다는 경고도 제공하지 않습니다. QR 렌더링 자체가 실패한 경우에만 QR을 표시할 수 없음을 알립니다.
- LunchTime은 링크 이후의 배민 앱 동작을 제어하거나 주문 성공을 추론하지 않습니다.

### 5.4 최근 Room 상세

최근 Room 목록은 재모집용 가게 히스토리와 구분합니다. 자동 신뢰된 사내망 Peer는 취소되지 않은 Room을 선택해 최대 14일 동안 구조화된 최종 상태 또는 `종료 · 동기화 불완전` 상태를 열람할 수 있습니다.

- 가게명·배민 링크, 최종 주문 상태, 참여자와 개인 메뉴 스냅샷, 주요 시스템 이벤트 및 최종화 완전성을 표시합니다.
- Room 참여 여부와 관계없이 자동 신뢰된 사내망 Peer가 평문 상세를 열람할 수 있습니다.
- 늦은 재연결에서 14:30 이전의 유효 이벤트를 더 받으면 스냅샷을 다시 계산할 수 있으며, 화면에 마지막 재계산 시각을 표시합니다.
- 사용자는 Recent Room Detail에서 수동 새로고침으로 새 제한 동기화 세션을 시작할 수 있습니다.
- 어떤 값도 수정할 수 없으며, 재모집은 별도의 가게 히스토리에서 시작합니다.
- 취소된 Room은 최근 Room 목록에 표시하지 않습니다.
- 앱 종료 시 삭제된 Room 채팅은 최근 Room 상세에 복원하거나 영구 기록하지 않습니다.

### 5.5 연결 화면

연결 화면은 Peer별 닉네임, 온라인 여부, 마지막 확인 시각과 동기화 상태만 표시합니다. 같은 닉네임이 둘 이상이면 로컬에서 안정적인 짧은 접미사를 붙여 구분합니다. 신뢰 여부, 개별 변경 ACK와 암호화 키 정보는 이 화면에 표시하지 않습니다.

## 6. 기능 요구사항

### FR-01. 일일 라이프사이클

1. 앱의 당일 쓰기 기능은 11:00부터 활성화합니다.
2. 방 생성 시 주문 마감 시각을 필수로 입력하며, 현재 시각 이후이면서 14:30보다 빨라야 합니다.
3. 주문 마감 시각이 되면 신규 참여만 닫습니다.
4. 기존 참여자는 14:30 전까지 방 열람과 방 채팅을 계속할 수 있습니다. 자신의 메뉴는 주문 상태가 `진행 중`일 때만 관리할 수 있습니다.
5. 매일 14:30에 당일 이벤트 쓰기를 막고 당일 Room을 MenuBar에서 제거한 뒤 백그라운드 finalization을 시작합니다. 현재 앱 세션에서는 Room과 라운지 창을 메모리 내용의 열람 전용 Surface로 계속 열 수 있습니다.
6. finalization은 14:30 이전에 생성된 이벤트만 받아 14:32까지 최대 120초 동안 대조합니다. 제한 안에 필수 최신 리비전을 확인하면 finalization을 완료하고, 확인하지 못하면 사용자 상태를 `종료 · 동기화 불완전`으로 기록합니다.
7. 14:30에 잠자기·종료 상태였던 기기는 다시 실행되거나 깨어나는 즉시 쓰기를 재개하지 않고 그 시점부터 최대 120초 동안 최종화를 수행합니다. 보존 중인 14:30 이전 이벤트를 나중에 더 받으면 최근 Room 스냅샷을 다시 계산할 수 있습니다.
8. 14:30 이후에는 당일 Room을 쓰기 가능한 활성 상태로 다시 열거나 생성·참여·수정·채팅·상태 변경할 수 없습니다. 현재 앱 세션의 Room·라운지 채팅은 열람 전용 창에서 읽을 수 있고, 구조화된 종료 결과는 최근 Room 상세에서도 확인합니다.
9. 활성 데이터의 주기적 anti-entropy는 14:30에 중단합니다.
10. Peer 간 시계 차이가 허용 범위를 넘으면 방 생성, 참여 수락, 주문 마감 수정, 주문 상태 변경, 참여 철회와 Room 취소처럼 시간 경계에 영향을 받는 쓰기를 막고 `확인 필요`와 기기 시계 점검 안내를 표시합니다. 14:30 쓰기 종료는 안전하게 적용하되 종료 스냅샷의 유효성·완전성은 자동 확정하지 않습니다.
11. 11:00 전사 알림은 MVP에서 제공하지 않습니다.

추적: D-08, D-10, D-13~D-14, D-31, D-38~D-39, F-11, F-17, F-40, F-64, F-74

### FR-02. 점심방 생성과 참여

1. 새 방에는 가게명을 필수로 입력합니다.
2. 배민 링크와 주문 마감 시각을 함께 필수로 입력합니다.
3. 방은 가게 하나만 가집니다. 후보 가게와 투표 단계는 두지 않습니다.
4. 생성자는 생성과 동시에 참여자로 등록합니다.
5. 같은 운영일에 여러 방이 존재할 수 있습니다.
6. 방은 참여자가 한 명뿐이어도 유효합니다.
7. 신규 참여는 요청 기기가 주문 마감 전에 유효한 수락 ACK를 받은 경우에만 확정합니다. 마감 전에 참여를 요청했더라도 요청 기기가 마감 전까지 ACK를 받지 못하면 참여하지 않은 것으로 처리합니다.
8. 마감 전에 ACK를 받아 이미 확정된 참여 이벤트는 마감 뒤 다른 Peer에 도착해도 유효하며 참여자와 메뉴 상태에 반영합니다.
9. 인식된 Peer는 현재 참여자 목록과 각 참여자의 메뉴 제출·동기화 상태를 읽을 수 있습니다.
10. 사용자는 같은 운영일에 하나의 방에만 참여할 수 있습니다. 다른 방에 참여하거나 새 방을 만들려면 주문 상태가 `진행 중`일 때 기존 참여를 먼저 철회해야 합니다.
11. 재동기화 결과 같은 사용자의 여러 Room 참여가 발견되면 자동으로 승자를 고르지 않고 `확인 필요`로 표시합니다. 사용자가 유지할 Room 하나를 선택할 때까지 해당 사용자의 참여·메뉴 쓰기와 관련 Room의 주문 완료를 막고, 선택 후 나머지 참여와 메뉴를 무효화하는 이벤트를 전파합니다.
12. 참여 철회는 자신의 메뉴를 함께 무효화하는 삭제 이벤트로 동기화합니다.
13. 생성자가 현재 참여자이고 주문 상태가 `진행 중`이며 현재 시각이 기존 주문 마감 전일 때만 주문 마감 시각을 수정할 수 있습니다. 새 시각도 현재보다 이후이고 14:30보다 빨라야 하며, 한 번 마감된 참여 모집은 다시 열 수 없습니다.
14. 생성자가 철회했지만 다른 참여자가 남아 있으면 Room은 유지되고 잔여 참여자는 메뉴·채팅과 주문 완료를 계속할 수 있습니다. 철회한 생성자의 관리 권한은 즉시 끝나며 재참여해도 복구되지 않고 다른 참여자에게 승계하지 않습니다. 이후 가게명, 배민 링크와 주문 마감 시각은 잠급니다.
15. 생성자가 현재 참여자이고 주문 상태가 `진행 중`일 때 Room을 명시적으로 취소할 수 있습니다. 마지막 참여자가 철회하면 Room은 자동으로 취소됩니다.
16. `주문 완료` Room은 먼저 완료를 되돌려야 취소하거나 마지막 참여를 철회할 수 있습니다. 취소는 비가역이며 앱 창의 오늘 목록에서 14:30까지 `모집 취소`·열람 전용으로 접근할 수 있지만, 최근 Room 상세와 가게 히스토리에서는 제외합니다.

추적: D-21, D-26, D-29, D-33, D-40, F-09, F-11~F-15, F-17~F-18, F-22~F-24, C-05

### FR-03. 개인 메뉴

1. 참여자는 자신에게 귀속된 메뉴 항목을 한 개 이상 등록할 수 있습니다.
2. 각 메뉴 항목의 입력 구조는 다음과 같습니다.

| 필드 | 필수 여부 | 규칙 |
|------|-----------|------|
| 메뉴명 | 필수 | 한 줄 텍스트 |
| 옵션 | 선택, 여러 개 | 옵션 한 개당 한 줄 텍스트 |

3. 자유 형식의 여러 줄 텍스트 한 덩어리 대신 위 구조를 고정된 폼으로 제공합니다.
4. 참여자는 주문 상태가 `진행 중`이고 14:30 전인 동안 자신의 메뉴를 수정할 수 있습니다. 주문 마감 후 수정도 가능하지만 변경 사실과 동기화 상태를 눈에 띄게 표시합니다.
5. 주문 완료 상태에서는 메뉴를 잠그고, 주문 완료를 되돌려 `진행 중`으로 돌아오면 다시 수정할 수 있습니다.
6. 다른 참여자의 메뉴는 수정할 수 없습니다.
7. 참여자별 메뉴를 한 화면에서 주문용 목록으로 확인하고 복사할 수 있습니다. 누락·대기·충돌 상태가 있으면 복사를 비활성화하고 이유를 표시합니다.
8. 모든 메뉴 생성·수정에는 고유 변경 ID와 작성 Peer를 부여해 중복 또는 누락 여부를 추적할 수 있어야 합니다.
9. `이 기기에 저장됨`, `복제됨`, `동기화됨`, `확인 필요`의 네 안정 상태를 구분해 표시합니다. 교환 중에는 별도 안정 상태를 만들지 않고 일시적인 `동기화 중` 진행 표시를 덧붙입니다.
10. 옵션 안의 금액 표기는 문자열의 일부로 보존하며 MVP에서 가격을 별도 계산하거나 검증하지 않습니다.
11. 인식된 Peer는 활성 Room의 참여자와 메뉴를 읽을 수 있지만, 메뉴 생성·수정·삭제는 해당 Room의 현재 참여자 본인에게만 허용합니다.

예시:

```text
메뉴명: 다운타우너 버거 세트
옵션:
- 오리지널 프라이즈 / +2,000원
- 코카콜라 제로 255ml
- 싱글
```

추적: D-07, D-32, D-35, F-34~F-37, F-50, F-55, F-58

### FR-04. 방별 채팅

1. 각 점심방은 해당 주문 협의를 위한 채팅을 제공합니다.
2. 채팅은 구조화된 가게·참여·메뉴 정보를 대체하지 않습니다.
3. 인식된 Peer는 활성 Room 채팅을 읽을 수 있지만 메시지 작성은 해당 Room의 현재 참여자만 할 수 있습니다. 참여가 마감된 뒤에도 기존 참여자는 14:30까지 메시지를 주고받을 수 있습니다.
4. 14:30 이후에는 Room 채팅 입력을 제공하지 않습니다. 현재 앱 세션의 메모리에 남은 채팅은 열람 전용 Room 창에서 읽을 수 있고 앱 종료 뒤에는 복원하지 않습니다.
5. 채팅 데이터는 메모리에서만 관리하며 앱 종료 시 해당 기기에서 삭제합니다.
6. 앱 재시작 시 현재 활성 일일 세션에 한해 연결 가능한 Peer가 메모리에 보유한 당일 메시지를 다시 받을 수 있지만, 모든 Peer에서 삭제된 메시지의 복구는 보장하지 않습니다.
7. 각 메시지에는 전역적으로 충돌 가능성이 낮은 메시지 ID, Room ID, 작성 사용자·Peer와 논리적 순서 정보를 포함합니다.
8. 같은 메시지 ID는 한 번만 표시합니다. 논리적 순서를 먼저 적용하고 순서가 같은 메시지는 메시지 ID로 결정적으로 정렬하며, 늦게 받은 메시지는 정해진 위치에 삽입할 수 있습니다.
9. 메시지 전송과 재수신은 best-effort이며 영구 전달이나 모든 Peer 수신을 보장하지 않습니다. 전송·복구 진행과 마지막 동기화 시각은 표시합니다.

추적: D-04, D-13, D-19, D-35, D-37, F-19~F-21, F-51, F-62, F-69

### FR-05. 주문 상태

1. 참여자 누구나 실제 주문 이후 방을 `주문 완료`로 표시할 수 있습니다.
2. 주문 완료는 14:30 전까지 참여자 누구나 되돌릴 수 있습니다. 되돌리기는 주문 취소 또는 재주문 상황을 위한 것입니다.
3. 주문 완료와 되돌리기에는 수행자, 시각과 변경 ID를 기록하고 Peer에 전파합니다.
4. 주문 완료 상태에서도 14:30 전까지 기존 참여자의 방과 채팅은 유지됩니다.
5. 주문 완료 상태에서는 신규 참여, 참여 철회, 주문 마감 시각, 가게 정보와 개인 메뉴 변경을 잠급니다.
6. 주문 완료를 되돌리면 참여 가능 여부는 원래 주문 마감 시각으로 다시 계산합니다.
7. 14:30 쓰기 종료 이후에는 주문 완료 상태를 포함한 모든 변경을 막습니다.
8. `출발` 상태는 두지 않습니다.
9. Room 취소가 필요하면 먼저 주문 완료를 되돌려 `진행 중`으로 바꿔야 합니다.

추적: D-09, D-10, D-31, D-33, F-38~F-40, F-73

### FR-06. 주문 전 누락 확인

주문 완료 동작을 실행하기 전에 다음 요약을 보여줍니다.

- 참여자 수와 명단
- 메뉴를 제출한 참여자와 제출하지 않은 참여자
- 각 메뉴의 마지막 변경 시각과 동기화 상태
- 현재 연결된 Peer 수와 마지막 전체 대조 시각
- 현재 참여자의 최신 리비전 보유 여부
- 아직 전파 중이거나 충돌 해결이 필요한 항목

동기화되지 않은 변경, 메뉴 미제출 참여자 또는 충돌이 있으면 경고와 새로고침 동작을 제공하고 주문 완료를 막습니다.

- 다인 방에서는 각 현재 참여자의 최신 참여·메뉴 리비전이 최소 한 대의 다른 인식된 Peer에 복제되고, 주문 완료를 실행하는 기기가 ACK된 최신 리비전을 모두 보유해야 합니다.
- 과거에 Room 이벤트를 보유하거나 ACK했지만 현재 오프라인인 Peer는 위 조건을 충족한 경우 주문 완료를 막지 않습니다.
- 해당 Room과 무관하거나 같은 Room 요약을 보유하지 않았다고 확인된 Peer의 미대조·미응답도 주문 완료를 막지 않습니다. 현재 정상 응답 Peer가 더 최신이거나 서로 다른 Room 요약을 알린 경우에만 대조가 끝날 때까지 막습니다.
- 단일 참여자 방의 ACK 없는 완료는 생성자만 참여하고 어떤 Room 이벤트도 다른 Peer와 공유하거나 ACK받은 적이 없는 Room에만 허용합니다. 사용자는 로컬 최신 리비전을 직접 확인해야 합니다.
- 한 번이라도 다른 Peer와 공유된 Room은 현재 단일 참여자이고 다른 Peer가 오프라인이어도 ACK 없는 완료 예외를 적용하지 않습니다.
- 다인 방에는 누락 위험을 무시하는 강제 완료를 제공하지 않습니다.

### FR-08. 가게 히스토리와 재모집

1. 히스토리의 목적은 자주 주문하는 가게로 새 방을 빠르게 만드는 것입니다.
2. 히스토리에는 가게명, 배민 링크, 최근 주문 완료 시각과 로컬 이용 횟수를 저장합니다.
3. 채팅, 과거 참여자와 개인 메뉴는 재사용 히스토리에 저장하지 않습니다.
4. 가게 히스토리는 각 사용자의 Mac에 암호화해 저장하며, 14일 운영 데이터 보존과 별개로 사용자가 삭제할 때까지 유지합니다.
5. Peer 연결 시 서로의 가게 히스토리를 합쳐 앱 재시작 또는 기기별 차이를 복구합니다.
6. 같은 가게를 판별하는 키는 `가게명의 양끝 공백 제거 결과`와 `사용자가 입력한 배민 링크 원문`의 정확한 조합입니다. 대소문자, URL 구성요소 또는 링크 문자열을 정규화하지 않습니다.
7. Peer에는 가게명, 배민 링크와 최근 주문 완료 시각만 공유합니다. 이용 횟수는 기기별 로컬 값으로 유지하며 Peer 값과 더하거나 덮어쓰지 않습니다.
8. 히스토리의 `다시 모집`을 선택하면 가게명과 배민 링크가 채워진 방 생성 화면을 엽니다.
9. 사용자가 주문 마감 시각과 복사 내용을 확인한 뒤 방을 생성합니다. 이전 참여자와 개인 메뉴는 복사하지 않습니다.
10. 가게 히스토리에는 `주문 완료` 상태로 finalization이 완료된 Room만 성공 이용으로 반영합니다. 취소됐거나 `종료 · 동기화 불완전`인 Room은 반영하지 않습니다.

### FR-09. Peer 발견과 동기화

1. 지원 대상은 사내 두 Wi-Fi가 연결되는 동일 내부망입니다.
2. 지원 사내 Wi-Fi에서 발견한 LunchTime Peer는 별도 사용자 승인 없이 자동 신뢰하며, 각 Peer는 로컬 데이터를 보유하고 연결 가능한 Peer와 변경사항을 직접 교환합니다.
3. 중앙 리더 또는 항상 켜진 방 생성자에 의존하지 않고, 보유한 Peer끼리 상태를 이어 전달할 수 있어야 합니다.
4. 새 Peer 연결 시 전체 데이터를 매번 복사하는 대신 보유 버전 또는 요약을 비교해 누락된 변경을 교환합니다.
5. 같은 변경을 여러 Peer에서 받아도 한 번만 적용합니다.
6. 방 생성자가 잠자기·종료·이탈해도 다른 Peer가 데이터를 가지고 있다면 방을 계속 사용할 수 있어야 합니다.
7. 정확한 복제 자료구조, 충돌 해결 알고리즘과 Peer 발견 방식은 기술 스파이크에서 결정합니다. 블록체인이나 작업증명은 요구사항이 아닙니다.

자동 동기화 트리거는 다음과 같습니다.

- Mac이 잠자기에서 깨어남
- 앱이 전면으로 전환됨
- 네트워크 연결 또는 SSID가 바뀜
- 새 Peer가 연결됨
- 로컬 활성 데이터가 변경됨
- 앱이 실행 중이고 Mac이 깨어 있는 동안의 주기적 anti-entropy 대조

11:00~14:30에 현재 운영일의 활성 Room·라운지 데이터 또는 미해결 동기화 데이터가 있고 정상 응답 Peer가 하나 이상 있을 때만, 마지막 성공 anti-entropy 대조 후 30초 안에 다음 대조를 시작해야 합니다. 대상 데이터나 정상 응답 Peer가 없으면 주기 대조를 멈춥니다. 로컬 변경, 원격 차이 발견 또는 복귀 트리거는 주기와 별개로 즉시 일반 동기화 세션을 시작합니다. 14:30에는 활성 데이터의 주기적 대조를 중단합니다.

자동 트리거 또는 수동 새로고침으로 시작한 일반 동기화 세션은 최초 시도를 포함해 최대 3회이면서 시작 후 총 30초 이내에서만 교환을 시도합니다. 횟수 또는 시간 한도 중 하나라도 먼저 도달하면 세션을 종료하고 미해소 항목을 `확인 필요`로 전환합니다. 이후에는 깨우기, 전면 전환, 네트워크·Peer 변화, 로컬 변경 같은 의미 있는 새 트리거나 사용자의 수동 새로고침이 있을 때만 새 세션을 시작합니다. 제한을 소진한 실패 Peer를 정기 anti-entropy 타이머만으로 다시 활성화하지 않으며, 건강한 다른 Peer와 데이터에 대한 주기 대조만 계속할 수 있습니다.

14:30 최종화는 일반 동기화와 구분되는 최대 120초의 외곽 작업입니다. 경계에 깨어 있는 기기는 14:32까지, 경계에 잠든 기기는 다음 실행·깨우기부터 최대 120초 동안 수행합니다. 최종화 안의 개별 Peer 교환 세션도 최대 3회·총 30초 한도를 지키며, 120초 외곽 한도가 실패한 한 세션을 연장하지 않습니다.

추적: D-02, D-03, D-15~D-16, D-27, D-31~D-32, D-35, F-49~F-55, F-58~F-59, F-63, F-65

### FR-10. 복귀와 최신성 표시

1. 잠자기 또는 네트워크 단절 중에는 실시간 동기화를 보장하지 않습니다.
2. 복귀 트리거가 발생하면 Peer 요약을 비교하고 누락된 변경을 자동으로 요청합니다.
3. 각 변경 또는 집합의 안정 상태는 `이 기기에 저장됨`, `복제됨`, `동기화됨`, `확인 필요` 중 하나입니다.
4. `이 기기에 저장됨`은 로컬 반영만 확인된 상태, `복제됨`은 다른 인식된 Peer 한 대 이상의 ACK를 받은 상태입니다.
5. `동기화됨`은 현재 발견 가능한 모든 정상 응답 Peer와 해당 범위의 차이를 대조한 상태입니다. 과거에 연결됐지만 현재 오프라인인 Peer는 이 판정을 막지 않습니다.
6. `확인 필요`는 충돌, 복호화 실패, 과도한 시계 차이 또는 제한된 재시도 뒤에도 해소하지 못한 데이터 공백이 있는 상태입니다.
7. `동기화 중`은 위 네 안정 상태와 별개인 일시적 진행 표시입니다. 교환이 끝나면 결과에 맞는 안정 상태로 돌아갑니다.
8. 완료 시 마지막 동기화 시각, 확인한 Peer 수와 복구한 변경 수를 갱신합니다.
9. 데이터 공백을 해소하지 못하면 수동 새로고침과 문제 안내를 제공합니다. 연결 가능한 어떤 Peer도 데이터를 보유하지 않으면 복구할 수 없음을 명시합니다.
10. 연결 화면에는 Peer별 닉네임, 온라인 여부, 마지막 확인 시각과 동기화 상태만 표시합니다. 같은 닉네임은 안정적인 짧은 접미사로 구분하며 신뢰 여부나 개별 ACK는 표시하지 않습니다.

추적: D-32, D-35, D-38, F-21, F-53~F-54, F-58, F-62~F-63

### FR-11. 데이터 보존

데이터 종류별 보존은 다음과 같습니다.

| 데이터 | 저장 위치 | 보존 |
|--------|-----------|------|
| 채팅 메시지 | 실행 중인 앱 메모리 | 앱 종료 시 해당 기기에서 삭제, 현재 운영일에 살아 있는 Peer가 보유한 메시지만 best-effort 재수신 |
| 방·참여·메뉴·상태 운영 데이터 | 암호화된 로컬 저장소와 실행 중 메모리 | 생성 후 최대 14일 |
| 동기화용 변경 이벤트 | 암호화된 로컬 저장소 | 생성 후 최대 14일 |
| 가게 히스토리 요약 | 암호화된 로컬 저장소 | 사용자가 삭제할 때까지 |

1. 14일이 지난 운영 데이터와 변경 이벤트는 자동 삭제할 수 있습니다.
2. 삭제가 모든 Peer에서 동시에 일어날 필요는 없으며 고아 데이터가 일시적으로 남아도 됩니다.
3. 만료되거나 모든 Peer에서 삭제된 데이터는 복구를 보장하지 않습니다.
4. 가게 히스토리는 장기 재사용을 위한 최소 요약이며 원본 방·메뉴·채팅을 영구 보존하는 수단이 아닙니다.
5. 취소되지 않은 Room의 상세 열람은 운영 데이터가 남아 있는 최대 14일 동안만 제공하며, 이후에는 최소 가게 히스토리만 남습니다.
6. 최근 Room 상세는 Room 참여 여부와 관계없이 자동 신뢰된 사내망 Peer에 제공하고 가게명·링크, 최종 상태, 참여자·메뉴 스냅샷, 시스템 이벤트와 최종화 완전성을 열람 전용으로 표시합니다.
7. Room 채팅은 메모리 전용 계약을 따르므로 최근 Room 상세의 지속 데이터에 포함하지 않습니다.
8. 14일 운영 데이터가 만료되면 최근 Room 상세 진입점을 제거하되 장기 최소 가게 히스토리는 유지합니다.
9. `종료 · 동기화 불완전` Room도 최근 Room 상세에는 표시하되 성공 가게 히스토리에는 반영하지 않습니다.
10. 늦은 재연결에서 유효한 14:30 이전 이벤트를 더 받으면 보존 기간 안의 최근 Room 스냅샷과 완전성 상태를 다시 계산하고 마지막 재계산 시각을 표시합니다. 다시 계산한 결과가 성공 조건을 충족하지 않으면 기존 성공 가게 히스토리를 무효화하고, 조건을 충족할 때만 생성·복구합니다.
11. 취소된 Room은 최근 Room 상세와 가게 히스토리 모두에 남기지 않습니다.

### FR-12. 데이터 보호

1. Peer 간 모든 응용 데이터는 인증된 암호화 채널로 전송합니다.
2. 방·메뉴·동기화 이벤트와 가게 히스토리 등 디스크에 남는 데이터는 암호화합니다.
3. 암호화 키 또는 장기 비밀은 macOS Keychain을 이용해 보호합니다.
4. 지원 사내 Wi-Fi에서 발견한 LunchTime Peer는 자동 신뢰하고 별도 승인·확인 코드 또는 신뢰 관리 UI를 요구하지 않습니다.
5. 인식된 Peer는 활성 Room의 참여자·메뉴·채팅과 보존 중인 최근 Room 상세를 평문으로 읽을 수 있습니다. 메뉴·채팅 쓰기는 해당 Room의 현재 참여자에게만 허용하며 참여 요청은 FR-02의 수락 규칙을 따릅니다.
6. 복호화 실패와 변조된 변경 이벤트는 적용하지 않고 사용자에게 상태를 알립니다.
7. 로그와 진단 정보에 채팅 본문, 배민 링크 또는 개인 메뉴 원문을 평문으로 남기지 않습니다.
8. 전송·저장 암호화는 네트워크 도청과 저장 파일 탈취를 줄이기 위한 것입니다. 지원 사내 Wi-Fi 안에서 자동 신뢰된 악의적인 LunchTime Peer의 평문 열람·유효 권한 내 동작은 MVP 보호 범위 밖입니다.
9. 자동 신뢰를 유지하면서 암호화 채널과 저장 암호화에 필요한 기기 키 교환·회전 방식은 보안 스파이크에서 확정합니다.
10. 사용자 ID는 설치·기기 하나에 결합하며 닉네임과 분리합니다. 동일 인물의 여러 기기 연결과 권한 이전은 MVP에서 제공하지 않습니다.

## 7. 동기화 제품 계약

MVP는 Peer별 로컬 장부와 변경 교환을 이용한 **최종 일관성**을 목표로 합니다. 비트코인과 토렌트는 중앙 정본 없이 각 참여자가 데이터를 보유하고 교환한다는 개념적 참고일 뿐, 블록체인·채굴·전역 합의 자체를 구현 요구사항으로 삼지 않습니다.

### 변경 이벤트의 최소 속성

- 전역적으로 충돌 가능성이 낮은 변경 ID
- 데이터 종류와 대상 ID
- 작성 Peer와 사용자 식별자
- 작성 시각과 논리적 순서 정보
- 변경 내용 또는 삭제 표시
- 무결성을 확인할 수 있는 인증 정보

### 사용자에게 보여줄 동기화 상태

| 상태 | 의미 |
|------|------|
| 이 기기에 저장됨 | 로컬에는 반영됐지만 다른 Peer의 수신을 확인하지 못함 |
| 복제됨 | 다른 인식된 Peer 한 대 이상이 해당 변경을 ACK함 |
| 동기화됨 | 현재 발견 가능한 모든 정상 응답 Peer와 해당 범위의 차이를 대조함 |
| 확인 필요 | 충돌, 복호화 실패, 과도한 시계 차이 또는 제한된 재시도 뒤에도 해소되지 않은 데이터 공백이 있음 |

`동기화 중`은 네 상태 중 하나가 아니라 교환이 진행되는 동안 덧붙이는 일시적 표시입니다. `동기화됨`은 회사 전체의 모든 Mac이나 과거에 연결된 오프라인 Peer가 수신했다는 의미가 아닙니다. UI 도움말에서 현재 발견 가능한 정상 응답 Peer 기준이라는 한계를 설명해야 합니다.

## 8. 수용 시나리오

### AC-01. 단일 참여자 주문

- 한 사용자가 11:00 이후 방을 만들고 자동 참여됩니다.
- 가게명, 배민 링크와 주문 마감 시각을 입력합니다.
- QR에는 입력한 링크 원문이 그대로 인코딩·표시되며 형식·도메인 경고를 제공하지 않습니다. 스캔 뒤 어떤 앱이나 페이지가 열리는지는 수용 조건에 포함하지 않습니다.
- 생성자만 참여하고 어떤 Room 이벤트도 다른 Peer와 공유하거나 ACK받은 적이 없으면 구조화된 개인 메뉴를 로컬에서 확인하고 ACK 없이 주문 완료할 수 있습니다.
- Room이 한 번이라도 다른 Peer와 공유됐다면 현재 다른 Peer가 오프라인이어도 ACK 없는 단일 참여자 예외를 적용하지 않습니다.
- 14:30에 Room 입력과 MenuBar 항목을 닫고 현재 앱 세션의 Room 창을 열람 전용으로 제공한 뒤 백그라운드 finalization을 시작합니다.

### AC-02. 메뉴 누락 방지

- 두 명 이상이 참여한 방에서 한 참여자가 메뉴를 생성·수정하면 동기화 상태가 표시됩니다.
- 주문 완료 전 요약에 모든 참여자와 메뉴 제출 여부가 나타납니다.
- 아직 동기화되지 않은 메뉴가 있으면 경고와 새로고침 동작이 나타나며 주문 완료가 차단됩니다.
- 다인 방의 각 현재 참여자 최신 참여·메뉴 리비전은 최소 한 대의 다른 인식된 Peer가 ACK하고, 주문 완료 기기가 그 리비전을 모두 보유해야 합니다.
- 위 조건을 만족하면 과거에 Room 이벤트를 보유하거나 ACK했지만 현재 오프라인인 Peer는 완료를 막지 않습니다.
- 누락·대기·충돌 항목을 무시하고 주문 완료하는 우회 동작은 제공하지 않습니다.
- 모든 조건을 통과하면 주문용 메뉴 요약을 복사할 수 있고, 복사 결과는 점검에 사용한 최신 리비전과 일치해야 합니다.
- 지원 네트워크에서 성공적으로 동기화된 메뉴가 주문 요약에서 조용히 누락되는 테스트 케이스가 없어야 합니다.

### AC-03. 잠자기 후 복구

- 한 Mac이 잠자기 상태에서 다른 Peer가 참여, 메뉴와 채팅을 변경합니다.
- 잠든 Mac이 깨어나면 별도 재실행 없이 자동 대조를 시작합니다.
- 연결 가능한 Peer가 데이터를 보유하면 누락분을 복구합니다.
- 자동 트리거와 수동 새로고침으로 시작한 일반 동기화 세션은 최초 시도를 포함해 최대 3회, 총 30초 안에서만 시도하고 둘 중 먼저 한도에 도달하면 미해소 항목을 `확인 필요`로 표시합니다.
- 교환 중에는 기존 안정 상태에 `동기화 중`을 일시적으로 덧붙이고, 완료 후 네 안정 상태 중 하나와 마지막 동기화 시각을 표시합니다.
- 11:00~14:30에 활성 Room·라운지 또는 미해결 데이터와 정상 응답 Peer가 모두 있으면 마지막 성공 anti-entropy 대조 후 30초 안에 다음 대조를 시작하고, 어느 한쪽도 없으면 멈춥니다. 활성 데이터 변경·차이 발견 또는 복귀 트리거는 주기와 별개로 즉시 일반 동기화 세션을 시작합니다.
- 제한을 소진한 실패 Peer는 정기 타이머만으로 다시 시도하지 않고 네트워크·Peer 상태 변화나 수동 새로고침 같은 새 트리거를 기다립니다.
- 14:30 이후 깨어난 기기는 당일 쓰기나 Room의 쓰기 가능 상태를 다시 열지 않고, 열람 전용 상태를 적용한 뒤 깨어난 시점부터 최대 120초 동안 14:30 이전 이벤트로 finalization을 수행합니다. 그 내부의 개별 Peer 교환 세션에는 최대 3회·총 30초 한도를 적용하고, 이후 유효 이벤트 수신 시 최근 Room 스냅샷만 다시 계산합니다.

### AC-04. 앱 재시작

- 앱을 종료하면 해당 기기의 채팅 메모리가 삭제됩니다.
- 재시작 후 방·상태와 가게 히스토리는 로컬 저장소에서 복구됩니다.
- 현재 운영일의 활성 시간이고 연결 가능한 Peer가 당일 채팅을 메모리에 보유하고 있으면 다시 동기화합니다.
- 같은 메시지 ID는 한 번만 표시합니다. 동일한 메시지 집합을 보유한 Peer는 논리적 순서와 메시지 ID로 같은 결정적 순서를 계산하지만, 메시지 누락으로 인한 전역 순서 일치는 보장하지 않습니다.
- 어떤 Peer도 채팅을 보유하지 않으면 복구 불가를 오류가 아닌 허용된 보존 결과로 처리합니다.

### AC-05. 주문 완료 취소와 일일 종료

- 참여자가 주문 완료로 표시한 뒤 14:30 전에는 참여자 누구나 이를 되돌릴 수 있습니다.
- 변경 수행자와 시각이 모든 연결 가능한 Peer에 전파됩니다.
- 14:30이 되면 주문 상태와 무관하게 쓰기와 MenuBar 항목을 막고 현재 앱 세션에서는 Room 창을 현재 메모리 내용의 열람 전용 Surface로 제공합니다.
- 백그라운드 finalization은 14:32까지 최대 120초만 수행하고, 필요한 최신 리비전을 확인하지 못하면 `종료 · 동기화 불완전`으로 끝냅니다.
- 자동 신뢰된 사내망 Peer는 취소되지 않은 Room의 완료 또는 불완전 상세를 최대 14일 동안 열 수 있지만, 앱 종료로 삭제된 채팅은 복원되지 않습니다.
- `종료 · 동기화 불완전` Room은 성공 가게 히스토리에 반영하지 않습니다. 늦게 받은 14:30 이전 이벤트로 최근 Room 스냅샷과 완전성을 다시 계산하며, 결과에 따라 성공 히스토리를 생성·복구하거나 기존 성공 레코드를 무효화합니다.
- 14일 뒤에는 최근 Room 상세가 사라지고 성공적으로 최종화된 Room의 최소 가게 히스토리만 남습니다.

### AC-06. 라운지

- 사용자는 별도 앱 창에서 라운지에 들어가 11:00~14:30에 대화할 수 있습니다.
- 점심방 채팅과 라운지 메시지가 섞이지 않습니다.
- 14:30에는 라운지 입력을 닫고 현재 앱 세션에서 메모리의 메시지를 열람 전용 라운지 창으로 볼 수 있게 합니다.
- 앱 종료 시 해당 기기의 라운지 메시지가 삭제됩니다.

### AC-07. 히스토리 재모집

- 과거 가게에서 `다시 모집`을 선택하면 양끝 공백이 제거된 가게명과 입력 당시 링크 원문이 채워진 생성 화면이 열립니다.
- 주문 마감 시각을 확인해야 방을 생성할 수 있습니다.
- 이전 참여자, 개인 메뉴와 채팅은 복사되지 않습니다.
- 가게 히스토리 중복 키는 양끝 공백을 제거한 가게명과 링크 원문의 정확한 조합이며 URL 정규화나 대소문자 변환을 하지 않습니다.
- 취소됐거나 최종화가 불완전한 Room은 성공 가게 히스토리에 나타나지 않습니다.

### AC-08. 암호화와 신뢰 경계

- 패킷 캡처에서 LunchTime 응용 데이터가 평문으로 노출되지 않습니다.
- 지원 사내 Wi-Fi에서 발견한 LunchTime Peer는 별도 승인 없이 자동 신뢰됩니다.
- 자동 신뢰된 비참여 Peer도 활성 Room 메뉴·채팅과 최근 Room 상세를 읽을 수 있지만 메뉴·채팅 쓰기는 할 수 없습니다.
- 로컬 저장 파일만 복사해서 방·메뉴·히스토리 원문을 읽을 수 없습니다.
- 연결 화면은 닉네임, 온라인 여부, 마지막 확인 시각과 동기화 상태만 표시하고 신뢰·개별 ACK 정보는 표시하지 않습니다. 중복 닉네임에는 안정적인 접미사를 붙입니다.
- 자동 신뢰된 내부 악성 Peer의 평문 열람과 허용된 권한 안의 악성 동작은 MVP 보호 범위 밖입니다.

### AC-09. 참여 마감과 늦은 전파

- 사용자가 마감 직전에 참여를 요청하고 요청 기기가 마감 전에 유효한 수락 ACK를 받으면 참여가 확정됩니다.
- 이미 확정된 참여 이벤트가 네트워크 지연으로 마감 뒤 다른 Peer에 도착해도 참여와 이후 유효한 메뉴를 반영합니다.
- 참여 요청은 마감 전이지만 요청 기기가 마감 전까지 수락 ACK를 받지 못하면 참여를 거부합니다.
- 사용자의 기기 시계 차이가 허용 범위를 넘으면 참여 수락을 막고 시계 점검 안내를 표시합니다.

### AC-10. 마감 수정과 Room 취소

- 현재 참여 중인 생성자는 주문이 `진행 중`이고 기존 마감 전일 때만 미래의 14:30 이전 시각으로 마감을 수정할 수 있습니다.
- 이미 모집이 마감된 Room은 마감을 늦춰 다시 열 수 없습니다.
- 현재 참여 중인 생성자는 `진행 중` Room을 취소할 수 있고, 마지막 참여자가 철회하면 Room이 자동 취소됩니다.
- 생성자가 철회해도 다른 참여자가 남아 있으면 Room은 유지되고 잔여 참여자는 메뉴·채팅과 주문 완료를 계속할 수 있습니다. 생성자 관리 권한은 재참여해도 복구되지 않고 승계되지 않으며 가게명, 배민 링크와 마감 시각은 잠깁니다.
- `주문 완료` Room은 먼저 완료를 되돌리지 않으면 취소하거나 마지막 참여를 철회할 수 없습니다.
- 취소된 Room은 다시 열리지 않고 최근 Room 상세와 가게 히스토리에 표시되지 않습니다.

### AC-11. 중복 참여 충돌

- 네트워크 분리 뒤 재동기화에서 같은 사용자의 당일 여러 Room 참여가 발견되면 자동 승자를 선택하지 않습니다.
- 사용자가 유지할 Room을 직접 선택할 때까지 해당 사용자의 참여·메뉴 쓰기와 관련 Room의 주문 완료를 막습니다.
- 선택 뒤 나머지 참여와 메뉴를 무효화하는 이벤트가 전파되고 모든 정상 응답 Peer가 같은 결과로 수렴합니다.

## 9. 출시 전 기술 스파이크

다음 항목은 제품 방향은 확정됐지만 구현 방식을 검증해야 합니다. 각 스파이크는 `가능/불가능`만 결론 내리지 않고, 채택할 구현 방식, 보장 범위, 실패 UX와 테스트 근거를 남겨야 합니다.

| Spike | 확인할 질문 | PRD에 미치는 결과 |
|-------|-------------|--------------------|
| SP-01 사내망 Peer 발견 | 두 Wi-Fi가 연결된 내부망에서 mDNS/Bonjour 등으로 Peer를 안정적으로 발견하고 지원 사내 Wi-Fi 여부를 판별할 수 있는가? VLAN, 방화벽과 절전 상태별 범위는 무엇인가? | 자동 신뢰 대상 네트워크와 발견 실패 UX의 기술 기준 확정 |
| SP-02 복제·충돌 | 변경 로그, CRDT 또는 다른 방식 중 메뉴 누락 방지와 구현 복잡도에 적합한 것은 무엇인가? 마감 전 참여 수락 ACK, 늦은 전파, Room 취소, 메시지 순서와 동일 사용자의 다중 Room 충돌을 어떻게 결정적으로 수렴시킬 것인가? | 확정된 사용자 결과를 만족하는 변경 순서, 병합과 무효화 구현 확정 |
| SP-03 복귀·시간 경계 | 깨우기·전면 전환·네트워크 변경 이벤트를 얼마나 신뢰할 수 있는가? 마지막 성공 대조 후 다음 대조 시작 간격 30초 이하, 세션당 최대 3회·총 30초 제한과 14:30~14:32 최종화를 어떤 스케줄링으로 지킬 수 있는가? 기기 시계 차이를 어느 허용치부터 과도하다고 판정할 것인가? | 자동 트리거, backoff, 최종화 타임아웃과 시계 차이 허용치 확정 |
| SP-04 보안·자동 신뢰 | 지원 사내 Wi-Fi Peer 자동 신뢰를 유지하면서 인증된 암호화 채널의 기기 키를 어떻게 교환·회전하고 저장할 것인가? | 자동 신뢰 범위를 바꾸지 않는 키·암호화 프로토콜 확정 |
| SP-05 데이터 경계 | 앱 종료 시 채팅 삭제와 현재 운영일의 best-effort Peer 재수신을 구현하면서 어떤 데이터를 디스크에 남기지 않을 것인가? 불완전 최종화와 늦은 재동기화 스냅샷을 14일 동안 어떻게 재계산할 것인가? | 메모리/디스크 모델, 최근 Room 재계산과 진단 로그 정책 확정 |
| SP-06 규모·신뢰성 | 예상 사용자·Peer·동시 방 규모에서 발견, 전파 지연과 복구 성공률이 허용 가능한가? | 파일럿 규모, 성능 목표와 출시 게이트 확정 |

기술 검증 결과가 요구한 안정성을 충족하지 못하면 중앙 서버를 임의로 추가하지 않습니다. 지원 범위를 좁히거나 사용자에게 보이는 동작을 재설계하고 제품 결정을 다시 받습니다.

## 11. 구현 추적 원칙

- 구현 Issue는 최소 하나의 `FR-*` 또는 `SP-*` ID를 연결합니다.
- 세부 상태 전이, 권한, 충돌, 삭제와 만료 동작은 관련 Policy ID도 함께 연결합니다.
- 메뉴 누락 가능성에 영향을 주는 변경은 `AC-02`와 복귀 관련 수용 시나리오를 회귀 테스트합니다.
- PR을 올릴 때 [`update-product-docs` Skill](../../.agents/skills/update-product-docs/SKILL.md)로 PRD·Policies 변경 필요 여부를 확인합니다.
- 제품 동작 또는 기술 보장 범위가 달라졌다면 코드만 병합하지 않고 관련 문서를 함께 갱신합니다.
