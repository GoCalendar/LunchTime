# POL-02. 복제·정합성·보존

> 상태: 정본 초안
>
> 최우선 목표: 사용자가 참여하고 메뉴를 등록했는데 다른 Peer가 이를 보지 못해 실제 주문에서 누락되는 일을 방지한다.

## 1. 중앙 정본 없는 로컬 장부

- LunchTime은 독자적인 중앙 서버나 중앙 데이터베이스를 운영하지 않는다.
- 각 기기는 자신이 받은 구조화된 운영 이벤트를 로컬 장부에 보관하고, 인식된 회사 WiFi Peer와 서로 부족한 이벤트를 교환한다.
- 이 구조는 토렌트나 분산 장부의 아이디어를 참고한 **Peer 간 복제**이며 블록체인, 채굴, 금전적 합의나 전 세계 합의를 구현한다는 뜻이 아니다.
- 특정 기기나 Room 생성자를 항상 정본으로 지정하지 않는다.
- 현재 화면 상태는 검증된 이벤트 집합에서 결정적으로 계산한 결과다.

### 운영 이벤트의 최소 식별 정보

구현은 각 구조화된 변경에 최소한 다음 정보를 부여해야 한다.

- 전역적으로 충돌 가능성이 충분히 낮은 이벤트 ID
- 일일 세션과 Room ID
- 작성 사용자와 기기 ID
- 대상 엔터티와 리비전
- 논리적 순서와 로컬 발생 시각
- 이벤트 종류와 내용
- 위·변조 검증 정보

이벤트는 수신 뒤 내용을 제자리에서 덮어쓰지 않는다. 수정과 삭제는 새 이벤트 또는 삭제 표식으로 표현한다.

## 2. 동기화 트리거

### 자동 동기화

앱은 다음 시점에 Peer 발견과 누락 이벤트 대조를 즉시 시작해야 한다.

- 앱 실행
- Mac이 잠자기에서 복귀
- 앱이 다시 활성화되거나 MenuBar에서 열림
- 네트워크 인터페이스 또는 회사 WiFi 변경
- 의미 있는 새로운 Peer 발견

각 자동 트리거는 하나의 제한된 동기화 세션을 시작한다.

- 한 세션은 시도 횟수 `최대 3회`와 전체 경과 시간 `최대 30초`를 모두 적용하며, 둘 중 하나에 먼저 도달하면 멈춘다.
- 실패한 세션을 타이머로 무한히 다시 시작하지 않는다. 이후 앱 실행·복귀·foreground·네트워크 변경·의미 있는 새 Peer 발견 또는 사용자의 수동 새로고침 같은 **새 의미 있는 트리거**가 있어야 새 세션을 시작한다.
- 한도를 소진하면 대상 데이터를 `확인 필요`로 표시한다.

11:00~14:30에 앱과 Mac이 깨어 있고 지원 네트워크에 연결된 동안에는 **현재 운영일의 활성 Room·라운지 데이터 또는 미해결 동기화 데이터가 있고 건강하게 연결된 대상 Peer가 있을 때만** 장부 요약을 주기적으로 교환한다.

- 마지막 성공 대조로부터 다음 대조 시작까지의 간격은 30초 이하다.
- 새 참여, 메뉴 등록·수정, 주문 완료·되돌리기는 주기 대조를 기다리지 않고 즉시 전파한다.
- 즉시 전파와 주기 대조도 각각 위의 최대 3회·총 30초 제한을 지킨다.
- macOS 절전이나 네트워크 상태로 목표 간격을 지키지 못하면 마지막 동기화 시각과 `확인 필요` 상태를 표시한다.
- 대상 데이터나 건강한 Peer가 없으면 주기 대조를 멈추고, 14:30에는 활성 데이터의 주기 대조를 종료한다.
- 제한을 소진한 실패 Peer는 단순 주기 타이머만으로 계속 재시도하지 않는다. 연결 상태가 실제로 바뀌거나 다른 의미 있는 트리거 또는 수동 새로고침이 있어야 다시 시도한다.
- 단순 polling만 사용하지 않고 Peer 발견 이벤트, 즉시 전파와 주기 대조를 함께 사용한다.

### 잠자기·단절 뒤 복구

1. 복귀한 기기는 현재 시각을 먼저 적용해 14:30 쓰기 종료 여부를 계산한다.
2. 발견 가능한 건강한 Peer에 자신이 가진 일일 세션·Room별 장부 요약을 보낸다.
3. 양쪽이 서로 없는 이벤트 ID와 삭제 표식을 요청해 교환한다.
4. 현재 상태를 다시 계산하고 충돌 또는 불완전 상태를 표시한다.
5. 대조가 끝나기 전까지 Room을 최신 상태로 표시하지 않는다.

14:30에 잠자기 상태였던 기기는 복귀 즉시 사용자 쓰기를 차단한 뒤 복귀 시점부터 최대 120초의 최종 대조를 수행한다. 120초는 최종 대조 전체의 바깥 한도이며, 그 안의 개별 교환 세션도 최대 3회·총 30초 제한을 지킨다. 의미 있는 Peer·연결 변화 없이 실패 세션을 반복하지 않으며, 바깥 한도를 넘기면 `종료 · 동기화 불완전`으로 멈춘다.

사용자는 언제든 `새로고침`을 실행할 수 있어야 한다. 수동 새로고침은 자동 동기화와 같은 전체 대조를 수행하며, 단순 화면 재렌더링으로 끝나면 안 된다. 수동 세션도 최대 3회·총 30초 제한을 지킨다.

## 3. 사용자에게 보이는 동기화 상태

| 상태 | 의미 | UI 규칙 |
|------|------|---------|
| `이 기기에 저장됨` | 현재 기기에 저장됐지만 다른 Peer의 수신을 확인하지 못함 | `등록 완료`라고 표시하지 않고 경고한다. |
| `복제됨` | 최소 한 대의 다른 Peer가 같은 최신 리비전을 저장했다고 ACK | 전달 사본이 생겼음을 표시한다. |
| `동기화됨` | 현재 발견 가능하고 건강하게 연결된 Peer와 장부 대조가 끝남 | 마지막 대조 시각을 함께 제공한다. 과거에 보였던 오프라인 Peer만으로 이 상태를 막지 않는다. |
| `확인 필요` | Peer 단절, 대조 지연, 충돌 또는 수신 확인 부족 | 주문 누락 위험을 눈에 띄게 표시한다. |

- `동기화 중`은 위 네 상태 중 하나를 대체하지 않는 일시적인 활동 표시다.
- 연결되지 않은 Peer가 가진 미전달 정보를 추측해서 `동기화됨`으로 표시하는 것은 금지한다.
- 아래의 엄격한 단독 Room 예외가 아니면 최소 한 대의 다른 Peer가 수신하기 전에는 참여와 메뉴 등록을 `완료`로 표현하지 않는다.
- 자신의 메뉴가 `이 기기에 저장됨` 또는 `확인 필요`인 상태에서 앱을 종료·잠자기하려 하면 누락 위험을 경고한다.

## 4. 참여·메뉴 누락 방지

### 4.1 참여 확정

- 참여 요청은 Peer가 수신한 참여 이벤트를 회신하기 전까지 `참여 확인 중`이다.
- 요청 기기가 방별 주문 마감 전에 다른 Peer의 유효한 승인 ACK를 받은 참여만 확정한다. 마감 전에 확정된 참여 이벤트는 이후 다른 Peer에 늦게 도착해도 유효하다.
- 요청 기기가 마감 전까지 승인 ACK를 받지 못한 요청은 `참여 실패`로 처리하며 이후 도착한 ACK로 소급 확정하지 않는다.
- 단, 아래 주문 전 점검의 엄격한 never-shared 단독 Room 조건을 계속 만족하는 생성자의 자동 참여만 로컬 예외로 유지할 수 있다. 다른 Peer와 공유·발견되거나 원격 이력이 생기면 이 예외는 즉시 사라진다.
- 참여 ACK 또는 위 단독 로컬 예외가 성립하지 않은 사용자는 Room의 메뉴 이벤트를 생성·전파할 수 없다. 메뉴 작성은 둘 중 하나로 현재 참여자가 된 뒤 허용한다.
- 참여 철회는 기존 참여와 메뉴를 가리는 삭제 표식으로 전파한다.

### 4.2 메뉴 확정

- 메뉴 등록·수정은 [일일·방 생명주기 정책](./01_daily_room_lifecycle.md)의 고정 템플릿 전체를 하나의 리비전으로 전파한다.
- 본인만 자신의 메뉴 리비전을 생성할 수 있다.
- 수신 Peer는 저장한 정확한 리비전 ID를 회신해야 한다. 단순 네트워크 전송 성공은 수신 확인으로 보지 않는다.
- 같은 사용자의 서로 다른 리비전이 충돌하면 조용히 마지막 쓰기를 선택하지 않고 `확인 필요`로 표시해 본인이 하나를 다시 확정하게 한다.

### 4.3 주문 전 점검

누구든 주문 완료를 요청하면 앱은 먼저 해당 Room의 참여·메뉴를 보유하거나 서로 다른 Room 버전 요약을 광고한 현재 정상 응답 Peer와 Room 범위 anti-entropy 대조를 실행하고 다음 체크리스트를 보여준다.

- 현재 알고 있는 참여자 전체
- 참여자별 `메뉴 등록`, `미제출` 상태
- 각 메뉴의 최신 리비전과 복제 상태
- 마지막 Peer 대조 시각
- 미확정 참여, 미확인 리비전과 미해결 충돌

주문 완료 이벤트는 점검에 사용한 참여자와 최신 메뉴 리비전의 스냅샷 식별자를 함께 기록한다. 이후 그 스냅샷에 없던 유효 참여·메뉴 이벤트가 도착하면 해당 완료 표식을 그대로 안전한 완료로 표시하지 않는다.

- 14:30 전이면 주문 상태를 `진행 중 · 확인 필요`로 돌리고 새 정보를 확인한 뒤 다시 완료하게 한다.
- 14:30 이후면 Room을 다시 쓰기 가능하게 만들지 않고 종료 스냅샷을 다시 계산한다. 불일치가 남으면 `종료 · 동기화 불완전`을 표시한다.
- 데이터 불일치 Room은 성공 주문 히스토리를 자동 갱신하지 않는다. 이미 성공 히스토리를 만든 뒤 유효한 늦은 이벤트로 완료 조건이 깨지면 해당 성공 레코드를 무효화하고, 후속 대조로 다시 확정될 때만 복구한다.

다음 중 하나라도 해당하면 앱 안에서 주문 완료를 확정할 수 없다.

- 현재 참여자로 알려진 사용자 중 참여가 미확정이거나 메뉴가 미제출임
- 주문을 처리하는 기기가 현재 참여자 전원의 최신 ACK된 메뉴 리비전을 보유하지 않음
- 메뉴 또는 참여 이벤트가 `이 기기에 저장됨`이나 `확인 필요`임
- 해결되지 않은 충돌이 있음
- 현재 정상 응답 Peer가 해당 Room에 더 최신이거나 서로 다른 요약이 있음을 알렸지만 아직 대조가 끝나지 않아 참여자·최신 리비전 범위가 불명확함

과거에 Room 데이터를 보유하거나 ACK했지만 현재 오프라인인 Peer가 있다는 사실만으로 주문 완료를 막지는 않는다. 주문 기기에 위 조건을 만족하는 현재 참여자 스냅샷과 최신 ACK 리비전이 있으면 된다.

해당 Room과 무관한 Peer 또는 같은 Room 요약을 보유하지 않았다고 확인된 Peer의 미대조·미응답만으로 주문 완료를 막지 않는다.

다음 조건을 **모두** 만족하는 단독 Room만 원격 ACK 없이 완료할 수 있다.

- 현재 사용자가 Room 생성자이자 유일한 참여자다.
- Room이 한 번도 다른 Peer에 발견·공유되지 않았다.
- 알려진 원격 Peer, 원격 참여 또는 원격 ACK 이력이 없다.

한 조건이라도 만족하지 않으면 단독 Room이어도 ACK가 필요하다. 다인 Room에는 누락 위험을 무시하는 강제 완료 기능을 MVP에서 제공하지 않는다.

중앙 서버가 없으므로 한 번도 Peer에 전달되지 않은 채 기기가 잠자기·종료된 정보의 존재까지 알아낼 수는 없다. 앱은 이를 완전 동기화로 오인하지 않도록 마지막 확인 시각, 확인한 Peer 수와 미확인 상태를 노출한다.

## 5. 충돌 처리 원칙

- 사용자 본인의 참여와 메뉴는 다른 사용자가 덮어쓸 수 없다.
- 동일 사용자·동일 대상의 분기된 리비전은 사용자가 재확정하기 전까지 주문 전 점검을 통과하지 못한다.
- 동일 사용자의 두 Room 참여가 충돌하면 어느 쪽도 자동 채택하지 않고 둘 다 `참여 충돌`로 둔다. 사용자가 하나를 고르면 다른 Room의 참여와 메뉴를 철회 이벤트로 무효화한다.
- Room 생성자만 바꿀 수 있는 가게·링크·마감 정보는 생성자 기기의 단조 증가 리비전으로 정렬한다.
- 주문 완료와 되돌리기는 모든 참여자가 만들 수 있는 순서 이벤트로 복제한다. 14:30 전에는 모든 이벤트를 적용한 결정적 순서의 마지막 유효 동작을 사용한다.
- 같은 논리 순서의 충돌은 이벤트 ID 등 모든 Peer가 공유하는 결정적 보조 키로 정렬하되, 사용자에게 동시 변경 사실을 표시한다.
- 14:30 쓰기 종료는 쓰기 가능 여부에서 복제된 이벤트보다 우선한다. 단, 검증된 14:30 이전 이벤트는 늦게 도착해도 열람 전용 최종 스냅샷 계산에 포함한다.
- Peer와 macOS 시스템 시각 차이가 허용 범위를 벗어나면 `기기 시각 확인 필요`로 표시하고 참여 승인, 마감 변경, 주문 상태 변경처럼 시간에 민감한 동작을 막는다.
- 시계 차이가 과도해도 안전을 위해 14:30 쓰기 종료는 적용한다. 종료 스냅샷의 유효성·완전성은 자동 확정하지 않고 `확인 필요`로 둔다.
- 기기 시계 차이 또는 임의 변경 때문에 14:30 이전 생성 여부를 검증할 수 없는 늦은 이벤트는 `확인 필요`로 두고 완료 분류나 성공 히스토리를 자동 정정하지 않는다. 허용 오차는 사용자 환경을 고려해 여유 있게 잡되 정확한 값과 검증 방식은 출시 전 시간 경계 스파이크에서 확정한다.
- Room 취소와 마지막 참여자 철회가 만든 삭제 표식은 해당 Room의 참여·메뉴와 성공 히스토리를 무효화한다.
- 삭제 표식이 유지되는 동안 이전 데이터가 다시 들어와도 부활시키지 않는다.

## 6. 보존 범위

### 6.1 구조화된 운영 이벤트와 삭제 표식

- 현재 일일 세션과 직전 13개 일일 세션, 최대 14일의 구조화된 운영 이벤트와 삭제 표식을 암호화해 로컬에 보존할 수 있다.
- 14일이 지난 운영 이벤트와 삭제 표식은 다음 앱 실행 또는 정기 정리 시 삭제한다.
- 14일은 모든 Peer의 즉시 삭제를 보장하는 기한이 아니라 각 기기가 독립적으로 수행하는 최대 로컬 관리 창이다.
- 장기간 오프라인이었던 Peer가 만료된 이전 이벤트를 다시 제시하면 이미 종료된 일일 세션의 활성 상태를 되살리지 않는다.
- 삭제 표식이 만료된 뒤 오래된 Peer에서 비핵심 데이터가 다시 보이거나 고아 데이터가 남을 수 있음을 허용한다.
- 취소되지 않은 종료 Room의 구조화된 참여자·메뉴·최종 상태와 시스템 이벤트는 모든 인식된 회사 WiFi Peer가 최대 14일 동안 열람 전용으로 재구성할 수 있다.
- 최종 대조가 불완전해도 현재 확인된 상세는 표시한다. 이후 의미 있는 자동 트리거나 수동 새로고침으로 유효한 늦은 이벤트를 받으면 종료 스냅샷을 다시 계산할 수 있다.
- 불완전 종료는 성공 히스토리를 만들지 않는다. 후속 대조 결과에 따라 성공 히스토리를 생성·갱신하거나 기존 성공 레코드를 무효화하며, 일일 상태가 `종료`이고 주문 상태가 `주문 완료`로 확정된 때에만 유지한다.
- 이 상세 기록은 재사용 히스토리와 분리하며 채팅 본문을 포함하지 않는다.

### 6.2 채팅

- Room과 라운지의 채팅 본문은 프로세스 메모리에만 보관하며 평문 로컬 데이터베이스, 파일, 검색 인덱스나 로그에 기록하지 않는다.
- 채팅 캐시는 현재 운영일의 메시지만 보유한다. 다음 운영일을 시작할 때 이전 운영일 채팅은 메모리에서도 제거한다.
- 앱 정상 종료와 비정상 종료 뒤 다음 실행 과정에서 해당 기기의 채팅 캐시는 비어 있어야 한다.
- 앱을 다시 실행한 날이 현재 활성 일일 세션이면, 살아 있는 Peer가 가진 **당일** 채팅을 다시 받을 수 있다. 해당 Peer가 없으면 복구를 보장하지 않는다.
- 채팅 메시지는 전역적으로 충돌 가능성이 충분히 낮은 메시지 ID를 가지며, 재수집할 때 같은 ID를 중복 표시하지 않는다.
- 동일한 메시지 집합을 보유한 Peer끼리는 가능한 범위에서 논리 시각·발생 시각과 메시지 ID를 사용해 결정적 순서를 계산한다. 완전 복구나 모든 Peer의 동일한 전역 순서는 보장하지 않는다.
- 이전 일자의 채팅을 재수집해 장기 히스토리로 복원하지 않는다.
- 14:30 이후 현재 메모리에 남아 있는 당일 채팅은 열람만 가능하다. 앱 종료 뒤에는 다시 보인다고 보장하지 않는다.
- OS의 메모리 압축·스왑·진단 덤프까지 앱이 완전히 통제할 수 없다는 한계는 [보안·신뢰 경계 정책](./03_security_and_trust.md)에 명시한다.

### 6.3 재사용 히스토리

재사용 히스토리는 운영 이벤트와 별도의 최소 로컬 레코드다.

| 저장 필드 | 용도 |
|-----------|------|
| 가게 이름 | 히스토리 표시와 새 Room 기본값 |
| 배민 링크 | 새 Room의 QR 생성과 링크 기본값 |
| 마지막 주문 완료 일시 | 최근 이용 순 정렬 |
| 로컬 이용 횟수 | 자주 이용한 가게 정렬 |

- 채팅, 참여자, 개인 메뉴, 완료 처리자와 Room 시스템 이벤트는 재사용 히스토리에 저장하지 않는다.
- 재사용 히스토리는 14일 운영 이벤트 창이 지나도 각 기기에 암호화해 지속 보관한다.
- 앱 재실행 시 로컬 히스토리를 유지하고, 인식된 Peer와 가게 이름·배민 링크·최근 주문 완료 일시를 합칠 수 있다.
- 동일 가게 키는 `trim(가게 이름) + 사용자가 입력한 링크 원문`이 모두 완전히 일치할 때만 성립한다. 링크 원문에는 공백도 포함하며 URL 파싱, 리디렉션 추적, 대소문자·슬래시·공백 정규화 또는 링크 유효성 검사를 하지 않는다.
- 동일 가게 키의 최근 주문 완료 일시는 더 나중인 값으로 갱신한다.
- 로컬 이용 횟수는 해당 기기가 관찰한 완료 횟수다. Peer의 횟수를 더하거나 덮어쓰지 않으며 기기별로 달라도 허용한다.
- 중앙 정본이 없으므로 모든 사용자에게 동일한 히스토리를 보장하지 않는다.
- 사용자는 자신의 로컬 히스토리를 삭제할 수 있다. 삭제 정보는 14일 동안 Peer에 전파할 수 있지만 모든 기기의 영구 삭제를 보장하지 않으며 나중에 고아 레코드가 다시 나타날 수 있다.

## 7. 동기화 관측 가능성

Room과 연결 화면은 최소한 다음 정보를 제공해야 한다.

- 현재 발견한 Peer 수
- 마지막 전체 대조 시각
- 대조 진행 중 여부
- 로컬에만 있는 참여·메뉴 변경 수
- 충돌 또는 복구가 필요한 Room
- 수동 새로고침

Connection / Settings는 Peer별 닉네임, 온라인·오프라인, 마지막 확인 시각과 동기화 상태를 목록으로 보여준다. 닉네임이 겹칠 때만 짧은 기기 ID를 덧붙인다. 특정 Room 데이터를 보유·ACK했는지 또는 별도 신뢰 상태는 표시하지 않는다.

개발·진단 로그에는 메시지 본문, 메뉴 이름·옵션, 배민 링크, 참여자 표시 이름을 남기지 않는다. 진단은 이벤트 ID, 상태 코드, 소요 시간과 익명화된 기기 식별자만 사용한다.
