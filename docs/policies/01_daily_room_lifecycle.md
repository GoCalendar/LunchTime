# POL-01. 일일·방 생명주기

> 상태: 정본 초안
>
> 관련 결정: D-06~D-11, D-13~D-14, D-26, D-29~D-33, D-38, D-40, C-03, C-05

## 1. 일일 세션

LunchTime의 하루는 한국 시간 기준 `11:00:00`부터 `14:29:59`까지 쓰기 가능한 일일 세션으로 운영한다.

| 일일 상태 | 시간 | 허용 동작 |
|-----------|------|-----------|
| `대기` | 11:00 전 | 과거 히스토리 열람과 설정만 허용한다. 당일 Room 생성·참여와 라운지 쓰기는 허용하지 않는다. |
| `활성` | 11:00 이상 14:30 미만 | Room 생성·참여·메뉴·채팅·주문 완료와 라운지 쓰기를 정책에 따라 허용한다. |
| `최종 확정 중` | 14:30부터 최대 120초 | 사용자 쓰기를 막고 Peer 대조로 종료 스냅샷을 계산한다. |
| `종료` | 최종 대조 완료 또는 120초 경과 | 당일 Room과 라운지를 열람 전용으로 유지한다. |

- 14:30 쓰기 종료는 특정 Peer가 전송하는 이벤트에 의존하지 않고 각 기기가 현재 시각으로 계산하는 **파생 상태**다.
- Peer 간 시계 차이가 허용 범위를 넘더라도 안전을 위해 14:30 쓰기 종료는 적용한다. 다만 종료 스냅샷의 유효성·완전성은 자동 확정하지 않고 `확인 필요`로 둔다.
- 14:30에 기기가 잠자기, 종료 또는 네트워크 단절 상태였더라도 다음 실행·복귀 시 쓰기를 다시 열지 않고 즉시 `최종 확정 중`을 적용한다. 해당 복귀 시점부터 최대 120초간 대조한 뒤 종료 스냅샷을 계산한다.
- 120초 안에 대조를 끝내지 못하면 `종료 · 동기화 불완전`으로 표시하며, 후속 동기화 규칙은 [복제·정합성·보존 정책](./02_replication_consistency_retention.md)을 따른다.
- MVP는 11:00 전사 알림을 포함하지 않는다. 11:00 점심 선택 알림은 후속 기능이다.
- 일일 세션 밖에서 지난 Room을 다시 쓰기 가능 상태로 되돌리는 기능은 제공하지 않는다. 필요한 경우 히스토리로 새 Room을 만든다.

## 2. Room이 나타내는 것

- 하나의 Room은 하나의 가게와 그 가게에서 함께 주문할 사람들의 주문 단위를 나타낸다.
- 이 정책의 `사용자`와 본인 권한은 표시용 닉네임이 아니라 [보안·신뢰 경계 정책](./03_security_and_trust.md)의 설치·기기 결합 사용자 ID로 판정한다.
- MVP에는 가게 후보 투표, 득표 수, 투표 종료 또는 최종 가게 선정 상태가 없다.
- 사용자의 `참여`는 해당 가게 주문에 포함되겠다는 의사다. 단순 선호 투표로 해석하지 않는다.
- Room 생성자는 자동으로 첫 참여자가 된다. 이미 다른 Room에 참여 중이면 현재 참여를 철회하기 전까지 새 Room을 만들 수 없다.
- 참여자가 한 명뿐인 Room도 유효하며 주문 완료 처리할 수 있다.
- 한 사용자는 같은 운영일에 하나의 Room에만 참여한다.
- 다른 Room으로 이동하려면 주문 상태가 `진행 중`일 때 현재 참여를 먼저 철회해야 한다. 철회 이벤트에는 자신의 활성 메뉴 전체를 무효화하는 삭제 표식을 포함한다.
- 동일 사용자의 참여가 두 Room에서 충돌하면 어느 참여도 자동 채택하지 않는다. 사용자가 하나를 선택할 때까지 양쪽을 `참여 충돌`로 표시하고 메뉴 제출과 주문 완료를 차단한다. 선택하지 않은 Room의 참여와 메뉴는 철회 처리한다.
- 마지막 참여자가 철회하면 Room을 즉시 `모집 취소`로 전환한다. 이후 신규 참여와 모든 수정을 막고 14:30까지 읽기만 허용한다.

### Room 생성 필수값

- 가게 이름
- 배달의민족 공유 링크
- 방별 주문 마감 시각

주문 마감 시각은 생성 시각보다 늦고 당일 14:30보다 빨라야 한다. MVP의 Room 식별 정보는 가게 이름이며 별도 사용자 정의 제목이나 메모는 두지 않는다.

## 3. 독립된 상태 축

Room을 하나의 순차 상태값으로만 표현하지 않고 다음 축을 독립적으로 계산한다.

| 축 | 값 | 의미 |
|----|----|------|
| 일일 상태 | `대기` / `활성` / `최종 확정 중` / `종료` | 11:00 활성화, 14:30 쓰기 종료와 최종 대조 여부 |
| 모집 상태 | `모집 중` / `모집 마감` / `모집 취소` | 신규 참여 가능 여부와 Room 취소 여부 |
| 주문 상태 | `진행 중` / `주문 완료` | 실제 주문이 완료되었다는 현재 표식 |
| 동기화 상태 | `이 기기에 저장됨` / `복제됨` / `동기화됨` / `확인 필요` | 현재 기기가 데이터 전달을 얼마나 확인했는지 |

`동기화 중`은 상태값이 아니라 대조가 실행 중임을 나타내는 일시적인 활동 표시다. UI의 단일 상태 아이콘은 이 축들을 요약할 수 있지만 원본 상태를 잃어서는 안 된다. 동기화 이상은 주문 상태보다 우선해 경고한다.

## 4. 상태 전이

```text
11:00 일일 활성
  └─ Room 생성 → 모집 중 + 주문 진행 중
       ├─ 방별 주문 마감 시각 → 모집 마감 + 주문 진행 중
       ├─ 전체 취소 또는 마지막 참여자 철회 → 모집 취소 + 열람 전용
       ├─ 주문 완료 처리 → 주문 완료(14:30 전까지 되돌릴 수 있음)
       │    └─ 완료 되돌리기 → 주문 진행 중
       └─ 14:30 → 사용자 쓰기 종료 + 최대 120초 최종 대조 → 일일 종료
```

### 4.1 모집 마감

- 방별 주문 마감 시각이 되면 신규 참여만 닫는다.
- 참여 요청은 요청 기기가 마감 전에 다른 Peer의 유효한 승인 ACK를 받아야 확정된다. 이렇게 확정된 참여 이벤트는 다른 Peer에 마감 뒤 늦게 도착해도 유효하며, 요청 기기가 마감까지 ACK를 받지 못한 요청은 참여 실패로 처리한다.
- 기존 참여자는 14:30 전까지 채팅을 계속할 수 있고, 주문 상태가 `진행 중`일 때 자신의 메뉴 수정과 참여 철회를 계속할 수 있다.
- 마감 뒤의 메뉴 수정과 참여 철회는 주문 누락·오주문 위험이 크므로 일반 변경보다 강한 변경 알림과 동기화 확인을 요구한다.
- 마감 시각이 지난 Room은 다시 모집 중으로 되돌리지 않는다.

### 4.2 주문 완료

- 참여자 누구나 주문 완료를 요청할 수 있으며 생성자나 별도 파티장 권한을 요구하지 않는다.
- 주문 완료 전에는 [복제·정합성 정책](./02_replication_consistency_retention.md)의 주문 전 점검을 통과해야 한다.
- 14:30 전의 주문 완료는 취소나 주문 변경을 고려한 **되돌릴 수 있는 표식**이다.
- 주문 완료 상태에서는 참여, 가게 정보, 마감 시각과 개인 메뉴의 변경을 잠근다. 방 채팅은 14:30 전까지 계속 사용할 수 있다.
- 참여자 누구나 14:30 전에 주문 완료를 되돌릴 수 있다. 되돌리면 주문 상태는 `진행 중`이 되고, 방별 모집 마감 여부는 원래 마감 시각에 따라 유지한다. 이미 마감된 모집은 연장하거나 다시 열지 않는다.
- 완료와 되돌리기는 누가 언제 실행했는지 Room 시스템 이벤트로 모든 Peer에 표시한다.

### 4.3 Room 모집 취소

- Room 생성자는 자신이 현재 참여 중이고 주문 상태가 `진행 중`일 때 확인 과정을 거쳐 전체 Room을 취소할 수 있다.
- 주문 완료 상태라면 먼저 주문 완료를 되돌린 뒤 별도 취소 이벤트를 만들어야 한다.
- 전체 취소는 모든 참여와 메뉴를 무효화하고 `모집 취소`로 전환한다. 이후 신규 참여, 상태 변경과 채팅 쓰기를 막고 14:30까지 기록만 보여준다.
- 마지막 참여자의 철회도 같은 `모집 취소` 전이를 만든다.
- 생성자가 참여를 철회했지만 다른 참여자가 남아 있으면 생성자의 가게·링크·마감 수정과 전체 취소 권한은 종료된다. Room 정보는 잠기며 남은 참여자는 기존 정보로 주문을 계속할 수 있다.
- 종료된 생성자 관리 권한은 같은 운영일에 다시 참여해도 복구되지 않고 다른 참여자에게 이전되지 않는다.
- 취소 Room은 최근 Room 상세와 재사용 히스토리에 포함하지 않는다.

### 4.4 14:30 자동 종료와 최종 대조

- 14:30이 되면 모집·참여·메뉴·가게·주문 상태·Room 채팅·라운지 채팅의 모든 사용자 쓰기를 금지한다.
- 14:30부터 최대 120초간 알려진 데이터를 대조하고 종료 스냅샷을 계산한다.
- 120초는 최종 확정 과정 전체의 바깥 한도다. 그 안의 개별 동기화 세션도 최대 3회·총 30초 제한을 지키며, 새 Peer 발견처럼 의미 있는 진전 없이 같은 실패 대상을 반복 호출하지 않는다.
- 대조가 끝났을 때 유효 주문 상태가 `주문 완료`면 일일 상태를 `종료`로 확정하고 재사용 히스토리를 갱신한다.
- 유효 주문 상태가 `진행 중`이면 일일 상태를 `종료`로 확정하되 주문 상태는 `진행 중`으로 유지하며 재사용 히스토리에 성공 주문으로 추가하지 않는다.
- 120초 안에 대조하지 못하면 `종료 · 동기화 불완전`으로 표시하고 성공 히스토리를 갱신하지 않는다.
- 14:30 이후 새 사용자 변경을 만들 수 없다.
- 네트워크 지연으로 늦게 도착했더라도 마감 전 승인 ACK 또는 14:30 전 유효 생성이 검증된 이벤트는 후속 동기화에서 장부와 종료 스냅샷에 반영할 수 있다. Room은 계속 열람 전용이다.
- 14:30 이후 생성됐거나 생성 시각을 검증할 수 없는 변경을 종료 이전 동작으로 소급 적용하지 않는다.
- 실제 주문 취소나 변경을 14:30 이후 기록해야 한다면 새 Room 또는 향후 별도 정정 기능을 사용한다.

## 5. 참여와 권한

| 동작 | 권한 |
|------|------|
| Room 생성 | 일일 활성 시간이며 다른 Room에 참여 중이지 않은 인식된 사내 사용자 |
| 참여·참여 철회 | 해당 사용자 본인 |
| 개인 메뉴 등록·수정·삭제 | 해당 메뉴의 참여자 본인 |
| 가게 이름·배민 링크·마감 시각 수정 | 현재 참여 중인 Room 생성자, 주문 진행 중이며 기존 마감 전 |
| 주문 완료·되돌리기 | Room 참여자 누구나 |
| Room 전체 취소 | 현재 참여 중인 Room 생성자, 주문 진행 중 |
| 14:30 자동 종료 | 사용자 권한이 아닌 시간 기반 파생 동작 |

- 생성자는 자신이 참여 중이고 주문 상태가 `진행 중`일 때 기존 마감 전까지만 마감 시각을 변경할 수 있다. 새 마감 시각도 현재 시각보다 늦고 14:30보다 빨라야 한다.
- 가게 이름이나 배민 링크 변경도 생성자가 참여 중이고 주문 상태가 `진행 중`이며 14:30 전일 때만 가능하며 전체 참여자에게 변경 사실을 표시한다.
- 생성자가 철회하면 위 관리 권한은 영구 종료된다. 같은 Room에 다시 참여해도 복구되지 않고 다른 사용자에게 이전하지 않는다.

## 6. 개인 메뉴 템플릿

개인 메뉴는 하나 이상의 메뉴 항목으로 구성하며 각 항목은 다음 고정 구조를 사용한다.

| 필드 | 필수 여부 | 규칙 |
|------|-----------|------|
| 메뉴 이름 | 필수 | 사용자가 직접 입력하는 한 줄 문자열 |
| 옵션 | 선택, 반복 가능 | 선택한 순서를 보존하는 한 줄 문자열 목록 |

예시:

```text
메뉴 이름: 다운타우너 버거 세트
옵션:
- (D) 오리지널 프라이즈 +2000
- (D) 코카콜라 제로 255ml
- (D) 싱글
```

- `메뉴 이름`과 `옵션`이라는 구조는 고정하지만 그 안의 문자열을 배민 데이터와 대조하거나 자동 보정하지 않는다.
- 메뉴를 단일 자유 형식 본문으로 저장하지 않는다.
- 참여자는 하나 이상의 메뉴를 등록해야 주문 전 점검을 통과한다.
- 함께 주문하지 않으려면 별도의 `주문하지 않음` 메뉴 상태를 남기지 않고 참여를 철회한다.
- 수정은 전체 메뉴 등록의 새 리비전으로 원자적으로 전파한다. 일부 옵션만 전달된 중간 상태를 다른 Peer에 확정 상태로 보여주지 않는다.

## 7. 히스토리 재사용

- 최종 대조를 통과해 일일 상태가 `종료`이고 주문 상태가 `주문 완료`인 Room만 재사용 히스토리를 생성하거나 갱신한다.
- 히스토리의 목적은 자주 이용하는 가게로 새 Room을 빠르게 만드는 것이다.
- 재사용 동작은 가게 이름과 배민 링크가 채워진 Room 생성 화면을 연다.
- 사용자는 당일 주문 마감 시각을 확인·입력한 뒤 Room 생성을 확정한다. 히스토리 선택만으로 Room을 즉시 생성하지 않는다.
- 이전 참여자, 개인 메뉴, 채팅과 과거 Room의 완료 상태는 새 Room에 복사하지 않는다.
- 장기 저장 필드와 삭제 동작은 [복제·정합성·보존 정책](./02_replication_consistency_retention.md)을 따른다.
