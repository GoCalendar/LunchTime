# POL-03. 보안·신뢰 경계

> 상태: 정본 초안
>
> 관련 결정: D-20, D-28, D-34~D-36, D-38, C-04
>
> 목표: 지원 회사 WiFi를 MVP의 자동 신뢰 경계로 사용하되 전송 중 도청과 로컬 데이터 탈취 위험을 줄이고, 그 경계가 방어하지 못하는 내부 Peer 위험을 명시한다.

## 1. 지원 네트워크와 자동 신뢰 경계

- 회사가 제공하는 두 종류의 WiFi가 실제로 같은 내부망에서 Peer 간 통신을 허용하는 경우 둘 다 지원 네트워크로 본다.
- 지원 회사 WiFi에서 발견한 LunchTime Peer는 별도 사용자 승인이나 확인 코드 없이 자동 신뢰한다.
- 자동 신뢰는 Peer가 LunchTime 프로토콜을 사용하며 지원 네트워크에서 발견됐다는 뜻이다. 회사 구성원 개인의 신원을 강하게 증명한다는 뜻은 아니다.
- 인터넷, 외부 네트워크, VPN, 개인 핫스팟과 서로 라우팅되지 않는 사내망까지 연결을 보장하지 않는다.
- 지원 네트워크 밖에서 발견한 장치는 자동 신뢰하지 않으며 운영 데이터를 보내지 않는다.
- 인식된 Peer 목록은 사용자가 현재 통신 범위를 볼 수 있게 하는 관측 수단이지 접근 차단 수단이 아니다.

### 데이터 가시성과 쓰기 권한

| 데이터 | 읽기 범위 | 쓰기 범위 |
|--------|-----------|-----------|
| 오늘의 Room 이름·가게·상태·참여자 | 자동 신뢰 회사 WiFi Peer | 생성자·참여자별 권한에 따름 |
| 활성 Room의 개인 메뉴와 채팅 | 자동 신뢰 회사 WiFi Peer | 해당 Room 참여자 |
| 종료 Room의 14일 구조화 상세 | 자동 신뢰 회사 WiFi Peer | 열람 전용 |
| 라운지 채팅 | 자동 신뢰 회사 WiFi Peer | 자동 신뢰 회사 WiFi Peer |
| 재사용 히스토리 | 해당 기기 사용자 | 동기화 시 자동 신뢰 Peer와 최소 가게 정보만 교환 |

- 자동 신뢰 Peer는 참여 전에도 활성 Room의 가게, 참여자, 개인 메뉴와 채팅을 읽을 수 있다.
- Room의 메뉴·채팅 작성과 주문 상태 변경은 현재 Room 참여자에게만 허용한다.
- 종료 Room 상세는 Room 참여 여부와 관계없이 최대 14일 열람할 수 있으며 채팅은 포함하지 않는다.
- ACK는 특정 리비전을 저장했다는 확인이다. 별도의 신뢰 상태나 추가 열람 권한을 만들지 않는다.
- 라운지는 모든 자동 신뢰 Peer가 읽고 쓸 수 있다.

## 2. 기기 식별과 암호 키

- 각 설치는 최초 실행 시 기기별 비대칭 키 쌍과 안정적인 기기 ID를 생성한다.
- 각 설치는 닉네임과 별개인 불투명 사용자 ID도 생성하고 기기 키에 결합한다. MVP에서 이 사용자 ID가 참여, 메뉴 소유권, 하루 한 Room과 충돌 판정의 권한 주체다.
- MVP는 사용자 ID 하나를 설치·기기 하나에 연결하며 여러 Mac의 사용자 ID를 같은 사람으로 합치지 않는다. 기기 교체·재설치로 기존 식별 정보를 잃으면 새 사용자로 취급한다.
- 개인키와 로컬 데이터 암호화 키는 macOS Keychain에 저장하고 내보내기 가능한 설정 파일이나 소스 코드에 넣지 않는다.
- 모든 Peer 연결은 상대 기기가 세션에 사용한 개인키를 보유했음을 확인한 뒤 암호화된 데이터를 교환한다. 이는 채널 상대의 기기 연속성과 무결성을 위한 절차이며 회사 구성원 인증을 뜻하지 않는다.
- 공개키, 사용자 ID, 안정적인 기기 ID와 사용자가 설정한 닉네임의 연결을 로컬 Peer 목록에 보관한다.
- 닉네임은 중복될 수 있다. 같은 닉네임이 동시에 보일 때만 짧은 기기 ID를 함께 표시한다.
- 닉네임은 화면 표시값일 뿐 권한·소유권·중복 참여 판정 키로 사용하지 않는다.
- 자동 신뢰 정책 때문에 MVP에는 수동 승인 코드, 사용자별 신뢰 상태 또는 신뢰 관리 UI를 두지 않는다.
- 기기 키 교환·회전과 지원 네트워크 판정 방식은 출시 전 보안 스파이크에서 확정한다.

## 3. 전송 암호화

- Peer 간 발견 이후의 모든 애플리케이션 데이터는 인증된 암호화 채널로 전송한다.
- 채널은 기밀성, 무결성, 재전송 공격 방지와 가능하면 순방향 안전성을 제공해야 한다.
- 평문 HTTP, 평문 WebSocket, 서명 없는 브로드캐스트에 사용자 데이터를 실어 보내지 않는다.
- Peer 발견 패킷은 내용을 최소화하고 사용자 이름, Room 제목, 가게, 메뉴, 채팅과 입력 링크를 포함하지 않는다.
- 이벤트가 전송 중 바뀌지 않았는지 검증한다. 검증에 실패한 이벤트는 적용하거나 다시 전파하지 않는다.

전송 암호화는 같은 네트워크의 수동 도청을 줄이기 위한 것이다. 자동 신뢰된 내부 Peer가 정상 프로토콜을 사용해 받은 평문을 악용하는 상황까지 막지는 않는다.

## 4. 로컬 저장 암호화

- 구조화된 운영 이벤트, 삭제 표식, Peer 목록과 재사용 히스토리는 앱 전용 키로 암호화해 저장한다.
- 암호화 키는 데이터 파일과 분리해 Keychain에서 관리한다.
- 메뉴, 입력 링크, 참여자 닉네임과 히스토리 평문을 앱 로그, 크래시 메타데이터, Spotlight 인덱스, 알림 본문 미리보기나 임시 파일에 기록하지 않는다.
- 채팅 본문은 [복제·정합성·보존 정책](./02_replication_consistency_retention.md)에 따라 프로세스 메모리에만 유지한다.
- 사용자가 명시적으로 실행하는 메뉴 요약 복사처럼 클립보드에 정보가 나가는 동작은 사전에 알리고, 앱이 가능한 범위에서 짧은 만료 또는 지우기 동작을 제공한다.

## 5. QR 경계

- QR은 사용자가 입력한 문자열을 검증·정규화·경고 없이 기기 안에서 그대로 인코딩하며, QR 생성을 위해 외부 웹 서비스에 입력값을 전송하지 않는다.
- URL 형식, 도메인, 리디렉션 목적지나 실제 배민 앱 이동 여부를 LunchTime이 확인하지 않는다.
- MVP는 검증됨·안전함·공식 링크임을 뜻하는 라벨이나 잘못된 링크 경고를 표시하지 않는다.
- LunchTime의 책임 범위는 QR을 생성·표시하는 데까지다. 휴대폰에서 스캔한 뒤 어떤 앱이나 페이지가 열리는지는 외부 동작이다.
- QR은 화면을 볼 수 있는 주변 사람도 촬영할 수 있으므로 인증 수단이나 비밀 전달 수단으로 취급하지 않는다.
- QR에는 LunchTime 내부 Room 데이터, 사용자 식별자, 메뉴, 채팅 또는 암호화 키를 포함하지 않는다.

## 6. 암호화와 자동 신뢰가 보장하지 않는 것

다음 위험은 MVP 보안 경계 밖이다.

- 지원 회사 WiFi에 접속한 악의적인 LunchTime Peer가 자동 신뢰 범위의 데이터를 읽거나 자신의 유효 권한으로 동작하는 행위
- 닉네임을 다른 사람과 같게 설정하거나 허위 닉네임을 사용하는 행위
- 사용자가 화면을 캡처하거나 내용을 복사해 외부로 내보내는 행위
- 잠금 해제된 Mac 또는 탈취된 로그인 세션에서의 열람
- 악성 소프트웨어가 실행 중인 앱의 메모리나 UI를 읽는 행위
- macOS가 메모리를 압축·스왑하거나 시스템 진단 덤프를 만드는 모든 경우
- QR이 보이는 물리적 공간에서의 촬영
- 각 Peer의 독립 삭제 때문에 모든 복제본을 즉시·영구 삭제하는 것

따라서 UI나 문서는 회사 WiFi를 강한 사용자 인증으로 표현하지 않는다. `종단간 암호화` 또는 `완전 삭제`라는 표현도 실제 프로토콜과 OS 동작이 해당 표현을 충족하는지 별도 보안 검토를 통과하기 전에는 사용하지 않는다.

## 7. 출시 전 보안 게이트

MVP 출시 전 다음 결과가 필요하다.

- 두 사내 WiFi 사이의 실제 Peer 라우팅·발견 가능 범위
- 지원 회사 WiFi 자동 판정, 네트워크 밖 Peer 차단과 내부 악성 Peer 위협 모델
- 전송 프로토콜과 사용 암호 스위트의 보안 검토
- 기기 키 교환·회전과 재설치 시 식별 처리
- Keychain 접근 제어와 암호화 데이터 파일 복구·삭제 테스트
- 위조 이벤트, 재전송, 네트워크 밖 Peer와 중간자 공격 테스트
- 로그·크래시·알림·클립보드에서 평문 데이터가 남지 않는지 확인
- 앱 종료 뒤 채팅 평문이 앱 저장소에 남지 않는지 확인
