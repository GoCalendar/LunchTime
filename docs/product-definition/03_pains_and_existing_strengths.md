# 03. 문제와 유지할 강점

> 상태: 검토가 필요한 초안
>
> 입력: [02. 현재 점심 주문 여정](./02_current_lunch_journey.md)
>
> 역할: 현재 방식에서 개선할 문제와 새로운 제품에서도 유지해야 할 장점을 분리한다.

실제 사용자 반응을 관찰하지 않았으므로 `Wow Point`라고 단정하지 않고 `현재 방식의 강점`으로 표현합니다.

## 개선해야 할 문제

### P-01. 활성 모집과 상태를 파악하기 어렵다

- 어떤 모집이 아직 열려 있는지 한눈에 알기 어렵습니다.
- 누가 어느 모집에 참여했는지 대화에서 다시 확인해야 합니다.
- 후보 가게, 확정 가게, 각자 메뉴, 마감, 주문 완료가 하나의 상태로 정리되지 않습니다.

### P-02. 서로 다른 맥락이 섞인다

- 여러 점심 모집의 메시지가 서로 섞일 수 있습니다.
- 점심 이야기와 업무 대화가 같은 공간에서 경쟁합니다.
- 중요한 정보가 새로운 메시지에 밀립니다.

### P-03. 대화와 상태 관리 사이에 긴장이 있다

- 메뉴와 옵션은 O/X 선택으로 환원할 수 없어 대화가 필요합니다.
- 대화만으로 상태까지 관리하면 중요한 결정과 현재 상태가 묻힙니다.
- 가게 결정과 메뉴 수정의 최신 내용을 다시 확인해야 합니다.

### P-04. 개인 메뉴 취합이 구조화되어 있지 않다

- 각자의 메뉴와 옵션이 여러 메시지에 흩어집니다.
- 누가 메뉴를 제출했고 수정했는지 확인하기 어렵습니다.
- 실제 주문자가 모든 메뉴를 다시 모아야 합니다.

### P-05. Mac과 휴대폰 사이의 흐름이 끊긴다

- Mac에서 받은 배민 링크를 휴대폰으로 넘겨야 메뉴를 확인할 수 있습니다.
- 전달 방식이 제품 경험으로 정리되어 있지 않습니다.
- 공식 공개 API가 없다면 가게와 메뉴 정보를 LunchTime 안으로 가져오기 어렵습니다.

### P-06. 모집 마감과 주문 완료가 암묵적이다

- 언제 신규 참여가 닫히는지 명확하지 않습니다.
- 실제 주문자가 방 생성자로 고정되지 않아 완료 주체가 모호해질 수 있습니다.
- 주문 완료 여부가 참여자 모두에게 같은 상태로 공유되지 않습니다.

### P-07. 반복 주문의 맥락을 다시 활용하기 어렵다

- 이전에 이용한 가게를 자주 다시 주문하지만 과거 맥락이 새 모집의 출발점으로 구조화되어 있지 않습니다.
- 무엇을 재사용해야 가장 유용한지 아직 정하지 않았습니다.

### P-08. 운영 비용을 피하면서 공유 상태를 유지해야 한다

- 별도 서버를 운영하지 않는다는 제약과 여러 기기에서 동일한 상태를 보는 경험을 동시에 만족해야 합니다.
- P2P에서 호스트 이탈, 재접속, 늦은 참여와 충돌을 처리할 방법이 아직 검증되지 않았습니다.

## 유지해야 할 현재 방식의 강점

### S-01. 자연어 대화의 유연성

Slack 대화는 메뉴 선택 이유, 옵션과 조건을 자유롭게 협의할 수 있습니다.

### S-02. 배민 공유 링크

배민 공유 링크로 특정 가게를 다른 사람에게 전달할 수 있습니다.

### S-03. 자유 형식 메뉴 표현

자유 형식 텍스트는 복잡한 메뉴와 옵션을 그대로 표현할 수 있습니다.

### S-04. 유연한 역할 분담

방을 처음 만든 사람과 실제 주문자가 같을 필요가 없습니다.

### S-05. 익숙한 최종 주문 경험

이미 사용하는 배민 앱에서 최종 주문을 유지하면 새로운 주문·결제 방식을 학습하지 않아도 될 가능성이 높습니다. 이 항목은 아직 사용자 검증이 필요한 추론입니다.

### S-06. 반복 가게에 대한 조직의 기억

자주 주문하는 가게에 대한 경험이 이미 구성원에게 축적되어 있습니다. 이를 어떻게 제품에서 재사용할지는 미확정입니다.

## 제품 설계 원칙으로 전환

- 대화를 제거하지 말고 상태와 연결합니다.
- 배민을 대체하지 말고 배민으로 이동하는 경계를 매끄럽게 만듭니다.
- 자유 형식 메뉴를 유지하되 사람별로 구조화합니다.
- 역할을 고정하지 않되 중요한 상태 변경에는 수행자와 결과를 명확히 보여줍니다.
- 반복되는 정보를 재사용하되 이전 주문을 무조건 그대로 복제하지 않습니다.
- 서버가 없다는 이유로 사용자에게 불명확한 동기화 상태를 숨기지 않습니다.
