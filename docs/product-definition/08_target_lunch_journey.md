# 08. 목표 사용자 여정

> 상태: 검토가 필요한 경험 제안
>
> 입력: [07. 경험 및 정보 구조](./07_experience_structure.md)
>
> 역할: LunchTime에서 사용자가 어떤 순서로 무엇을 하고 어떤 가치를 얻는지 연결한다.

## 핵심 여정 요약

> 회사의 Mac 사용자가 동료들과 점심방을 열고, 대화로 가게를 정하고, 배민에서 실제 주문한 뒤 완료 기록을 남깁니다.

## A. 점심 모집과 주문

| 단계 | 사용자 행동 | 기대 경험 | Surface / 기능 |
|------|-------------|-----------|----------------|
| 1. 현황 확인 | 지원되는 앱 실행·연결 상태에서 검증된 macOS 진입 Surface로 진행 중인 방과 마감 시각을 봅니다. | Slack을 뒤지지 않고 현재 모집을 파악하며, 발견·알림이 불가능한 상태는 숨기지 않습니다. | S-01, S-06 / F-02~F-04, F-63 |
| 2. 모집 시작 | 제목과 필수 주문 마감 시각을 정해 새 방을 엽니다. | 적은 입력으로 모집을 시작합니다. | S-02 / F-09~F-12 |
| 3. 발견·참여 | 동료가 알림 또는 목록에서 방을 발견하고 참여합니다. 늦게 참여했다면 현재 상태 스냅샷과 합의된 범위의 과거 채팅을 받습니다. | 누가 함께 주문하는지와 지금까지의 맥락이 즉시 보입니다. | F-06, F-13~F-15, F-21, F-62 |
| 4. 협의 | 참여자들이 방별 채팅으로 의견을 나눕니다. | 업무·다른 모집 대화와 섞이지 않습니다. | F-19~F-20 |
| 5. 가게 확인 | 후보와 배민 링크를 올리고, 검증된 한 가지 방식으로 링크를 Mac에서 휴대폰으로 전달해 배민 앱에서 확인합니다. | 웹을 지원하지 않는 실제 주문 환경으로 자연스럽게 이어집니다. | F-22~F-28, F-61 |
| 6. 가게 결정 | 대화로 결정하거나 필요할 때 투표하고 최종 가게를 표시합니다. | 메뉴 결정을 O/X로 강제하지 않습니다. | F-29~F-32 |
| 7. 개인 메뉴 등록 | 각자 메뉴와 옵션을 자유 형식으로 적습니다. | 주문자가 채팅에서 메뉴를 다시 모으지 않습니다. | F-34~F-37 |
| 8. 주문 마감 | 방 생성 시 설정한 시각에 신규 참여가 자동으로 닫힙니다. 기존 참여자는 방과 채팅을 계속 이용합니다. | 마감 여부를 사람이 반복 공지하지 않으면서 주문 협의를 이어갈 수 있습니다. | F-11, F-17, F-19~F-20 |
| 9. 외부 주문·완료 | 한 사람이 배민에서 주문하고 참여자 누구나 완료 처리합니다. | 주문자가 방 생성자인지와 무관하게 흐름을 끝냅니다. | F-25, F-37~F-40 |
| 10. 기록 확인 | 완료된 방이 히스토리에 남습니다. | 다음 주문에서 과거 맥락을 참고할 수 있습니다. | F-41~F-42 |

마감 후 개인 메뉴 수정 허용 여부와 수정 내용을 주문자에게 어떻게 알릴지는 Policies에서 결정할 질문으로 남깁니다.

## B. 참여자 관점

1. 새 모집 알림이나 활성 방 목록에서 방을 발견합니다.
2. 방 요약을 보고 참여합니다.
3. 방별 채팅에서 조건과 가게 의견을 확인합니다.
4. 검증된 전달 방식으로 배민 링크를 Mac에서 휴대폰으로 넘겨 메뉴를 확인합니다.
5. 자신의 메뉴와 옵션을 자유 형식으로 등록하고 필요하면 수정합니다.
6. 주문 마감 뒤에도 방과 채팅에서 협의를 이어가고 주문 완료 알림을 받습니다.

## C. 반복 가게로 다시 모집하기

이 여정은 아직 제품 방향이 아니라 검증할 후보입니다.

여기서 `한 번의 동작`은 재모집 흐름에 진입하는 동작을 뜻하며, 방 생성 완료까지 클릭이 한 번이라는 의미는 아닙니다. 다음 두 대안을 비교해 결정합니다.

| 대안 | 동작 | 장점 | 결정할 점 |
|------|------|------|-----------|
| A. 즉시 생성 | 과거 주문에서 재모집을 누르면 기본 주문 마감 시각으로 방을 즉시 만들고, 필요하면 이후 마감 시각을 수정합니다. | 가장 빠르게 모집을 시작합니다. | 안전한 기본 마감 시각, 생성 직후 수정 가능 범위, 잘못 만든 방의 처리 |
| B. 미리 채운 생성 화면 | 과거 가게명·배민 링크 등을 채운 새 방 화면을 열고, 사용자가 주문 마감 시각과 복사 정보를 확인한 뒤 생성합니다. | 생성 전에 필수 마감 시각과 복사 내용을 명확히 확인합니다. | 확인 단계가 반복 사용의 속도를 얼마나 늦추는지 |

두 대안 모두 히스토리의 `재모집` 한 번으로 해당 흐름에 진입합니다. 이전 참여자와 개인 메뉴는 자동 복사하지 않고 필요할 때 참고하는 방향을 우선 제안하며, 최종 복사 범위는 별도로 결정합니다.

관련 기능: F-41~F-44

## D. 라운지

라운지는 주문 거래 여정과 섞지 않는 별도 반복 경험입니다.

1. 사용자가 검증된 macOS 진입 Surface에서 라운지를 엽니다.
2. 점심방과 무관한 잡담을 주고받습니다.
3. 읽지 않은 활동과 알림을 점심방과 별도로 제어합니다.
4. 활성 점심방으로 다시 이동합니다.

관련 기능: F-45~F-48

## 실패·복구 여정에서 확인할 것

- 주변 Peer를 발견하지 못했을 때 무엇을 보여주나요?
- 앱이 종료됐거나 백그라운드 연결이 끊긴 동안 놓친 모집과 알림을 어떻게 설명하나요?
- 방 생성자가 앱을 종료했을 때 남은 사용자는 무엇을 보나요?
- 잠자기나 네트워크 전환 후 어떤 상태로 복구되나요?
- 늦게 참여한 사용자는 F-21의 현재 상태 스냅샷과 F-62의 과거 채팅을 각각 어느 범위까지 받나요?
- 두 사용자가 동시에 가게, 메뉴 또는 완료 상태를 바꾸면 어떻게 되나요?
- 완료된 히스토리가 기기마다 다를 때 이를 어떻게 설명하나요?

이 실패 여정은 P2P 기술 검증과 Policies 작성 전에 해결해야 합니다.
