# 08. 목표 사용자 여정

> 상태: 2026-07-24 회의 결정을 반영한 목표 여정
>
> 입력: [07. 경험 및 정보 구조](./07_experience_structure.md)
>
> 역할: LunchTime에서 11:00~14:30 동안 사용자가 무엇을 하고, 종료·단절 뒤 어떻게 복구하는지 연결한다.

## 핵심 여정 요약

> 회사의 Mac 사용자가 MenuBar에서 한 가게의 Room을 발견해 마감 전 승인 ACK로 참여하고, 입력 링크의 QR을 휴대폰으로 스캔한 뒤 외부 배민 앱·페이지에서 메뉴 확인을 시도하고 구조화된 개인 메뉴를 누락 없이 공유합니다. LunchTime은 QR 이후 외부 동작을 보장하지 않습니다. 주문 완료는 취소 상황에 맞춰 되돌릴 수 있고, 14:30에 사용자 쓰기를 닫은 뒤 최대 120초 finalization을 수행합니다.

## B. 메뉴 누락을 막는 참여자 여정

### 여러 명이 참여한 Room

1. 참여자가 메뉴명과 옵션을 저장합니다.
2. 화면은 우선 `이 기기에 저장됨`을 표시하고 `동기화 중`은 별도 활동으로 보여줍니다.
3. 다른 신뢰 Peer가 최신 참여·메뉴 리비전을 받으면 ACK 상태를 돌려줍니다.
4. ACK를 확인하면 `복제됨`, 건강한 대상 Peer와 대조가 끝나면 `동기화됨`으로 표시합니다.
5. 주문 전 점검은 현재 참여자 모두에게 `제출 확인` 메뉴가 있는지 검사합니다.
6. 한 명이라도 미제출·`이 기기에 저장됨`·`확인 필요`이면 주문 완료 동작을 막고 해결할 대상을 보여줍니다.

### 한 명만 참여한 Room

로컬 단독 예외는 생성자가 처음부터 만들었고 한 번도 다른 Peer에 공유되지 않은 Room에만 적용합니다. 한 번이라도 공유된 Room이거나 다른 참여자가 있었던 Room은 현재 한 명만 남아도 일반 ACK 규칙을 따릅니다.

## C. 잠자기·네트워크 단절 뒤 복귀

1. Mac이 잠자거나 네트워크에서 이탈한 동안 다른 Peer가 참여, 메뉴, 주문 상태와 채팅을 변경합니다.
2. 사용자의 Mac이 깨어나거나 앱이 전면으로 오거나 네트워크가 바뀌거나 새 Peer가 연결됩니다.
3. 앱이 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 끝나는 제한 세션을 시작해 건강한 Peer의 버전 요약을 대조하고 누락된 14일 이내 변경을 요청합니다.
4. 중복을 제거하고 충돌을 판정한 뒤 Room 상태를 다시 계산합니다.
5. `이 기기에 저장됨/복제됨/동기화됨/확인 필요` 상태와 별도 `동기화 중` 활동을 보여줍니다.
6. 실패한 세션은 멈추고 새 의미 있는 트리거나 사용자의 수동 요청만 새 제한 세션을 시작합니다.
7. 11:00~14:30에 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 대상 Peer가 있을 때만 30초 간격으로 anti-entropy 대조합니다. 대상이 없거나 14:30이면 멈추고, 제한을 소진한 실패 Peer는 정기 타이머만으로 재시도하지 않습니다.

연결 가능한 Peer가 없거나 모든 Peer가 이미 데이터를 삭제했다면 복구를 보장하지 않습니다. 앱은 이전 상태를 최신으로 추측하지 않고 `확인 필요`로 표시합니다. 14:30에 잠들어 있던 Mac은 wake 시 즉시 종료 상태를 적용하고 그때부터 최대 120초 finalization을 수행합니다.

## D. 반복 가게로 다시 모집하기

1. 사용자가 History Window에서 자주 이용한 가게를 찾습니다.
2. `다시 모집`을 선택합니다.
3. 가게명과 배민 링크가 채워진 새 Room Sheet가 열립니다.
4. 사용자가 주문 마감 시각과 복사된 내용을 확인합니다. 이 확인 전에는 Room을 즉시 만들지 않습니다.
5. 새 Room을 생성하고 생성자 본인이 참여합니다.

이전 참여자, 개인 메뉴와 채팅은 복사하지 않습니다. 히스토리는 가게명, 입력 링크, 최근 주문 완료 일시와 로컬 이용 횟수만 장기 보존합니다. 동일 가게는 `trim(storeName) + raw inputLink`가 모두 완전히 같을 때만 병합하며 입력 링크는 trim·URL 정규화·redirect 추적을 하지 않습니다.

관련 기능: F-41~F-44, F-52, F-59

## E. 라운지

라운지는 주문 여정과 섞지 않는 별도 MVP 경험입니다.

1. 사용자가 전체 앱 창에서 라운지를 엽니다.
2. 11:00~14:30에 점심방과 무관한 잡담을 주고받습니다.
3. 14:30이 되면 메시지 입력이 닫히고 현재 메모리의 당일 대화를 읽기만 합니다.
4. 앱을 종료하면 해당 기기의 라운지 채팅이 삭제됩니다.
5. 같은 날 재실행하면 살아 있는 Peer의 당일 메시지를 다시 받을 수 있지만 복구는 보장하지 않습니다.

MenuBar 라운지 바로가기, 읽지 않음 배지와 메시지 알림은 후속 범위입니다.

관련 기능: F-45~F-48, F-51, F-62

## F. 주문 취소와 14시 30분 경계

1. 참여자가 실제 배민 주문 후 `주문 완료`를 표시합니다.
2. 배민에서 주문이 취소되면 어떤 참여자든 14:30 전에 완료를 되돌립니다.
3. 되돌린 Room은 다시 `진행 중`으로 보이며 메뉴와 채팅을 계속 이용합니다.
4. 14:30에는 현재 주문 상태와 무관하게 사용자 쓰기가 자동 종료되고 최대 120초 finalization을 시작합니다.
5. 종료 뒤에는 Room, 메뉴, 주문 상태와 채팅을 수정할 수 없습니다.

`주문 완료`는 일일 종료가 아니라 가역적인 주문 상태이고, `14시 30분 자동 종료`가 사용자 쓰기의 최종 상태입니다.

## G. 참여 철회·Room 취소와 충돌

1. 주문 상태가 `진행 중`이면 참여자는 자신의 참여를 철회할 수 있습니다.
2. 마지막 참여자가 철회하면 Room은 취소되고 14:30까지 열람 전용으로 남지만 Recent Room과 재사용 히스토리에는 포함되지 않습니다.
3. 현재 참여 중인 생성자는 진행 중 Room을 전체 취소할 수 있고, 주문 완료 상태라면 먼저 완료를 되돌립니다.
4. 생성자가 철회하면 관리자 권한을 잃고 가게명·입력 링크·마감 메타데이터는 잠깁니다.
5. 같은 사용자의 여러 Room 참여가 분산 충돌로 동시에 유효해지면 양쪽을 `확인 필요`로 표시하고 두 Room의 주문 완료를 막습니다.
6. 사용자가 유지할 Room을 직접 선택하면 선택하지 않은 Room의 참여와 활성 메뉴를 철회 이벤트로 함께 정리합니다.

## 실패·복구 여정에서 기술 스파이크로 확인할 것

- 두 사내 Wi-Fi 사이에서 Peer를 발견하지 못했을 때 지원 네트워크 상태를 어떻게 구분하나요?
- 동일 메뉴를 두 기기에서 수정했을 때 어떤 리비전을 최신으로 보고 충돌을 어떻게 해결하나요?
- 승인·메뉴 ACK를 보낸 Peer가 곧바로 종료됐을 때 완료 기기가 최신 리비전을 어떻게 확인하나요?
- 트리거별 최대 3회이면서 총 30초 이내인 제한과 30초 주기 대조가 깨우기·전면 전환·네트워크 변경 누락을 충분히 복구하나요?
- 14:30 finalization과 이후 제한된 재동기화가 불완전 snapshot·히스토리를 결정적으로 다시 계산하나요?
- 넉넉한 시계 오차 허용값으로도 잘못된 참여 승인·마감·종료 판정을 막을 수 있나요?
- 14일 만료와 삭제 표식 이후 오래된 Peer가 지난 Room을 되살리지 않게 할 수 있나요?
- 자동 신뢰 회사 WiFi의 악의적인 내부 Peer 위험과 위조·복호화 실패를 어떻게 측정하고 수용하나요?
- 예상 Peer 수에서 동기화 지연과 수동 새로고침 완료 시간이 어느 정도인가요?

이 질문은 제품 동작을 다시 후보로 돌리는 것이 아니라, 확정된 안전 계약을 구현할 구체 방식과 지원 범위를 정하기 위한 스파이크입니다.
