# 00. 초기 아이디어

> 상태: 제품 정의 중
>
> 역할: LunchTime이 시작된 문제의식과 현재까지의 논의를 기록하는 출발점
>
> 주의: 이 문서는 확정 요구사항이나 동작 정책이 아니다.

이 문서는 기존 `LunchTime_idea.md`에서 논의한 내용을 제품 정의 문서 체계로 옮기고, 이후 피드백을 함께 반영한 기록입니다.

## 출발점

회사에서 점심을 주문할 때는 보통 Slack에서 사람을 모으고, 가게를 정하고, 각자 메뉴를 적은 뒤 누군가 배달의민족으로 주문합니다.

이 과정에는 대화가 반드시 필요합니다. 메뉴와 옵션은 단순한 O/X 투표만으로 정하기 어렵고, 가게를 바꾸거나 주문 조건을 맞추는 과정에서도 여러 사람이 의견을 주고받아야 합니다.

문제는 채팅 자체가 아니라, 서로 다른 점심 모집과 업무 대화가 한곳에 섞이고 모집 상태가 구조화되어 있지 않다는 점입니다.

- 어떤 모집이 진행 중인지 한눈에 보기 어렵습니다.
- 누가 어느 방에 참여했는지 다시 확인해야 합니다.
- 가게 후보, 배민 링크, 개인 메뉴가 대화 사이에 흩어집니다.
- 주문 마감과 주문 완료 여부가 명확하지 않습니다.
- 여러 점심 모집의 대화가 서로 섞입니다.

LunchTime은 대화를 없애는 대신, 점심방의 상태와 대화를 같은 맥락 안에 묶으려는 아이디어에서 출발합니다.

## 사용 환경

- 전 사원이 MacBook을 사용합니다.
- Windows 대응은 현재 고려하지 않습니다.
- 회사가 배달의민족 주문을 지원하며, 실제 주문은 대부분 배달의민족으로 이루어집니다.
- 과거에 주문했던 가게를 다시 이용하는 경우가 많습니다.
- 제품 운영을 위한 별도 서버 비용과 유지보수 부담은 원하지 않습니다.

## 현재까지 정한 제품 방향

### 점심방

점심 모집 한 건을 하나의 Room으로 관리합니다. Room에는 참여자, 가게 후보, 배민 링크, 각자의 메뉴, 주문 마감, 주문 완료 여부와 관련 대화가 모입니다.

### 구조화된 상태와 채팅의 결합

Room의 핵심 정보는 구조화하되, 그 정보만으로 해결되지 않는 협의를 위해 방별 채팅을 제공합니다.

점심방과 별개로 점심 메뉴와 무관한 잡담을 나눌 수 있는 라운지 공간도 제공합니다. 라운지의 세부 범위와 보존 방식은 추가 논의가 필요합니다.

### 별도 서버 없는 Peer-to-Peer

운영 비용을 만들지 않기 위해 독자적으로 운영하는 중앙 서버를 두지 않는 방향으로 설계합니다.

Peer-to-Peer 환경에서 기기 검색, 상태 동기화, 채팅 이력, 충돌 해결, 방을 만든 사용자가 자리를 비운 경우를 어떻게 처리할지는 기술 검증이 필요합니다.

### 히스토리에서 다시 모집하기

완료된 주문은 히스토리에 남깁니다. 자주 다시 이용하는 가게라면 히스토리에서 한 번의 동작으로 새 모집방을 여는 기능도 유력한 후보로 검토합니다.

새 방에 어떤 정보를 복사할지—가게와 링크만 복사할지, 이전 메뉴와 참여자까지 참고할 수 있게 할지—는 아직 정하지 않았습니다.

### 배달의민족 중심의 흐름

가게 추천에는 배달의민족 공유 링크를 첨부할 수 있어야 합니다.

```text
다운타우너 안국점
https://s.baemin.com/0l000jIOrP32Y
```

배달의민족에서 가게와 메뉴를 확인하는 과정은 휴대폰에서 이루어질 가능성이 높습니다. 따라서 Mac에서 링크를 받은 뒤 휴대폰으로 자연스럽게 넘기는 경험이 필요합니다.

AirDrop을 포함한 macOS 공유 시트, 링크 복사, QR 코드 등을 후보로 검토하되 아직 특정 방식으로 확정하지 않습니다. 배달의민족이 가게·메뉴 조회에 사용할 수 있는 공식 공개 API나 공식 연동 방식을 제공하는지도 별도로 확인해야 합니다.

## 예상 사용자 흐름

현재까지의 아이디어를 연결하면 다음과 같습니다.

1. 새 점심방을 만듭니다.
2. 동료가 방을 발견하고 참여합니다.
3. 참여자들이 방별 채팅으로 의견을 나눕니다.
4. 가게 후보와 배민 링크를 올리고, 필요할 때만 투표합니다.
5. 가게가 정해지면 각자 메뉴와 옵션을 직접 작성합니다.
6. 설정한 주문 마감 시각이 되면 신규 참여가 닫힙니다.
7. 참여자 중 한 명이 배달의민족에서 실제 주문을 진행합니다.
8. 주문이 끝나면 참여자 누구나 방을 주문 완료로 변경할 수 있습니다.
9. 완료된 방은 히스토리로 이동합니다.

주문 완료가 이 흐름의 종료 상태입니다. 별도의 `출발` 상태는 두지 않습니다.

히스토리의 가게로 새 방을 다시 여는 기능이 채택되면 1번의 또 다른 진입점이 됩니다.

개인 메뉴는 우선 자유롭게 작성하는 형태를 생각하고 있습니다.

```text
다운타우너 버거 세트
(D) 오리지널 프라이즈 +2000
(D) 코카콜라 제로 255ml
(D) 싱글
```

## 이전 초안에서 논의한 상세 범위

다음 내용은 기존 아이디어 문서에서 구체적으로 논의했던 범위입니다. 현재 제품 방향을 이해하기 위한 기록이며, 세부 요구사항은 아직 PRD로 확정하지 않았습니다.

### 방 생성과 참여

- 새 방을 만들 때 메뉴를 필수로 입력합니다.
- 가게, 주문 마감 시각, 메모는 선택 정보로 논의했습니다.
- 같은 네트워크의 사용자에게 새 모집 알림을 보냅니다.
- 사용자는 버튼 하나로 참여하고 참여자 목록을 실시간으로 확인합니다.
- 여러 점심방이 동시에 존재해도 각 Room을 분리해 보여줍니다.

### 가게 결정과 개인 메뉴

- 참여자는 가게 후보와 배민 링크를 방에 올릴 수 있습니다.
- 의견이 갈릴 때만 선택적으로 투표를 사용합니다.
- 가게가 정해지면 각 참여자가 자신의 메뉴와 옵션을 자유 형식으로 작성합니다.
- 주문 마감 시각이 되면 신규 참여를 자동으로 닫습니다.

### 완료와 히스토리

- 실제 주문을 누가 맡았는지와 무관하게 참여자 누구나 주문 완료로 변경할 수 있습니다.
- 완료 처리 후 참여자에게 알리고 방을 히스토리로 이동합니다.
- 주문 완료가 종료 상태이며 별도의 출발 상태는 두지 않습니다.

### 상태에 대한 초기 구상

초기 문서에서는 아래 상태를 구상했습니다.

| 상태 | 의미 |
|------|------|
| 모집 중 | 참여자를 모으는 중 |
| 진행 중 | 참여와 가게 논의가 진행되는 중 |
| 투표 중 | 가게 또는 메뉴 투표 중 |
| 메뉴 등록 중 | 참여자들이 각자 주문할 메뉴와 옵션을 작성하는 중 |
| 주문 마감 | 설정한 시각이 되어 신규 참여가 닫힌 상태 |
| 주문 완료 | 주문이 끝나 히스토리로 이동한 종료 상태 |

채팅이 추가된 현재 방향에서는 이 상태 구성이 충분한지 다시 검토해야 합니다.

### 인터페이스에 대한 초기 구상

- 모든 사용자가 빠르게 접근할 수 있는 macOS MenuBar App
- 터미널 사용자를 위한 Interactive CLI
- 두 인터페이스가 하나의 Core와 상태를 공유하는 구조

MenuBar App은 현재도 유력한 방향입니다. CLI의 필요성과 우선순위는 추가 검증이 필요합니다.

### 기술 구조에 대한 초기 구상

- SwiftUI, MenuBarExtra, AppKit 기반 UI
- 공통 로직과 상태를 담당하는 `LunchCore`
- Bonjour, Multipeer Connectivity, Network.framework 기반 네트워킹 후보
- 로컬 히스토리와 상태 저장을 위한 SwiftData 후보

이는 초기 기술 가설이며, 제품 요구사항과 Peer-to-Peer 기술 검증을 거쳐 별도의 기술 문서에서 결정해야 합니다.

## 현재 가설

다음 항목은 유력한 아이디어이지만 아직 검증하거나 확정하지 않았습니다.

- 메뉴바 앱이 가장 빠른 진입점일 것입니다.
- 터미널을 자주 쓰는 사용자를 위한 Interactive CLI도 유용할 것입니다.
- 수기로 작성하는 개인 메뉴만으로 초기 사용에 충분할 것입니다.
- macOS 공유 시트나 AirDrop이 배민 링크를 휴대폰으로 넘기는 데 적합할 것입니다.
- Bonjour, Multipeer Connectivity, Network.framework 같은 기술로 회사 환경의 Peer-to-Peer 동기화를 구현할 수 있을 것입니다.
- 로컬 저장소만으로 방, 채팅, 히스토리를 사용자가 기대하는 수준까지 유지할 수 있을 것입니다.

## 열린 질문

### 제품 경험

- 방별 채팅과 라운지 채팅은 각각 얼마나 오래 보관해야 할까요?
- 알림이 과도해지지 않도록 방과 라운지의 알림을 어떻게 구분해야 할까요?
- 히스토리에서 새 방을 열 때 어떤 정보를 복사해야 할까요?
- 여러 점심방에 동시에 참여할 수 있을까요?
- 개인 메뉴의 수정은 언제까지 허용해야 할까요?

### Peer-to-Peer

- 회사 네트워크에서 기기 검색과 직접 연결이 안정적으로 가능할까요?
- 방을 만든 사용자가 앱을 종료하거나 네트워크를 벗어나도 방을 유지할 수 있을까요?
- 늦게 참여한 사용자가 이전 채팅과 현재 상태를 어떻게 전달받을까요?
- 서로 다른 기기에서 동시에 상태를 바꾸면 무엇을 기준으로 병합해야 할까요?
- 사용자 식별, 접근 제어, 전송 암호화는 어느 수준까지 필요할까요?
- 잠자기, 네트워크 전환, 오프라인 상태를 어떻게 처리해야 할까요?

### 배달의민족 연동

- 가게와 메뉴 정보를 조회할 수 있는 공식 공개 API 또는 공식 파트너 연동이 있을까요?
- 배민 공유 링크에서 공식적으로 사용할 수 있는 정보의 범위는 어디까지일까요?
- Mac에서 휴대폰으로 넘길 때 공유 시트, AirDrop, QR 코드 중 어떤 경험이 가장 빠를까요?
- 배민 앱으로 이동한 뒤 LunchTime으로 돌아오는 과정에서 사용자가 직접 입력해야 하는 정보는 무엇일까요?

## 현재 비목표

- Windows 클라이언트
- 별도 백엔드 서버 운영
- LunchTime 안에서 배달 주문이나 결제를 직접 처리하는 기능
- 투표만으로 점심 메뉴를 결정하는 도구
- 배달의민족의 비공개 API나 스크래핑에 의존하는 구현
- 주문 완료 이후의 별도 `출발` 상태

## 다음에 정의할 것

1. 회사 네트워크에서 Peer-to-Peer 연결이 가능한지 작은 기술 검증을 진행합니다.
2. 배달의민족의 공식 연동 수단과 모바일 전달 방식을 조사합니다.
3. 점심방 상태와 방별 채팅·라운지 채팅의 경계를 정합니다.
4. 히스토리에 남길 정보와 `다시 모집하기` 동작을 구체화합니다.
5. 논의를 거쳐 확정된 내용만 PRD와 Policies 문서로 옮깁니다.
