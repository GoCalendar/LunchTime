# 00. 초기 아이디어

> 상태: 초기 아이디어와 이후 결정의 변화 이력
>
> 역할: LunchTime이 시작된 문제의식과 현재까지의 논의를 기록하는 출발점
>
> 주의: 이 문서는 확정 요구사항이나 동작 정책이 아니다. 현재 정본은 [LunchTime MVP PRD](../prd/01_lunchtime_mvp.md)와 Policies다.

이 문서는 기존 `LunchTime_idea.md`에서 논의한 내용을 제품 정의 문서 체계로 옮기고, 이후 피드백과 2026-07-24 회의에서 무엇이 달라졌는지 함께 보존한 기록입니다.

## 읽는 방법

- **현재 방향**은 최근 회의에서 채택한 제품 모델입니다.
- **대체된 초기 구상**은 아이디어가 발전한 맥락을 남기기 위한 기록이며 구현 기준이 아닙니다.
- 기술 선택이 남은 내용은 **기술 스파이크**로 표시합니다.

## 출발점

회사에서 점심을 주문할 때는 보통 Slack에서 사람을 모으고, 가게를 정하고, 각자 메뉴를 적은 뒤 누군가 배달의민족으로 주문합니다.

이 과정에는 대화가 반드시 필요합니다. 메뉴와 옵션은 단순한 O/X 투표만으로 정하기 어렵고, 가게를 바꾸거나 주문 조건을 맞추는 과정에서도 여러 사람이 의견을 주고받아야 합니다.

문제는 채팅 자체가 아니라, 서로 다른 점심 모집과 업무 대화가 한곳에 섞이고 모집 상태가 구조화되어 있지 않다는 점입니다.

- 어떤 모집이 진행 중인지 한눈에 보기 어렵습니다.
- 누가 어느 방에 참여했는지 다시 확인해야 합니다.
- 가게 후보, 배민 링크, 개인 메뉴가 대화 사이에 흩어집니다.
- 주문 마감과 주문 완료 여부가 명확하지 않습니다.
- 여러 점심 모집의 대화가 서로 섞입니다.

LunchTime은 대화를 없애는 대신, 점심방의 상태와 대화를 같은 맥락 안에 묶으려는 아이디어에서 출발합니다.

## 사용 환경

- 전 사원이 MacBook을 사용합니다.
- Windows 대응은 현재 고려하지 않습니다.
- 회사가 배달의민족 주문을 지원하며, 실제 주문은 대부분 배달의민족으로 이루어집니다.
- 과거에 주문했던 가게를 다시 이용하는 경우가 많습니다.
- 제품 운영을 위한 별도 서버 비용과 유지보수 부담은 원하지 않습니다.

## 현재까지 정한 제품 방향

### 점심방

점심 모집 한 건을 하나의 Room으로 관리합니다. **한 Room은 한 가게**를 나타내며, 사용자는 후보에 투표하는 대신 그 가게 주문에 참여합니다. 참여자가 한 명뿐이어도 유효한 Room입니다. 한 사용자는 운영일에 하나의 Room에만 참여하며, 주문 상태가 `진행 중`일 때 기존 참여를 철회해야 다른 Room으로 이동하거나 새 Room을 만들 수 있습니다. 여러 Room 참여 충돌은 사용자가 하나를 고를 때까지 양쪽 주문 완료를 막고, 선택하지 않은 Room의 참여·활성 메뉴를 철회합니다.

Room에는 참여자, 배민 링크, `메뉴명 + 반복 옵션` 구조의 개인 메뉴, 주문 마감, 주문 상태와 관련 대화가 모입니다.

주문 진행 중·기존 마감 전에는 현재 참여 중인 생성자만 마감 시각을 바꿀 수 있습니다. 주문 완료 상태에서는 잠기고 한 번 도달한 마감을 다시 열 수 없습니다. 마지막 참여자의 철회는 Room을 취소합니다. 현재 참여 중인 생성자는 진행 중 Room을 전체 취소할 수 있고, 주문 완료 상태라면 먼저 완료를 되돌립니다. 생성자가 철회하면 관리자 권한을 잃고 가게명·입력 링크·마감 메타데이터가 잠깁니다. 취소 Room은 14:30까지 열람 전용이고 최근 상세와 히스토리에서 제외합니다.

### 구조화된 상태와 채팅의 결합

Room의 핵심 정보는 구조화하되, 그 정보만으로 해결되지 않는 협의를 위해 방별 채팅을 제공합니다.

점심방과 별개로 점심 메뉴와 무관한 잡담을 나누는 `라운지`를 MVP에 포함합니다. 라운지는 별도 앱 창에서 11:00~14:30에 사용하며, MenuBar 바로가기와 읽지 않음 표시는 후속 범위입니다.

### 별도 서버 없는 Peer-to-Peer

운영 비용을 만들지 않기 위해 LunchTime이 독자적으로 운영하는 중앙 서버를 두지 않습니다. 현재 사내 두 Wi-Fi가 연결되는 내부망에서 각 Peer가 변경 이벤트를 로컬에 보유하고 서로 차이를 교환하는 방향입니다.

Mac이 잠자거나 꺼진 동안에는 통신할 수 없으므로, 깨우기·앱 전면 전환·네트워크 변경·의미 있는 새 Peer 연결과 수동 요청마다 최대 3회이면서 총 30초 이내인 동기화 세션을 시작하고 둘 중 먼저 도달하면 종료합니다. 11:00~14:30에 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 Peer가 있을 때만 30초 간격으로 대조합니다. 대상이 없거나 14:30이면 멈추고, 제한을 소진한 실패 Peer는 정기 타이머만으로 재시도하지 않습니다.

참여·메뉴·주문 데이터가 조용히 누락되는 일을 최우선으로 방지합니다. 참여는 마감 전에 승인 ACK를 받아야 하고, 주문 완료 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다. 과거·오프라인 Peer의 미응답만으로 완료를 막지 않으며, 로컬 단독 예외는 생성자가 처음부터 만들고 한 번도 공유하지 않은 Room에만 적용합니다.

동기화 상태는 `이 기기에 저장됨/복제됨/동기화됨/확인 필요`이고 `동기화 중`은 별도 활동입니다. Peer 발견, 복제 자료구조, 충돌 병합, ACK, 회사 WiFi 자동 신뢰 경계, 암호화 세션과 시계 오차는 기술 스파이크가 필요합니다.

지원 회사 WiFi에서 발견한 Peer는 자동 신뢰합니다. 인식된 Peer는 활성 Room의 메뉴·채팅을 읽고 참여자만 쓸 수 있으며, 악의적인 내부 Peer가 평문을 읽을 수 있는 위험은 수용합니다. Connection / Settings에는 닉네임, 온라인 여부, 마지막 확인 시각과 동기화 상태만 표시하고 중복 닉네임에는 짧은 기기 ID를 붙입니다.

### 히스토리에서 다시 모집하기

히스토리의 목적은 자주 다시 이용하는 가게로 새 Room을 빠르게 만드는 것입니다. 가게명, 입력 링크, 최근 주문 완료 시각과 로컬 이용 횟수만 각 기기에 암호화해 장기 보존합니다.

`다시 모집`은 가게명과 링크가 채워진 생성 화면을 열고 사용자가 주문 마감 시각을 확인하게 하며 즉시 Room을 만들지 않습니다. 같은 가게는 `trim(storeName) + raw inputLink`가 완전히 같을 때만 합칩니다. 입력 링크는 공백 제거·URL 정규화·redirect 추적을 하지 않습니다. 이전 참여자, 개인 메뉴와 채팅은 복사하지 않습니다.

### 배달의민족 중심의 흐름

가게 추천에는 배달의민족 공유 링크를 첨부할 수 있어야 합니다.

```text
다운타우너 안국점
https://s.baemin.com/0l000jIOrP32Y
```

배달의민족에서 가게와 메뉴를 확인하는 과정은 휴대폰에서 이루어집니다. MVP에서는 사용자가 입력한 링크 문자열을 검증·정규화·경고 없이 그대로 Mac 화면의 **QR 코드로 표시**하고 휴대폰으로 스캔합니다.

AirDrop과 macOS 공유 시트는 MVP에서 제외합니다. LunchTime은 실제 메뉴 탐색, 주문과 결제를 배민 앱에 남기며 비공개 API나 스크래핑에 의존하지 않습니다.

### 11:00~14:30 일일 라이프사이클

- 11:00부터 당일 Room과 라운지의 쓰기 기능을 사용합니다.
- Room별 주문 마감 시각에는 ACK 없는 참여 요청이 실패하고 신규 참여가 닫힙니다. 마감 전에 승인된 참여는 늦게 전달돼도 유효합니다.
- 주문 완료는 14:30 전까지 주문 취소 상황에 맞춰 되돌릴 수 있는 상태입니다.
- 14:30에 모든 Room과 라운지의 쓰기를 닫고 MenuBar에서 당일 Room을 제거한 뒤 최대 120초 finalization을 수행합니다.
- 잠든 Mac은 wake 시 즉시 종료를 적용하고 그때부터 최대 120초 finalization을 수행합니다.
- 11:00 전사 점심 선택 알림은 후속 범위입니다.

### 데이터 보존과 보호

- Room·참여·메뉴·상태와 동기화 이벤트는 최대 14일 관리합니다.
- Room과 라운지 채팅은 고유 메시지 ID로 중복 제거하고 best-effort 순서로 교환하며, 프로세스 메모리에만 두고 앱 종료 시 해당 기기에서 삭제합니다.
- 같은 날 재실행하면 살아 있는 Peer의 당일 채팅을 받을 수 있지만 완전 복구나 전역 순서를 보장하지 않습니다.
- 최소 가게 히스토리는 14일 운영 데이터와 분리해 로컬에 장기 보존합니다.
- Peer 간 응용 데이터와 디스크에 남는 운영 이벤트·히스토리는 암호화합니다.
- 삭제된 데이터가 일부 Peer에 고아 데이터로 남을 수 있으며 모든 복제본의 동시·영구 삭제는 보장하지 않습니다.
- 종료 Room 상세는 모든 자동 신뢰 회사 WiFi Peer가 최대 14일 읽을 수 있습니다. 불완전 종료는 성공 히스토리를 만들지 않지만 이후 제한된 재동기화로 다시 계산할 수 있습니다.
- Peer 시계 차이가 허용 범위를 넘으면 참여 승인, 마감 변경과 주문 상태 변경처럼 시간에 의존하는 사용자 쓰기를 차단합니다. 14:30 쓰기 종료는 안전하게 적용하되 종료 스냅샷의 유효성·완전성 판정은 `확인 필요`로 둡니다.

## 예상 사용자 흐름

최근 회의에서 채택한 흐름은 다음과 같습니다.

1. 11:00 이후 MenuBar에서 오늘의 Room과 상태를 봅니다.
2. 가게명, 배민 링크와 주문 마감 시각으로 한 가게의 Room을 만듭니다.
3. 동료가 그 가게 주문에 참여합니다. 한 명만 참여해도 진행할 수 있습니다.
4. Mac의 QR을 휴대폰으로 스캔하고 외부 배민 앱·페이지에서 메뉴 확인을 시도합니다. LunchTime은 QR 생성 이후 동작을 보장하지 않습니다.
5. 각자 메뉴명과 반복 가능한 옵션을 고정 폼에 입력합니다.
6. 참여자들이 방별 채팅으로 필요한 협의를 이어갑니다.
7. 설정한 주문 마감 시각이 되면 신규 참여만 닫힙니다.
8. 주문 전 점검에서 모든 현재 참여자의 메뉴와 Peer ACK 상태를 확인합니다.
9. 참여자 중 한 명이 배민에서 실제 주문하고, 누구나 주문 완료로 표시합니다.
10. 주문이 취소되면 14:30 전에 완료 표시를 되돌릴 수 있습니다.
11. 14:30에 Room과 라운지의 쓰기를 닫아 현재 메모리를 열람 전용으로 전환하고, MenuBar에서 당일 Room을 제거한 뒤 최대 120초 finalization을 수행합니다.
12. 최소 가게 히스토리는 다음 재모집을 위해 로컬에 남습니다.

주문 완료를 종료로 보던 초기 아이디어는 대체됐습니다. `14시 30분 사용자 종료`가 최종 쓰기 상태이며 별도의 `출발` 상태는 두지 않습니다.

개인 메뉴 예시는 같지만 입력은 자유 형식 블록이 아니라 구조화된 필드입니다.

```text
메뉴명: 다운타우너 버거 세트
옵션:
- 오리지널 프라이즈 / +2,000원
- 코카콜라 제로 255ml
- 싱글
```

## 대체된 초기 구상

다음 내용은 아이디어가 발전한 맥락을 보존하기 위한 기록입니다. **현재 구현 기준이 아니며**, 최신 동작은 위 `현재까지 정한 제품 방향`과 PRD를 따릅니다.

### 방 생성과 참여

- 초기에는 새 방의 `메뉴`와 선택적인 가게를 별개로 생각했습니다.
- 이후 한 Room이 한 가게를 나타내고 가게명·배민 링크·주문 마감으로 생성하는 모델로 대체했습니다.
- 새 모집 알림은 아이디어에 있었지만 11:00 전체 알림과 정교한 알림은 MVP 이후로 미뤘습니다.
- 버튼 하나로 참여하고 Room을 서로 분리해 보여주는 방향은 유지했습니다.

### 가게 결정과 개인 메뉴

- 초기에는 참여자가 가게 후보를 올리고 필요할 때 선택적으로 투표하는 모델을 검토했습니다.
- 회의 후 후보·투표를 제거하고, 이미 정한 한 가게의 Room에 사용자가 참여하는 모델로 대체했습니다.
- 자유 형식 메뉴 블록은 메뉴명과 반복 옵션 필드로 대체했습니다.
- 주문 마감 시각에 신규 참여만 닫는 방향은 유지했습니다.

### 완료와 히스토리

- 참여자 누구나 주문 완료로 바꾸는 방향은 유지했습니다.
- `주문 완료가 즉시 종료 상태`라는 초기 구상은 폐기했습니다. 주문 완료는 14:30 전까지 되돌릴 수 있습니다.
- 14:30 사용자 종료 뒤 최대 120초 finalization을 수행하며 별도의 출발 상태는 두지 않습니다.
- 완료된 Room 전체를 장기 히스토리로 남기는 대신 최소 가게 정보만 재사용 히스토리로 남깁니다.

### 상태에 대한 초기 구상

초기 문서에서는 아래 상태를 구상했습니다. 이 표는 **현재 대체된 모델**입니다.

| 상태 | 의미 |
|------|------|
| 모집 중 | 참여자를 모으는 중 |
| 진행 중 | 참여와 가게 논의가 진행되는 중 |
| 투표 중 | 가게 또는 메뉴 투표 중 |
| 메뉴 등록 중 | 참여자들이 각자 주문할 메뉴와 옵션을 작성하는 중 |
| 주문 마감 | 설정한 시각이 되어 신규 참여가 닫힌 상태 |
| 주문 완료 | 주문이 끝나 히스토리로 이동한 종료 상태 |

현재는 `일일 상태`, `모집 상태`, `주문 상태`, `동기화 상태`를 서로 독립적인 네 축으로 봅니다. 메뉴의 제출 여부와 동기화 표시는 참여자별 파생 정보이며, 투표 상태는 존재하지 않습니다.

### 인터페이스에 대한 초기 구상

- macOS MenuBar App과 Interactive CLI를 함께 제공하는 아이디어가 있었습니다.
- MenuBar는 기본 진입점으로 확정했고 상세 기능은 별도 창에서 진행합니다.
- CLI는 MVP에서 제외했습니다.

### 기술 구조에 대한 초기 구상

- SwiftUI, MenuBarExtra, AppKit 기반 UI
- 공통 로직과 상태를 담당하는 `LunchCore`
- Bonjour, Multipeer Connectivity, Network.framework 기반 네트워킹 후보
- 로컬 히스토리와 상태 저장을 위한 SwiftData 후보

이는 초기 기술 후보입니다. 현재는 분산 로컬 이벤트, Peer ACK, 복귀 catch-up과 암호화라는 제품 계약만 정했으며, 프레임워크와 복제 알고리즘은 기술 스파이크에서 결정합니다.

## 현재 기술 가설

제품 방향은 정했지만 다음 구현 전제는 실제 환경에서 아직 검증하지 않았습니다.

- 사내 두 Wi-Fi 사이에서 Bonjour/mDNS 또는 다른 방식으로 Peer를 안정적으로 발견하고 회사 WiFi 경계 안에서 자동 신뢰할 수 있습니다.
- 변경 로그, CRDT 등 적합한 자료구조로 중앙 리더 없이 중복 제거와 충돌 병합을 수행할 수 있습니다.
- 마감 전 참여 승인 ACK와 현재 참여자의 최신 메뉴 ACK만으로 fail-closed 주문 완료 계약을 구현할 수 있습니다.
- 트리거별 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 끝나는 세션과 활성 Room·라운지 또는 미해결 데이터·건강한 Peer 대상 30초 anti-entropy로 단절 중 누락분을 복구할 수 있습니다.
- 회사 WiFi 자동 신뢰 경계와 암호화 세션을 적용하고 악의적인 내부 Peer 위험을 수용할 수 있습니다.
- 운영 이벤트 14일, 메모리 전용 채팅과 장기 최소 히스토리의 서로 다른 보존 규칙을 안전하게 구현할 수 있습니다.
- 14:30 종료·최대 120초 finalization과 이후 제한된 재동기화가 final snapshot과 성공 히스토리를 다시 계산할 수 있습니다.
- 넉넉한 시계 오차 허용값으로 정상 기기는 허용하면서 잘못된 시간 의존 동작을 차단할 수 있습니다.
