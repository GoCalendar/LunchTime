# 01. 제품 방향

> 상태: 2026-07-24 회의 결정을 반영한 제품 방향
>
> 입력: [00. 초기 아이디어](./00_initial_idea.md)
>
> 역할: 이후 제품 정의가 벗어나지 않아야 할 목적, 대상, 가치와 제약을 정리한다.

현재 구현 요구사항의 정본은 [LunchTime MVP PRD](../prd/01_lunchtime_mvp.md)이며, 이 문서는 그 결정에 이른 방향과 아직 기술 검증이 필요한 전제를 설명합니다.

## 표현 범례

- **방향 확정**: 제품 책임자가 명시적으로 선택한 방향입니다. 구현 가능성 검증이 끝났다는 뜻은 아닙니다.
- **기술 스파이크**: 원하는 제품 동작은 정했지만 실제 사내 환경에서 구현 방식과 보장 범위를 검증해야 합니다.
- **미확정**: 현재 정보만으로 답할 수 없어 후속 결정이 필요합니다.

## 제품 목적

LunchTime은 반복되는 점심 모집과 배달 주문 협의를 하나의 맥락으로 묶어, 대화의 유연성은 유지하면서도 현재 상태와 주문에 포함할 메뉴를 누구나 정확하게 파악할 수 있게 하는 macOS 앱입니다.

- **방향 확정**: Room 하나는 가게 하나에 대한 주문 모집입니다.
- **방향 확정**: 사용자는 후보 가게에 투표하는 대신 주문하려는 가게의 Room에 참여합니다. 참여자가 한 명뿐이어도 유효합니다.
- **방향 확정**: Room별로 참여자, 배민 링크, 구조화된 개인 메뉴, 주문 마감, 주문 상태와 대화를 함께 관리합니다.
- **방향 확정**: 점심방과 별개로 점심과 무관한 잡담을 나누는 `라운지`를 MVP에 포함합니다.
- **방향 확정**: 자주 이용하는 가게의 최소 히스토리를 재사용해 새 모집을 빠르게 시작합니다.
- **방향 확정**: 데이터가 조용히 누락되어 참여자의 주문이 빠지는 일을 제품의 최우선 실패로 봅니다.

## 주요 대상

- 회사에서 배달의민족으로 점심을 함께 주문하는 구성원
- 사내 두 Wi-Fi가 연결되는 내부망에서 업무용 MacBook을 사용하는 구성원
- Room을 만든 사람, 중간에 참여한 사람과 실제 주문을 수행한 사람
- **방향 확정**: 주문 완료 표시와 되돌리기는 Room 생성자 전용 권한이 아닙니다.
- **방향 확정**: 한 사용자는 같은 운영일에 하나의 Room에만 참여하며, 주문 상태가 `진행 중`일 때 현재 참여를 철회해야 다른 Room으로 이동할 수 있습니다.
- **방향 확정**: 참여 철회는 자신의 활성 메뉴를 함께 무효화합니다.
- **방향 확정**: 마지막 참여자 철회는 Room을 취소합니다. 생성자가 철회하면 관리자 권한을 잃고 가게명·입력 링크·마감 메타데이터가 잠깁니다.
- **방향 확정**: 같은 사용자의 여러 Room 참여 충돌은 자동 선택하지 않고 사용자의 수동 선택 전까지 양쪽 주문 완료를 차단하며, 선택하지 않은 Room의 참여·활성 메뉴를 철회합니다.
- **방향 확정**: 현재 참여 중인 생성자는 진행 중 Room을 전체 취소할 수 있고, 주문 완료 상태라면 먼저 완료를 되돌립니다.
- **미확정**: 초기 파일럿 사용자 수와 동시에 연결될 최대 Peer 규모는 기술 스파이크에서 정합니다.

## 핵심 가치

### 1. 메뉴 누락 방지

로컬 저장 성공과 다른 Peer 전달 성공을 구분합니다. 참여자 메뉴가 없거나 동기화 상태가 `이 기기에 저장됨`·`확인 필요`이거나 충돌이 있으면 주문 완료를 막아, 실제 주문자가 불완전한 목록을 정상 상태로 오인하지 않게 합니다.

### 2. 상태와 대화의 결합

구조화된 주문 정보와 자유로운 대화를 같은 Room 안에 두되, 채팅이 참여·메뉴·주문 상태의 정본을 대신하지 않게 합니다.

### 3. 낮은 참여 비용

MenuBar에서 오늘의 Room, 참여자와 상태를 확인하고 별도 창에서 참여, QR 확인, 메뉴 등록과 채팅을 이어갑니다.

### 4. 반복 가게의 재사용

가게명, 입력 링크, 최근 주문 완료 시각과 로컬 이용 횟수로 최소 히스토리를 남겨, 가게명과 링크가 채워진 생성 화면에서 마감 시각을 확인한 뒤 새 Room을 엽니다. 같은 가게는 `trim(storeName) + raw inputLink` 완전 일치로만 합치며 입력 링크는 trim·URL 정규화·redirect 추적을 하지 않습니다.

### 5. 회사 환경에 맞춘 집중

전 사원이 MacBook과 배달의민족을 사용하는 실제 환경, 그리고 11:00~14:30 점심 라이프사이클에 맞춥니다.

### 6. 운영 부담 최소화

제품을 유지하기 위한 독자적인 중앙 서버와 상시 운영 비용을 만들지 않습니다.

## 제품 제약

| 항목 | 현재 판단 |
|------|-----------|
| 지원 플랫폼 | **방향 확정**: macOS만 지원하며 Windows는 비목표 |
| 기본 Surface | **방향 확정**: MenuBar는 요약·진입점, 주요 기능은 별도 앱 창 |
| 점심 시간 | **방향 확정**: 매일 11:00~14:30에 쓰기 가능, 14:30부터 당일 데이터 열람만 가능 |
| Room 모델 | **방향 확정**: 한 Room은 한 가게, 투표 없이 참여, 한 명만 참여해도 유효, 한 사용자는 운영일당 한 Room에만 참여 |
| 주문 수단 | **방향 확정**: 메뉴 탐색, 실제 주문과 결제는 배민 앱에서 수행 |
| 모바일 전달 | **방향 확정**: MVP는 입력 링크 문자열을 검증·정규화·경고 없이 그대로 QR로 표시 |
| 개인 메뉴 | **방향 확정**: 메뉴명과 반복 가능한 옵션 행으로 고정된 입력 구조 |
| 주문 상태 | **방향 확정**: 주문 완료는 14:30 전까지 되돌릴 수 있고, 14:30 사용자 종료 뒤 최대 120초 finalization 수행 |
| 백엔드 운영 | **방향 확정**: LunchTime 전용 중앙 서버를 운영하지 않음 |
| 통신 구조 | **방향 확정**: 각 Peer가 로컬 변경 이벤트를 보유하고 직접 교환하는 분산 동기화 |
| 복귀 동기화 | **방향 확정**: 의미 있는 트리거와 수동 요청마다 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 끝나는 세션, 활성 Room·라운지 또는 미해결 데이터와 건강한 Peer가 있을 때만 30초 주기 대조, 실패 Peer는 타이머만으로 재시도하지 않음 |
| 데이터 보존 | **방향 확정**: 운영 이벤트 최대 14일, 채팅은 앱 메모리에서만 관리하고 종료 시 삭제, 최소 가게 히스토리는 로컬에 장기 보존 |
| 데이터 보호 | **방향 확정**: 전송 데이터와 디스크에 남는 데이터를 암호화하되 회사 WiFi 발견 Peer는 자동 신뢰 |
| Peer 가시성 | **방향 확정**: 인식된 Peer는 활성 Room 메뉴·채팅과 최대 14일 종료 상세를 읽고, 참여자만 활성 Room에 씀 |
| 기술 보장 | **기술 스파이크**: Peer 발견, 충돌 병합, ACK, 자동 신뢰 네트워크 경계, 암호화 세션과 시계 오차의 구체 방식 |
| 배민 연동 | **방향 확정**: 비공개 API나 스크래핑에 의존하지 않음 |
| 결제 | **방향 확정**: LunchTime 안에서 주문이나 결제를 직접 처리하지 않음 |

잠자기 또는 전원이 꺼진 Mac은 실시간 통신할 수 없습니다. 복귀 후 연결 가능한 Peer가 보유한 변경을 제한된 세션으로 가져오되, 모든 Peer가 이미 데이터를 삭제했다면 복구를 보장하지 않습니다. 14:30에 잠든 Mac은 wake 시 즉시 종료를 적용하고 그때부터 최대 120초 finalization을 수행합니다.

## 핵심 데이터 계약

- 참여·메뉴·주문 상태는 고유 변경 ID를 가진 이벤트로 로컬에 기록하고 Peer에 중복 없이 전파합니다.
- 참여 요청은 마감 전에 승인 ACK를 받아야 유효하고, 마감 전 승인 이벤트는 늦게 전달돼도 유효합니다. ACK가 없으면 마감 시 실패합니다.
- 주문 완료 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다. 과거·오프라인 Peer의 미응답만으로 차단하지 않습니다.
- 로컬 단독 예외는 생성자가 처음부터 만들고 한 번도 다른 Peer에 공유하지 않은 Room에만 적용합니다.
- 현재 참여자의 메뉴가 없거나 최신 리비전이 `이 기기에 저장됨`, `확인 필요`이면 주문 완료를 허용하지 않습니다.
- 동기화 상태는 `이 기기에 저장됨/복제됨/동기화됨/확인 필요`이며 `동기화 중`은 별도 활동 표시입니다.
- 14:30에 MenuBar에서 당일 Room을 제거하고, 불완전 finalization은 성공 히스토리를 만들지 않습니다. 이후 제한된 재동기화가 final snapshot과 히스토리를 다시 계산할 수 있습니다.
- 시계 차이가 허용 범위를 넘으면 참여 승인·마감 변경·주문 상태처럼 시간에 의존하는 사용자 쓰기를 차단합니다. 14:30에는 쓰기를 안전하게 닫고 종료 스냅샷의 유효성·완전성은 `확인 필요`로 표시합니다.
- 중앙 정본 없이 완전한 전역 합의를 약속하지 않으며, 지원할 수 없는 상태를 성공처럼 표시하지 않습니다.

## 제품 가치 판단 후보

현재 방식의 기준선을 관찰한 뒤 다음 신호에 목표를 설정합니다.

| 신호 | 확인하려는 가치 |
|------|----------------|
| 활성 Room과 현재 상태를 파악하는 데 걸리는 시간 | Slack 탐색보다 상태 가시성이 좋아졌는가 |
| 실제 주문에서 누락되거나 다시 확인한 개인 메뉴 수 | 메뉴 취합 오류가 줄었는가 |
| 메뉴 변경의 Peer ACK 성공률과 복구 시간 | 분산 구조가 핵심 주문 데이터를 충분히 전달하는가 |
| 생성→참여→메뉴→주문 표시→14시 30분 종료·finalization 흐름 완주율 | 핵심 여정을 막힘없이 끝낼 수 있는가 |
| 일정 기간 내 반복 사용과 히스토리 재사용률 | 일회성 데모가 아니라 매일 쓸 가치가 있는가 |
| P2P 연결·동기화 실패와 복구 성공률 | 서버 없는 구조가 허용 가능한 신뢰성을 제공하는가 |

## 현재 비목표

- Windows 클라이언트
- 독자적인 중앙 백엔드 서버 운영
- CLI 인터페이스
- 가게 후보 목록과 투표
- QR 이외의 AirDrop·공유 시트 기반 배민 전달
- MenuBar의 라운지 바로가기와 읽지 않음 표시
- 11:00 전사 점심 선택 알림
- LunchTime 안에서 배달 주문이나 결제를 직접 처리하는 기능
- 배달의민족의 비공개 API나 스크래핑에 의존하는 구현
- `출발` 상태
- Mac이 잠자기·전원 꺼짐인 동안의 실시간 통신 보장
