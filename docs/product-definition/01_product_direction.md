# 01. 제품 방향

> 상태: 검토가 필요한 초안
>
> 입력: [00. 초기 아이디어](./00_initial_idea.md)
>
> 역할: 이후 제품 정의가 벗어나지 않아야 할 목적, 대상, 가치와 제약을 정리한다.

## 표현 범례

- **방향 확정**: 제품 책임자가 명시적으로 선택한 방향입니다. 사용자 검증이 끝났다는 뜻은 아닙니다.
- **가설**: 유력하지만 사용자 또는 기술 검증과 세부 결정이 필요합니다.
- **미확정**: 현재 정보만으로 답할 수 없어 후속 논의가 필요합니다.

## 제품 목적

LunchTime은 반복되는 점심 모집과 배달 주문 협의를 하나의 맥락으로 묶어, 대화의 유연성은 유지하면서도 현재 상태를 누구나 빠르게 파악할 수 있게 하는 macOS 앱입니다.

- **방향 확정**: 대화를 투표나 상태 버튼으로 대체하지 않습니다.
- **방향 확정**: 점심방별로 참여자, 가게, 메뉴, 마감, 주문 완료 상태와 대화를 함께 관리합니다.
- **방향 확정**: 서로 다른 점심 모집과 업무 대화가 뒤섞이는 문제를 줄입니다.
- **방향 확정**: 점심방과 별개로 잡담을 나눌 수 있는 라운지를 제공합니다.
- **가설**: 과거 주문 맥락을 재사용하면 반복 모집에 드는 수고를 줄일 수 있습니다.
- **미확정**: 제품 도입 후 어떤 지표로 가치가 생겼다고 판단할지 정하지 않았습니다.

## 주요 대상

- 회사에서 배달의민족으로 점심을 함께 주문하는 구성원
- 업무용 MacBook에서 모집과 협의를 확인하는 구성원
- 방을 처음 만든 사람, 중간에 참여한 사람, 실제 주문한 사람
- **방향 확정**: 주문 완료 처리는 방 생성자 전용 권한이 아닙니다.
- **미확정**: 방 삭제, 마감 변경과 최종 가게 지정 등 다른 권한을 누구에게 허용할지 정하지 않았습니다.
- **미확정**: 초기 사용 범위가 특정 팀인지 전사인지 정하지 않았습니다.
- **미확정**: 동시에 연결될 사용자와 활성 방의 예상 규모를 정하지 않았습니다.

## 핵심 가치

### 1. 상태와 대화의 결합

구조화된 정보와 자유로운 대화를 같은 점심방 안에 둡니다.

### 2. 낮은 참여 비용

모집 발견, 참여, 메뉴 등록, 완료 확인까지 짧은 동작으로 끝내는 것을 지향합니다.

### 3. 유연한 협업

가게 결정은 필요할 때만 투표하고, 실제 주문과 완료 처리는 특정 역할에 고정하지 않습니다.

### 4. 반복 맥락의 재사용

자주 주문하는 가게와 과거 주문 정보를 새 모집의 출발점으로 활용할 가능성을 탐색합니다.

### 5. 회사 환경에 맞춘 집중

범용 플랫폼보다 전 사원이 MacBook과 배달의민족을 사용하는 실제 환경에 최적화합니다.

### 6. 운영 부담 최소화

제품을 유지하기 위해 별도 서버 비용과 운영 업무가 생기지 않도록 합니다.

## 제품 제약

| 항목 | 현재 판단 |
|------|-----------|
| 지원 플랫폼 | **방향 확정**: macOS만 고려하고 Windows는 비목표 |
| 주문 수단 | **방향 확정**: 실제 메뉴 탐색, 주문과 결제는 배민 앱에서 수행 |
| 주문 마감 | **방향 확정**: 방을 만들 때 주문 마감 시각을 설정하며 해당 시각에는 신규 참여만 닫힘 |
| 백엔드 운영 | **방향 확정**: 독자적인 중앙 서버를 운영하지 않음 |
| 통신 구조 | **가설**: P2P로 필요한 경험과 안정성을 제공할 수 있음 |
| 실행 가용성 | **미확정**: 앱 실행 중에만 발견·알림을 보장할지, 로그인 시 자동 실행 또는 백그라운드 상주를 요구할지 |
| 메뉴 입력 | **가설**: 초기에는 자유 형식 텍스트로 충분함 |
| 모바일 전달 | **가설**: macOS 공유 시트, AirDrop 또는 QR이 적합함 |
| 배민 연동 | **미확정**: 공식 가게·메뉴 조회 API 또는 연동 수단 존재 여부 |
| 결제 | **방향 확정**: LunchTime 안에서 주문이나 결제를 직접 처리하지 않음 |

`별도 서버를 운영하지 않는다`는 제품 제약은 확정했지만, `P2P로 충분한 안정성을 얻을 수 있다`는 아직 기술 가설입니다.

## 제품 가치 판단 후보

아직 기준값은 정하지 않았습니다. 현재 방식의 기준선을 관찰한 뒤 다음 신호에 목표를 설정합니다.

| 신호 | 확인하려는 가치 |
|------|----------------|
| 활성 모집과 현재 상태를 파악하는 데 걸리는 시간 | Slack 탐색보다 상태 가시성이 좋아졌는가 |
| 실제 주문에서 누락되거나 다시 확인한 개인 메뉴 수 | 메뉴 취합 오류가 줄었는가 |
| 같은 모집을 Slack에도 중복 작성한 비율 | 방별 채팅이 실제 협의 공간으로 작동하는가 |
| 생성→참여→메뉴→마감→완료 흐름 완주율 | 핵심 여정을 막힘없이 끝낼 수 있는가 |
| 일정 기간 내 반복 사용과 히스토리 재사용률 | 일회성 데모가 아니라 매일 쓸 가치가 있는가 |
| P2P 연결·동기화 실패와 복구 성공률 | 서버 없는 구조가 허용 가능한 신뢰성을 제공하는가 |

## 현재 비목표

- Windows 클라이언트
- 독자적인 중앙 백엔드 서버 운영
- LunchTime 안에서 배달 주문이나 결제를 직접 처리하는 기능
- 투표만으로 점심 메뉴를 결정하는 도구
- 배달의민족의 비공개 API나 스크래핑에 의존하는 구현
- 주문 완료 이후의 별도 `출발` 상태
