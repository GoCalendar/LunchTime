# 04. 가설과 검증 계획

> 상태: 검토가 필요한 초안
>
> 입력: [01. 제품 방향](./01_product_direction.md), [03. 문제와 유지할 강점](./03_pains_and_existing_strengths.md)
>
> 역할: 제품 방향이 성립하기 위해 맞아야 하는 전제를 드러내고 검증 방법을 제안한다.

우선순위는 기능 개발 순서가 아니라 `틀렸을 때 제품 방향에 미치는 위험`을 뜻합니다.

## 핵심 가설

| 위험도 | 가설 | 검증 질문 | 권장 검증 |
|--------|------|-----------|-----------|
| P0 | 회사 네트워크에서 Mac끼리 안정적으로 발견하고 연결할 수 있습니다. | 실제 사내망에서 기기 발견과 양방향 통신이 가능한가요? | 실제 사내망에서 2대 이상의 Mac으로 기술 스파이크 |
| P0 | 방 생성자가 이탈해도 방과 채팅을 유지할 수 있습니다. | 앱 종료, 잠자기, 네트워크 전환 후에도 다른 참여자가 계속 사용할 수 있나요? | 호스트 이탈·재접속·네트워크 변경 시나리오 테스트 |
| P0 | 중앙 서버 없이 늦은 참여자에게 현재 방 상태를 전달할 수 있습니다. | 어떤 Peer가 상태 스냅샷을 제공하고 충돌은 어떻게 처리하나요? | 상태 복제 프로토타입 |
| P0 | 중앙 서버 없이 늦은 참여자에게 합의된 범위의 과거 채팅을 전달할 수 있습니다. | 어느 범위까지 전달하며 이력이 서로 다를 때 무엇을 기준으로 하나요? | 메시지 복제 프로토타입과 이력 범위별 비교 |
| P0 | 사용자가 모집과 알림을 기대하는 시간에 앱이 Peer 연결을 유지할 수 있습니다. | 앱 실행 중에만 보장할지, 로그인 시 자동 실행 또는 백그라운드 상주가 필요한가요? 연결되지 않은 동안 놓친 이벤트를 어떻게 알리나요? | 앱 종료·백그라운드·로그인 직후 상태별 발견 및 알림 테스트 |
| P0 | 사용자는 Slack 외 별도 앱을 열 만큼 구조화된 점심방에서 가치를 느낍니다. | 앱 전환 비용보다 상태 가시성의 이득이 큰가요? | 실제 점심 흐름 관찰과 클릭 가능한 프로토타입 과업 테스트 |
| P1 | 방별 채팅이 Slack의 점심 대화를 충분히 대체하거나 보완합니다. | 사용자가 두 공간에 같은 내용을 중복 작성하지 않나요? | 실제 모집 한 건에서 대화 위치와 이탈 구간 확인 |
| P1 | 자유 형식 메뉴 입력이 초기 사용에 충분합니다. | 복사, 수정, 옵션 누락과 주문자 취합 과정에 문제가 없나요? | 실제 배민 메뉴 예시를 이용한 메뉴 등록·취합 테스트 |
| P1 | 검증된 한 가지 방식으로 배민 링크를 Mac에서 휴대폰에 빠르게 전달할 수 있습니다. | 공유 시트, AirDrop, QR, 복사 중 어느 방식이 원하는 기기와 배민 앱으로 가장 안정적으로 이어지나요? | 실제 배민 링크로 각 전달 방식 비교 후 기본 방식 1개 선정 |
| P1 | 참여자 누구나 완료 처리해도 혼란이나 오작동이 크지 않습니다. | 중복·오완료·동시 처리 시 어떤 복구가 필요한가요? | 완료 확인과 되돌리기 흐름 프로토타입 |
| P1 | 히스토리에서 가게와 링크를 재사용하면 모집 생성 수고가 줄어듭니다. | 새 방에 어떤 정보까지 복사해야 유용하고 부담스럽지 않나요? | 가게·링크만 복사하는 안과 이전 정보를 참고하는 안 비교 |
| P2 | 점심방과 별도인 라운지가 지속적으로 사용됩니다. | Slack과 중복되는 라운지를 왜 LunchTime에서 사용하나요? | 점심방과 함께 검증하는 안과 후속 검증하는 안을 제품 책임자가 비교 |
| P2 | 메뉴바 앱이 가장 적합한 진입점입니다. | 방 발견, 채팅과 긴 메뉴 입력에도 메뉴바 UI가 충분한가요? | 메뉴바 팝오버와 일반 창 방식 비교 |
| P2 | CLI가 별도 구현 비용을 정당화할 사용자 가치를 만듭니다. | 실제로 CLI를 반복 사용할 사용자가 있나요? | 수요 확인 전 구현 보류 |

## 반드시 별도로 확인할 사항

- 배민이 소비자용 가게·메뉴 조회 공식 공개 API 또는 파트너 연동을 제공하나요?
- 회사 네트워크가 Bonjour 또는 로컬 피어 발견을 허용하나요?
- 앱 실행, 로그인 시 자동 실행과 백그라운드 상주 중 어떤 상태를 지원 범위로 삼나요?
- 서로 다른 네트워크나 VPN 상태도 지원 범위에 포함되나요?
- 방별 채팅과 라운지 채팅을 얼마나 오래 보존해야 하나요?
- 로컬 히스토리를 기기 간 어떤 범위로 공유해야 하나요?
- 계정 서버 없이 사용자를 어떻게 식별하고 기기 변경을 어떻게 다루나요?
- 마감 이후 메뉴 수정은 허용하나요?
- 주문 완료를 되돌릴 수 있나요?
- 라운지의 접근 범위, 보존 기간과 알림 기본값은 무엇인가요?

## 권장 검증 순서

1. 사내망 P2P 발견·연결·재접속과 앱 실행 상태별 가용성 기술 스파이크
2. 방 상태와 채팅 복제의 최소 프로토타입
3. 현재 점심 모집 2~3건 관찰
4. MenuBar와 일반 창을 비교할 수 있는 핵심 흐름 프로토타입 테스트
5. 배민 링크의 Mac→휴대폰 전달 방식 비교
6. 실제 점심 모집에서 제한된 도그푸딩
7. 제품 책임자가 선택한 시점에 라운지 검증, 이후 CLI와 히스토리 재모집 후보 검증

검증하지 않은 가설을 `사용자가 원한다`, `문제가 해결된다`와 같은 확정 문장으로 PRD에 옮기지 않습니다.
