# 04. 가설과 검증 계획

> 상태: 2026-07-24 회의 이후 기술 스파이크 중심으로 갱신
>
> 입력: [01. 제품 방향](./01_product_direction.md), [03. 문제와 유지할 강점](./03_pains_and_existing_strengths.md)
>
> 역할: 확정된 제품 방향을 실제 사내 환경에서 제공하기 위해 검증해야 할 전제를 드러낸다.

MenuBar, QR, 구조화된 메뉴, 라운지 MVP와 11:00~14:30 라이프사이클은 더 이상 비교할 제품 가설이 아닙니다. 아래 우선순위는 기능 개발 순서가 아니라 `틀렸을 때 제품 방향에 미치는 위험`을 뜻합니다.

## 확정된 방향과 남은 검증의 경계

| 확정된 제품 동작 | 아직 검증할 것 |
|------------------|----------------|
| 사내 두 Wi-Fi가 연결되는 내부망에서 P2P 사용 | 두 Wi-Fi 사이 실제 라우팅, Peer 발견 방식과 방화벽 제약 |
| 각 Peer가 로컬 이벤트를 보유하고 서로 동기화 | 복제 자료구조, 충돌 병합, 삭제와 중복 제거 알고리즘 |
| 잠자기 복귀 후 제한된 catch-up과 활성 Room·라운지 또는 미해결 데이터 30초 대조 | OS 이벤트 신뢰도, 트리거별 최대 3회이면서 총 30초 이내인 세션과 실제 복구 시간 |
| 메뉴 누락 위험이면 주문 완료 차단 | 마감 전 참여 승인 ACK, 현재 참여자 최신 ACK와 never-shared 단독 예외 |
| 전송·로컬 데이터 암호화와 회사 WiFi 자동 신뢰 | 자동 신뢰 네트워크 경계, 암호화 세션과 악의적인 내부 Peer 위험 |
| 운영 데이터 최대 14일, 채팅은 앱 종료 시 삭제 | 메모리·디스크 경계, 만료 이벤트와 고아 데이터 처리 |
| 최소 가게 히스토리는 로컬에 장기 보존하고 공유 필드만 병합 | `trim(storeName) + raw inputLink` 완전 일치 키와 로컬 이용 횟수 분리 검증 |
| 14:30 쓰기 종료 뒤 최대 120초 finalization | 잠든 Mac의 wake 종료, 불완전 상세와 이후 제한된 재계산 |

| ID | 가설 | 실패 시 영향 | 권장 검증 |
|----|------|--------------|-----------|
| A-01 | 두 사내 Wi-Fi가 연결되는 내부망에서 Mac끼리 안정적으로 발견하고 양방향 연결할 수 있습니다. | 제품의 기본 연결 범위를 성립시키지 못함 | 두 Wi-Fi 교차 조합, VLAN·방화벽·절전 조건에서 2대 이상 Mac 테스트 |
| A-02 | 중앙 리더 없이 변경 이벤트를 여러 Peer가 교환해 Room 상태를 일관되게 재구성할 수 있습니다. | 생성자 이탈, 늦은 참여와 재접속에서 상태가 갈라짐 | 변경 로그·CRDT 등 후보로 상태 복제 프로토타입 |
| A-03 | 참여가 마감 전 승인 ACK를 받고 주문 완료 기기가 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 정확히 판정할 수 있습니다. | 참여나 메뉴가 조용히 누락되거나 과거·오프라인 Peer 때문에 주문이 불필요하게 막힘 | 마감 직전 승인·늦은 전달·ACK 없음·never-shared 단독·과거 Peer 이탈 조합 테스트 |
| A-04 | 의미 있는 트리거별 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 끝나는 세션과 활성 Room·라운지 또는 미해결 데이터·건강한 Peer 대상 30초 anti-entropy로 놓친 변경을 복구할 수 있습니다. | 잠자기 후 오래된 상태를 최신으로 오인하거나 실패 Peer가 타이머로 무한 재시도됨 | sleep/wake, SSID 전환, 앱 재실행, 제한 소진, 대상 없음과 14:30 정지 테스트 |
| A-05 | 회사 WiFi 자동 신뢰와 암호화를 함께 적용하고, 악의적인 내부 Peer가 평문을 읽을 수 있는 위험을 명확히 수용할 수 있습니다. | 자동 신뢰 경계가 외부망까지 넓어지거나 암호화가 무력화됨 | 네트워크 경계, MITM·재전송·로컬 파일 탈취와 내부 공격자 위협 모델 검토 |
| A-06 | 14일 이벤트 창과 메모리 전용 채팅을 유지하면서 현재 Room과 최소 히스토리를 정확히 복구할 수 있습니다. | 데이터가 과도하게 남거나 재시작 후 제품이 쓸 수 없음 | 앱 정상·비정상 종료, 모든 Peer 종료, 만료·삭제·고아 데이터 테스트 |
| A-07 | 14:30 종료·최대 120초 finalization과 이후 제한된 재동기화가 결정적으로 final snapshot과 성공 히스토리를 계산합니다. | 불완전 종료가 성공 주문으로 남거나 Peer별 상세가 갈라짐 | 14:30 직전 변경, 잠든 Mac wake, 120초 만료와 이후 수동 재계산 테스트 |
| A-08 | 예상 사용자와 활성 Room 규모에서 발견·전파 지연과 복구 성공률이 허용 범위 안에 있습니다. | 작은 데모 외에 사내 사용이 불가능함 | 파일럿 규모를 정한 뒤 부하·장시간 도그푸딩 |
| A-09 | 넉넉한 시계 오차 허용값으로 정상 사용은 막지 않으면서 잘못된 참여 승인·마감 변경을 차단하고, 14:30에는 쓰기를 안전하게 닫은 뒤 종료 스냅샷을 `확인 필요`로 보류할 수 있습니다. | 기기마다 참여·종료 유효성이 달라짐 | 정상 NTP 편차, 절전 복귀와 의도적 시계 변경 테스트 |

## 메뉴 누락 방지 검증 계약

P0 검증은 단순히 `메시지가 대체로 전달된다`로 통과할 수 없습니다.

1. 모든 참여·메뉴 변경에는 고유 이벤트 ID와 작성 Peer를 기록합니다.
2. 참여 요청은 마감 전에 승인 ACK를 받아야 하며, 마감 전 승인 이벤트는 늦게 도착해도 유효하고 ACK가 없으면 마감 시 실패합니다.
3. 주문 완료 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다. 과거·오프라인 Peer의 미응답만으로 차단하지 않습니다.
4. 로컬 단독 예외는 생성자가 처음부터 만들고 한 번도 다른 Peer에 공유하지 않은 Room에만 적용합니다.
5. 현재 참여자의 메뉴가 없거나 최신 리비전이 `이 기기에 저장됨`, `확인 필요`이면 주문 완료를 차단합니다.
6. 지원 조건에서 ACK된 메뉴가 주문 전 요약에서 조용히 누락되는 테스트가 한 건이라도 있으면 출시 게이트를 통과하지 못합니다.

## 반드시 별도로 확인할 사항

- Bonjour/mDNS 또는 다른 로컬 발견 방식이 두 사내 Wi-Fi 사이에서 동작하나요?
- 활성 Room·라운지 또는 미해결 데이터·건강한 Peer만 대상으로 하는 30초 anti-entropy와 트리거별 최대 3회이면서 총 30초 이내인 세션의 네트워크·배터리 부하는 감당 가능한가요?
- 앱 종료·잠자기·네트워크 전환 각각에서 어떤 이벤트를 OS로부터 신뢰할 수 있나요?
- 회사 WiFi 자동 신뢰가 정확히 어떤 네트워크 경계까지 적용되며 악의적인 내부 Peer 위험을 어떻게 수용하나요?
- 같은 메뉴·참여·주문 상태를 동시에 바꿨을 때 어떤 병합 규칙이 안전한가요?
- 같은 사용자의 여러 Room 참여 충돌과 마지막 참여자 취소를 모든 Peer가 같은 결과로 계산할 수 있나요?
- 14일이 지난 이벤트와 삭제 표식을 가진 오래된 Peer가 상태를 되살리지 않게 할 수 있나요?
- 메모리 전용 채팅이 macOS 스왑·진단 덤프에서 갖는 보안 한계를 어떻게 설명하나요?
- 입력 링크 문자열을 검증·정규화·경고 없이 그대로 QR로 만들 때 생성 불가능한 입력은 어떻게 기술적으로 처리하나요?
- Peer 닉네임 중복 시 짧은 기기 ID 표시가 사람을 충분히 구분하게 하나요?
- 초기 파일럿 사용자 수와 최대 Peer·Room 규모는 얼마인가요?

## 권장 검증 순서

1. 사내 두 Wi-Fi에서 Peer 발견·자동 신뢰 경계·암호화 연결 기술 스파이크
2. 고유 이벤트, 참여 승인·메뉴 ACK와 현재 참여자 기반 fail-closed 주문 점검 프로토타입
3. 트리거별 최대 3회이면서 총 30초 이내인 제한, 30초 주기 대조와 `이 기기에 저장됨/복제됨/동기화됨/확인 필요` 상태 테스트
4. 14:30 종료·최대 120초 finalization·잠든 Mac wake·시계 오차와 이후 재계산 테스트
5. 14일 만료, 메시지 ID 중복 제거·best-effort 순서와 완전 일치 히스토리 키 테스트
6. MenuBar→별도 창, raw 입력 QR과 구조화된 메뉴 과업 테스트
7. 실제 점심 모집에서 제한된 규모로 도그푸딩
8. 측정 결과로 지원 네트워크, 최대 규모, 복구 시간과 출시 게이트 확정

스파이크가 실패하면 중앙 서버를 임의로 추가하지 않습니다. 지원 범위를 좁히거나 실패 UX를 다시 설계하고 제품 결정을 받습니다.
