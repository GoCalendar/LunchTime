# 10. 결정 및 미결정 목록

> 상태: 제품 책임자 회의 결정을 반영한 체크포인트
>
> 입력: [01. 제품 방향](./01_product_direction.md)부터 [09. 범위 제안](./09_scope_proposal.md)까지
>
> 역할: 확정된 제품 결정, 이전 C-ID의 처리 결과와 구현 전에 기술 검증으로 확정할 계약을 추적한다.

## 제품 방향으로 정한 것

| ID | 결정 |
|----|------|
| D-01 | macOS만 지원하며 Windows는 고려하지 않습니다. |
| D-02 | LunchTime이 운영하는 독자적인 중앙 서버를 두지 않습니다. |
| D-03 | 중앙 정본 없이 각 Peer가 중요 데이터의 로컬 복제본을 가지고 버전 요약과 누락 레코드를 상호 동기화합니다. |
| D-04 | 점심방은 구조화된 정보와 방별 채팅을 함께 제공합니다. |
| D-05 | 점심방과 별개인 `라운지`를 MVP에 포함합니다. |
| D-06 | 배민 링크의 MVP Mac→휴대폰 전달 방식은 QR이며, 메뉴 탐색·실제 주문·결제는 배민에서 수행합니다. |
| D-07 | 개인 메뉴는 필수 메뉴명과 순서를 보존하는 0개 이상의 옵션 문자열로 구성된 고정 템플릿으로 작성합니다. |
| D-08 | 방 생성 시 주문 마감 시각을 반드시 설정합니다. 주문 상태가 `진행 중`이고 기존 마감 전이면 현재 참여 중인 생성자만 마감 시각을 변경할 수 있습니다. 주문 완료 상태에서는 잠기며, 한 번 도달한 마감은 뒤로 미뤄 모집을 다시 열 수 없습니다. |
| D-09 | F-67의 메뉴 완전성 조건을 충족하면 참여자 누구나 주문 완료 처리할 수 있습니다. |
| D-10 | 주문 완료는 Room의 종료 상태가 아니며 14:30 전에는 취소·정정할 수 있습니다. 14:30 사용자 종료가 최종 쓰기 경계이고, 이후 최대 120초의 시스템 finalization을 수행합니다. |
| D-11 | 히스토리는 각 기기에 로컬 저장합니다. 상세 주문 기록은 최대 14일 보존하고 재모집용 가게명·입력 링크·최근 주문 완료 시각·로컬 이용 횟수는 14일 이후에도 남깁니다. 같은 가게는 `trim(storeName) + raw inputLink`의 완전 일치로만 합치며 입력 링크에는 trim·URL 정규화·redirect 추적을 하지 않습니다. 재모집은 가게명과 입력 링크가 채워진 생성 화면에서 새 마감 시각을 확인한 뒤 시작하며 즉시 Room을 만들지 않습니다. |
| D-12 | 배민의 비공개 API나 스크래핑에 의존하지 않습니다. |
| D-13 | 주문 마감 뒤에도 승인된 기존 참여자는 14:30까지 채팅·주문 상태를 변경할 수 있고, 메뉴는 주문 상태가 `진행 중`일 때 수정할 수 있습니다. 14:30부터 모든 Room과 라운지는 사용자에게 닫히고 열람 전용입니다. |
| D-14 | 일일 점심 라이프사이클은 `Asia/Seoul` 기준 11:00~14:30입니다. |
| D-15 | 현재 사용하는 두 사내 WiFi의 같은 SSID와 교차 SSID를 P2P 지원 대상 내부망으로 취급합니다. |
| D-16 | 앱 시작·foreground·잠자기 복귀·네트워크 변경·의미 있는 새 Peer 발견과 수동 새로고침은 각각 최대 3회이면서 총 30초 이내인 독립 동기화 세션을 시작하고 둘 중 먼저 도달하면 종료합니다. 실패하면 멈추고, 이후 새 의미 있는 트리거나 수동 요청만 새 세션을 시작합니다. |
| D-17 | 참여·메뉴·마감·주문 상태는 누락 방지가 최우선인 중요 데이터입니다. 참여는 마감 전에 승인 ACK를 받아야 하고 메뉴 최신 리비전도 ACK되어야 합니다. 주문 완료 기기는 모든 현재 참여자의 최신 ACK 리비전을 보유해야 하지만, 과거에 보였거나 현재 오프라인인 Peer라는 이유만으로 완료를 막지는 않습니다. |
| D-18 | 중요 운영 이벤트와 상세 주문 기록은 최대 14일 관리하며 오래된 tombstone·고아 데이터는 정리할 수 있습니다. 재모집용 최소 히스토리 필드는 장기 로컬 보존합니다. |
| D-19 | 방·라운지 채팅은 앱 메모리에 두고 앱 종료 시 해당 기기에서 삭제합니다. 메시지 ID로 중복을 제거하고 순서는 best-effort로 복원합니다. 활성 운영일에 재실행하면 살아 있는 Peer가 보유한 당일 채팅을 best-effort로 다시 받되 완전 복구나 전역 순서는 보장하지 않습니다. |
| D-20 | Peer 간 통신과 로컬 중요 데이터를 암호화하며 장기 키는 macOS Keychain으로 관리합니다. 지원 회사 WiFi에서 발견한 Peer는 별도 승인 없이 자동 신뢰하고, 악의적인 내부 Peer 위험을 수용한 위협 모델을 사용합니다. |
| D-21 | 하나의 Room은 하나의 가게·주문 그룹을 나타내며 사용자는 투표 대신 Room에 참여합니다. 한 명만 참여해도 정상 주문할 수 있습니다. |
| D-22 | MenuBar를 고정 진입점으로 사용하고 오늘의 Room·참여자·주문 상태 아이콘을 보여주며, 실제 기능은 별도 Window에서 진행합니다. |
| D-23 | Interactive CLI는 MVP에서 제외하고 후속 후보로 둡니다. |
| D-24 | 라운지는 메인 Window에서 진입합니다. MenuBar 라운지 버튼과 라운지 읽지 않음 표시는 MVP에서 제외합니다. |
| D-25 | 11:00 전체 점심 선택 알림은 MVP 이후 기능으로 둡니다. |
| D-26 | 한 사용자는 같은 운영일에 하나의 Room에만 참여합니다. 정상 흐름에서는 주문 상태가 `진행 중`일 때 현재 참여를 먼저 철회한 뒤 다른 Room으로 이동합니다. 분산 충돌로 두 Room 참여가 동시에 생기면 D-40을 따릅니다. |
| D-27 | 11:00~14:30에 앱과 Mac이 깨어 있는 동안 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 대상 Peer가 있을 때만 30초 간격의 anti-entropy를 수행합니다. 대상이 없거나 14:30이 되면 주기 대조를 멈추고, 참여·메뉴·주문 상태는 주기를 기다리지 않고 즉시 전파합니다. 제한 세션을 소진해 실패한 Peer는 정기 타이머만으로 다시 시도하지 않습니다. |
| D-28 | 종료된 Room의 구조화 상세는 최대 14일 동안 모든 자동 신뢰 회사 WiFi Peer가 열람할 수 있습니다. 채팅은 포함하지 않으며, 14:30 finalization이 불완전해도 확인 가능한 상세를 표시합니다. |
| D-29 | 참여 요청은 요청 기기가 Room 마감 전에 유효한 승인 ACK를 받아야 확정됩니다. 이렇게 확정된 참여 이벤트는 다른 Peer에 늦게 도착해도 유효하며, 요청 기기가 마감까지 ACK를 받지 못한 요청은 실패합니다. |
| D-30 | 주문 완료 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다. 과거 또는 오프라인 Peer의 미응답만으로 완료를 차단하지 않습니다. 로컬 단독 예외는 생성자가 처음부터 만들었고 한 번도 다른 Peer에 공유되지 않은 Room에만 허용합니다. |
| D-31 | 14:30에 사용자 쓰기를 즉시 닫고 일일 상태를 `최종 확정 중`으로 바꿔 14:30~최대 14:32의 120초 동안 finalization을 수행한 뒤 `종료`합니다. 14:30에 잠든 Mac은 깨어날 때 같은 순서를 즉시 적용하고 그 시점부터 최대 120초 finalization을 수행합니다. |
| D-32 | 동기화 상태는 `이 기기에 저장됨`, `복제됨`, `동기화됨`, `확인 필요` 네 가지이며 `동기화 중`은 상태가 아니라 활동 표시입니다. 트리거별 세션은 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 종료하며, 실패 뒤에는 새 의미 있는 트리거나 수동 요청만 새 세션을 시작합니다. 주기 대조는 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 Peer가 있을 때만 30초 간격으로 실행하고 대상 부재 또는 14:30에 멈춥니다. 제한을 소진해 실패한 Peer는 정기 타이머만으로 재시도하지 않습니다. |
| D-33 | 주문 상태가 `진행 중`이면 참여자는 철회할 수 있고 마지막 참여자 철회는 Room을 취소합니다. 현재 참여 중인 생성자는 주문 상태가 `진행 중`이면 Room 전체를 취소할 수 있고, `주문 완료`이면 먼저 완료를 되돌린 뒤 취소합니다. 생성자가 철회하면 재참여해도 복구되지 않는 관리자 권한을 잃고 가게명·입력 링크·마감 메타데이터는 잠깁니다. 취소 Room은 14:30까지 열람 전용이며 최근 Room 상세와 재사용 히스토리에서 제외합니다. |
| D-34 | 종료 Room 상세는 모든 자동 신뢰 회사 WiFi Peer가 최대 14일 열람합니다. 불완전 종료는 성공 히스토리를 만들지 않지만 확인된 상세는 볼 수 있습니다. 이후 수동 요청이나 새 의미 있는 트리거가 제한된 재동기화를 시작해 final snapshot과 성공 히스토리를 다시 계산할 수 있습니다. |
| D-35 | 권한은 설치·기기에 결합한 불투명 사용자 ID로 판정하고 닉네임은 표시값으로만 사용합니다. Connection / Settings는 Peer 닉네임, 온라인 여부, 마지막 확인 시각과 동기화 상태만 표시하며 신뢰·ACK 관리는 노출하지 않습니다. 중복 닉네임에는 짧은 기기 ID를 붙입니다. 회사 WiFi에서 발견한 Peer는 자동 신뢰하며, 인식된 Peer는 활성 Room의 메뉴·채팅을 읽을 수 있고 쓰기는 참여자에게만 허용합니다. |
| D-36 | QR은 사용자가 입력한 링크 문자열을 검증·정규화·경고 없이 그대로 인코딩합니다. 히스토리의 동일 가게 키도 `trim(storeName) + raw inputLink`의 완전 일치이며 입력 링크에는 trim·URL 정규화·redirect 추적을 하지 않습니다. |
| D-37 | 채팅 메시지는 고유 ID로 중복 제거하고, Peer 복구 시 순서는 best-effort로 구성합니다. 완전 복구나 모든 Peer의 동일한 전역 순서는 보장하지 않습니다. |
| D-38 | Peer 간 시계 차이를 확인하고 허용 범위를 넘으면 참여 승인, 마감 변경과 주문 상태 변경처럼 시각에 의존하는 사용자 쓰기를 차단합니다. 14:30 쓰기 종료는 안전하게 적용하되 종료 스냅샷의 유효성·완전성 판정은 `확인 필요`로 두며, 허용 오차는 넉넉한 값으로 기술 스파이크에서 확정합니다. |
| D-39 | MenuBar는 14:30이 되면 당일 Room을 즉시 제거합니다. 종료 상세와 불완전 finalization 상태는 별도 Window에서 확인합니다. |
| D-40 | 같은 사용자의 여러 Room 참여 이벤트가 동시에 유효해진 충돌은 자동 선택하지 않습니다. 사용자가 유지할 Room을 직접 선택할 때까지 두 참여 모두 `확인 필요`로 두고 양쪽 Room의 주문 완료를 차단합니다. 선택하지 않은 Room에는 해당 사용자의 참여와 활성 메뉴를 철회하는 이벤트를 함께 기록·전파합니다. |

## 공통 상태와 용어

### 직교 상태 축

| 축 | 값 | 규칙 |
|----|----|------|
| 일일 상태 | 대기 / 활성 / 최종 확정 중 / 종료 | 11:00에 활성화하고 14:30에 사용자 쓰기를 차단한 뒤 finalization 동안 `최종 확정 중`, 완료 뒤 `종료`입니다. |
| 모집 상태 | 모집 중 / 모집 마감 / 모집 취소 | 마감은 신규 참여만 차단하고 취소는 Room 전체를 열람 전용으로 만듭니다. |
| 주문 상태 | 진행 중 / 주문 완료 | 참여자 누구나 변경하며 14:30 전에는 완료 취소·정정이 가능합니다. |
| 동기화 상태 | 이 기기에 저장됨 / 복제됨 / 동기화됨 / 확인 필요 | 지속 상태만 나타냅니다. `동기화 중`은 별도의 일시적 활동 표시입니다. |

이 축들은 UI 화면의 네 구역을 뜻하지 않습니다. 예를 들어 `모집 마감 + 주문 진행 중 + 동기화됨`처럼 동시에 성립할 수 있습니다.

동기화 상태의 의미는 다음과 같습니다.

- `이 기기에 저장됨`: 현재 기기에 기록됐지만 다른 Peer가 정확한 리비전을 ACK하지 않았습니다.
- `복제됨`: 적어도 한 건강한 Peer가 정확한 리비전을 ACK했습니다.
- `동기화됨`: 현재 제한 세션의 건강한 대상 Peer와 활성 Room·라운지 또는 미해결 동기화 데이터 대조를 마쳤습니다.
- `확인 필요`: 충돌, 실패, 과도한 시계 차이 또는 해결하지 못한 공백이 있습니다.
- `동기화 중`: 위 상태를 바꾸기 전 현재 세션이 실행 중임을 나타내는 활동이며 지속 상태가 아닙니다.

### 용어

| 용어 | 의미 |
|------|------|
| 일일 세션 | Asia/Seoul 기준 당일 11:00~14:30의 사용자 쓰기 가능 구간 |
| 주문 마감 시각 | Room 생성 시 정하는 11:00 이상 14:30 미만의 절대 시각 |
| 모집 열림 | 신규 참여를 받을 수 있는 상태 |
| 모집 마감 | 주문 마감 시각에 도달해 신규 참여만 닫힌 상태 |
| 주문 완료 | 실제 주문을 끝냈다는 되돌릴 수 있는 주문 상태 |
| 사용자 쓰기 종료 | 14:30에 자동 발생하며 Room·라운지를 즉시 열람 전용으로 전환 |
| finalization | 사용자 쓰기 종료 또는 잠든 Mac의 복귀 시점부터 최대 120초 동안 `최종 확정 중` 상태로 최종 스냅샷과 히스토리를 계산하는 시스템 활동 |
| 일일 종료 | finalization이 끝나 `최종 확정 중`에서 전환한 상태. 정상 시각에는 늦어도 14:32, 잠든 Mac은 wake 후 최대 120초 안에 도달 |
| 중요 데이터 | 참여·메뉴·마감·주문 상태처럼 주문 누락 방지를 위해 ACK와 복구가 필요한 구조화 데이터 |
| 최소 히스토리 | 재모집을 위해 14일 이후에도 남기는 가게명·raw 입력 링크·최근 주문 완료 시각·로컬 이용 횟수 |

## 메뉴 누락 방지 불변식

### 성공으로 표시하는 조건

- 모든 참여·메뉴 변경은 먼저 암호화된 로컬 중요 데이터 저장소에 기록합니다.
- 참여 요청은 마감 전에 승인 ACK를 받아야 `참여됨`입니다. 마감 전 승인됐으면 다른 기기에 늦게 도착해도 유효하고, 마감까지 ACK가 없으면 실패합니다.
- ACK 전에는 `이 기기에 저장됨`이며 성공한 것처럼 표시하지 않습니다.
- 메뉴 수정은 새 버전이므로 새 ACK를 받기 전까지 주문 가능한 최신 제출로 취급하지 않습니다.
- 로컬 단독 제출 예외는 생성자가 처음부터 만들고 한 번도 공유하지 않은 단독 Room에만 적용합니다.

### 주문 완료 조건

주문 완료를 실행하는 기기는 현재 참여자 목록과 모든 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다. 하나라도 누락됐거나 `이 기기에 저장됨`, `확인 필요`이면 완료 동작을 차단하고 문제 대상을 보여줍니다. 과거 또는 오프라인 Peer가 응답하지 않는다는 이유만으로는 차단하지 않으며, MVP에는 이 조건을 무시하는 강제 완료를 제공하지 않습니다.

이 규칙은 연결되지 않은 Peer가 만든 정보를 전역적으로 추측하지 않습니다. 대신 다른 Peer에게 알려지지 않은 고립 참여·메뉴를 성공으로 표시하지 않습니다.

## 데이터 보존 결정

| 데이터 | 로컬 저장 | Peer 동기화 | 보존 |
|--------|-----------|-------------|------|
| 참여·메뉴·마감·주문 상태 | 암호화된 지속 저장소 | 버전 레코드와 ACK | 최대 14일 |
| 방·라운지 채팅 | 앱 메모리 | 메시지 ID 중복 제거·best-effort 순서 | 앱 종료 시 해당 기기에서 삭제, 활성 운영일에 살아 있는 Peer가 보유한 당일 메시지만 재수신 |
| 최근 Room 구조화 상세 | 암호화된 지속 저장소 | 자동 신뢰 회사 WiFi Peer 간 복제·열람 | 최대 14일, 채팅 제외 |
| 가게명·입력 링크·최근 주문 완료 시각·로컬 이용 횟수 | 암호화된 지속 저장소 | 완전 일치 키의 공유 필드만 best-effort 병합 | 재모집을 위해 14일 이후에도 유지 |

삭제된 채팅, 기기별 히스토리 차이와 고아 데이터는 허용합니다. 중요 데이터는 누락 가능성을 성공 상태로 숨기지 않으며, 14일 보존 기간 안에서는 버전 대조와 누락 재전송으로 복구합니다.

## 기술 스파이크로 확정할 동작 계약

제품 선택은 끝났지만 다음 구현 세부는 실제 사내망·기기 검증 결과로 확정합니다.

### 네트워크와 복구

- 두 WiFi의 같은 SSID A, 같은 SSID B, A→B, B→A에서 Peer 발견과 직접 연결이 가능한지
- multicast DNS가 교차 SSID를 통과하지 않을 때 사용할 대체 발견 흐름
- 앱 시작·foreground·잠자기 복귀·네트워크 변경 후 동기화 완료 목표 시간
- 트리거별 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 끝나는 제한과 활성 Room·라운지 또는 미해결 동기화 데이터·건강한 Peer만 대상으로 하는 30초 주기 대조의 스케줄링·배터리·네트워크 비용
- 기기 시계 차이와 임의 변경이 있을 때 11:00 활성화, 마감 승인, 14:30 종료와 늦게 도착한 이벤트의 유효 시각을 판정하는 넉넉한 허용 오차
- Peer 버전 요약, 누락 레코드 요청과 재전송 프로토콜
- 동시 수정의 버전 비교와 충돌 표시·해결 규칙

### 보존과 복구

- 14일 TTL을 계산하는 기준 시각과 Peer 시계 차이 처리
- tombstone을 보존하는 기간과 garbage collection 안전 조건
- `trim(storeName) + raw inputLink` 완전 일치 키와 최소 히스토리 병합 규칙의 구현 적합성
- 14:30 직전 주문 상태 변경, 최대 120초 finalization과 늦은 재동기화가 경쟁할 때의 순서

### 식별과 보안

- 회사 WiFi 발견 Peer 자동 신뢰의 네트워크 경계와 악의적인 내부 Peer를 수용하는 위협 모델
- 설치·기기에 결합한 사용자 ID의 저장, 기기 키 연결과 식별 정보 손실 시 새 사용자 처리
- 기기별 키 생성과 암호화 세션 수립
- 기기 교체·분실 시 로컬 키 처리
- 전송 암호화 프로토콜과 로컬 저장 암호화 방식
- 발견 패킷에 노출해도 되는 최소 메타데이터

### QR

- 실제 기기의 QR 인식률과 스캔 뒤 외부 배민 앱·페이지 동작 관찰(제품 보장 조건은 아님)
- 입력 문자열을 그대로 인코딩했을 때의 QR 생성 가능 범위
- 링크 검증·경고를 하지 않는 사용자 흐름
