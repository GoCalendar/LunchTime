## PRD 작성을 막는 미결정

### 제품 경험

- 초기 사용 범위를 특정 팀으로 할지 전사로 할지
- 여러 점심방 동시 참여를 허용할지
- 방 생성 시 생성자를 자동 참여 처리할지
- 마감 후 개인 메뉴 수정을 허용할지
- 투표 결과를 최종 가게로 자동 확정할지
- 방 삭제, 마감 시각 변경과 최종 가게 지정 권한을 누구에게 허용할지
- 주문 완료 확인과 되돌리기를 제공할지
- 히스토리에 저장할 가게, 링크, 메뉴, 참여자와 채팅의 범위
- 히스토리 재모집에서 자동 복사할 정보
- 히스토리 재모집을 즉시 생성으로 할지, 미리 채운 생성 화면 진입으로 할지

### 채팅과 알림

- 방별 채팅의 보존 기간
- 라운지의 접근 범위와 보존 기간
- 새 모집, 마감, 완료와 채팅 알림의 기본값
- 방별·라운지별 음소거 방식
- 읽지 않음의 범위와 동기화 방식

### P2P와 데이터

- 실제 회사 네트워크에서 지원 가능한 연결 범위
- 앱 실행 중에만 발견·알림을 보장할지, 로그인 시 자동 실행 또는 백그라운드 상주를 지원할지
- 앱이 실행·연결되지 않은 동안 놓칠 수 있는 이벤트와 복귀 시 안내 방식
- Peer 사용자 식별 방식
- 방 상태와 메시지의 정본 선정 방식
- 방 생성자가 이탈했을 때의 연속성 수준
- 늦은 참여자에게 전달할 상태와 이력 범위
- 동시 수정 충돌 해결 방식
- 로컬 히스토리의 기기 간 공유·복구 범위
- 전송 보호와 접근 제어 수준

### 배민 경계

- 공식 가게·메뉴 조회 API 또는 파트너 연동 존재 여부
- Mac에서 휴대폰으로 링크를 넘길 기본 방식 1개와 실패 시 대안
- 공유 링크에서 공식적으로 이용 가능한 정보 범위

## 제품 가치 판단 기준의 미결정

[01. 제품 방향](./01_product_direction.md)의 지표 후보는 아직 목표값이 없습니다. 현재 방식의 기준선을 먼저 관찰하고 첫 PRD를 확정할 때 최소한 다음 항목의 판단 기준을 함께 정합니다.

- 활성 모집과 현재 상태를 파악하는 데 걸리는 시간
- 실제 주문에서 누락되거나 다시 확인한 개인 메뉴 수
- 같은 모집을 Slack에도 중복 작성한 비율
- 생성부터 주문 완료까지 핵심 흐름 완주율
- 일정 기간 내 반복 사용률과 히스토리 재사용률
- P2P 연결·동기화 실패율과 복구 성공률

모든 지표를 출시 차단 조건으로 삼는 것은 아닙니다. 현재 방식의 기준값, 관찰 기간과 표본을 확보한 뒤 제품 가치 지표와 기술 신뢰성 게이트를 구분해 목표를 정합니다.

## Policies로 분리할 후보

다음 항목은 기능 요구사항보다 세부 동작 규칙에 가깝습니다. 결정되면 `docs/policies/` 아래에 필요한 문서만 생성합니다.

- 방 생명주기와 상태 전이
- 참여, 이탈과 주문 마감
- 개인 메뉴 등록·수정
- 가게 후보와 선택적 투표
- 주문 완료 권한, 확인과 되돌리기
- 방·라운지 채팅 보존과 알림
- P2P 동기화, 충돌과 복구
- 히스토리 저장과 재모집
- 사용자 식별, 접근과 전송 보호

## PRD로 승격하기 위한 조건

1. [04. 가설과 검증 계획](./04_assumptions_and_validation.md)의 P0 기술·제품 가설을 검증합니다.
2. [09. 범위 제안](./09_scope_proposal.md)의 MVP 후보를 제품 책임자가 검토합니다.
3. 위 `PRD 작성을 막는 미결정` 중 첫 MVP에 영향을 주는 항목을 결정합니다.
4. 현재 방식의 기준선을 관찰하고 첫 MVP의 제품 가치 지표와 기술 신뢰성 목표를 정합니다.
5. 검증 결과가 제한적인 항목은 지원 동작 또는 명시적 제품 제약으로 결정합니다.
6. 채택된 F-ID만 PRD에 옮깁니다.
7. 상태 전이, 권한, 보존과 예외는 Policies로 분리합니다.

## 바로 이어서 할 일

1. 사내망에서 P2P 발견·통신 기술 스파이크를 진행합니다.
2. 현재 점심 모집 2~3건을 관찰해 [02](./02_current_lunch_journey.md)와 [03](./03_pains_and_existing_strengths.md)을 보정합니다.
3. MenuBar와 일반 창을 비교할 수 있는 클릭 가능한 흐름을 만듭니다.
4. 배민 링크의 Mac→휴대폰 전달 방식을 실제 기기에서 비교하고 기본 방식 1개를 선정합니다.
5. 라운지를 첫 점심방 MVP와 동시에 검증할지 제품 책임자가 결정합니다.
6. 범위 제안을 리뷰한 뒤 첫 PRD와 필요한 Policies를 작성합니다.
