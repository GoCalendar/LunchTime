# 02. 현재 점심 주문 여정

> 상태: 검토가 필요한 초안
>
> 입력: [01. 제품 방향](./01_product_direction.md)
>
> 역할: LunchTime을 사용하기 전, Slack과 배달의민족을 이용하는 현재 방식을 정리한다.

아래 여정은 현재까지 제공된 설명을 구조화한 것이며, 사용자 관찰이나 인터뷰로 검증된 연구 결과가 아닙니다. 확인되지 않은 구간은 사실처럼 채우지 않고 `미확정`으로 남깁니다.

## 기본 여정

| 단계 | 근거 상태 | 현재 행동 | 현재 방식의 장점 | 불편 또는 공백 |
|------|-----------|-----------|------------------|----------------|
| 1. 모집 시작 | 제공된 사실 | 누군가 Slack에서 점심 주문 이야기를 꺼냅니다. | 이미 모두 사용하는 대화 공간에서 바로 시작할 수 있습니다. | 여러 모집과 업무 대화가 섞이며 활성 모집을 한눈에 보기 어렵습니다. |
| 2. 참여와 협의 | 제공된 사실 | 구성원들이 Slack 대화로 참여 의사와 의견을 나눕니다. | 자연어로 조건과 취향을 자유롭게 이야기할 수 있습니다. | 참여자와 현재 결정 상태를 대화에서 다시 찾아야 합니다. |
| 3. 가게 추천 | 제공된 사실 | 배민의 가게 공유 문구와 링크를 Slack에 전달합니다. | 특정 가게를 정확하게 전달할 수 있습니다. | 후보, 링크, 의견이 메시지 사이에 흩어집니다. |
| 4. 메뉴 확인 | 사용자 설명 기반 추론 | Mac에서 확인한 배민 링크를 휴대폰으로 넘겨 배민 앱에서 메뉴를 확인하는 흐름이 필요합니다. | 최종 주문 환경인 배민 앱에서 실제 메뉴를 확인합니다. | 현재 실제 전달 방식과 끊김 정도는 확인되지 않았습니다. |
| 5. 가게 결정 | 제공된 사실 | 참여자들이 대화하고 필요하면 투표합니다. | O/X로 단순화하기 어려운 이유와 조건을 함께 논의할 수 있습니다. | 어느 가게가 확정됐는지 구조적으로 표시되지 않습니다. |
| 6. 개인 메뉴 작성 | 제공된 사실 | 각자가 메뉴명과 옵션을 자유 형식으로 적습니다. | 다양한 메뉴·옵션 구조를 제한 없이 표현할 수 있습니다. | 여러 사람의 메뉴를 모으고 변경 내용을 확인하기 어렵습니다. |
| 7. 모집 종료 | 미확인 | 현재 Slack에서 언제, 어떻게 신규 참여를 닫는지 확인되지 않았습니다. | 확인되지 않음 | 명시적인 마감 방식이 있는지 확인해야 합니다. |
| 8. 실제 주문 | 제공된 사실 | 참여자 중 누군가 휴대폰의 배민 앱에서 주문합니다. | 주문 담당자가 방 생성자로 고정되지 않습니다. | 누가 주문 중인지, 주문이 끝났는지 명시적인 상태가 없습니다. |
| 9. 완료 공유 | 미확인 | 현재 Slack에서 주문 완료를 어떻게 알리는지 확인되지 않았습니다. | 확인되지 않음 | 완료 여부를 참여자 모두가 동일하게 인지하는지 확인해야 합니다. |
| 10. 다음 주문 | 제공된 사실·현재 방식 미확인 | 과거에 주문했던 가게를 다시 이용하는 경우가 많습니다. | 익숙한 가게를 반복해서 선택할 수 있습니다. | 과거 정보를 찾고 새 모집을 시작하는 현재 방식은 미확정입니다. |

## 점심과 무관한 대화

점심 모집과 별개로 회사 구성원 사이에는 잡담 수요가 있습니다. 현재 이 대화가 어떤 Slack 채널이나 방식으로 이루어지는지, LunchTime의 라운지가 어떤 가치를 추가해야 하는지는 아직 확인되지 않았습니다.

## 현재 정보의 한계

- 실제 모집 한 건에 걸리는 시간과 메시지 수를 측정하지 않았습니다.
- 동시에 몇 개의 모집이 열리는지 확인하지 않았습니다.
- 메뉴를 취합할 때 가장 자주 발생하는 실수를 확인하지 않았습니다.
- Mac에서 휴대폰으로 배민 링크를 현재 어떻게 전달하는지 확인하지 않았습니다.
- Slack 대화를 별도 앱의 채팅으로 옮길 의향을 확인하지 않았습니다.
