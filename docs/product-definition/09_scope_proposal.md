# 09. 범위 제안

> 상태: 제품 책임자 검토가 필요한 제안
>
> 입력: [06. 기능 후보 목록](./06_feature_inventory.md), [08. 목표 사용자 여정](./08_target_lunch_journey.md)
>
> 역할: 첫 번째 사용 가능한 제품의 구현 범위와, 그 범위를 확정하기 전에 통과해야 할 검증을 서로 구분해 제안한다.

이 문서는 PRD가 아니며 기능 채택을 확정하지 않습니다. 제품 방향에 포함된 기능이라도 첫 MVP보다 뒤에 구현할 수 있고, MVP 후보라도 검증 결과에 따라 지원 범위나 동작을 다시 설계해야 합니다.

## 두 개의 독립적인 판단 축

- **제품 구현 범위**는 어떤 기능을 언제 제공할지에 대한 제안입니다.
- **선행 검증**은 제안한 기능의 구현 가능 범위와 실패 조건을 확인하는 활동입니다.
- 선행 검증은 기능의 범위 분류를 대신하지 않습니다. 검증이 필요한 기능도 아래 구현 범위에서 빠지지 않아야 합니다.
- F-01부터 F-63까지는 제품 구현 범위 표에서 각각 한 번만 분류합니다. 하나의 기능이 여러 검증 게이트에 연결되는 것은 허용합니다.

## 제품 구현 범위

### 분류 기준

- **MVP 핵심 구현 후보**: 점심 모집 한 건을 처음부터 끝까지 완결하는 데 필요한 기능
- **MVP 동작 계약 확정 후보**: MVP 전에 지원 동작을 구현하거나, 지원하지 못하는 조건과 사용자 동작을 Policies에 명시해야 하는 기능
- **MVP 확장 후보**: 핵심 흐름이 성립한 뒤 같은 출시 범위에서 추가할 수 있는 기능
- **후속 제품 범위 후보**: 제품 방향에는 포함하지만 첫 주문 협의 흐름 이후에 다룰 수 있는 기능
- **외부 의존 조사 후보**: 외부의 공식 지원 범위에 따라 채택 여부가 달라지는 기능
- **인터페이스 가설**: 제품 진입 방식으로서 수요와 적합성을 먼저 확인할 기능
- **출시 시점 결정 대기**: 제품 방향에는 포함하지만 MVP 동시 제공 여부를 제품 책임자가 결정해야 하는 기능

| 구현 범위 분류 | 기능 ID | 제안 |
|----------------|---------|------|
| MVP 핵심 구현 후보 | F-02~F-04, F-06~F-07, F-09~F-15, F-17~F-26, F-32, F-34~F-42, F-49~F-52, F-56, F-58, F-61 | 생성→발견→참여→채팅→가게·배민 링크→휴대폰 전달→개인 메뉴→모집 마감→주문 완료→기본 히스토리 흐름을 구현합니다. F-21의 현재 상태 스냅샷과 F-61의 검증된 단일 모바일 전달 방식도 여기에 포함합니다. |
| MVP 동작 계약 확정 후보 | F-53~F-55, F-57, F-62~F-63 | 각 기능은 검증 뒤 지원 동작을 구현하거나, 지원하지 못하는 조건과 그때의 사용자 경험을 MVP 제약·정책으로 명시합니다. 검증 결과만 남기고 구현 범위에서 제거하지 않습니다. |
| MVP 확장 후보 | F-27, F-29~F-31, F-43~F-44 | 추가 공유 방식, 선택적 투표, 히스토리 기반 재모집을 핵심 흐름 뒤에 붙입니다. |
| 후속 제품 범위 후보 | F-05, F-08, F-16, F-28, F-59 | 정교한 활동 표시·알림, 다중 참여, 추가 모바일 전달 방식, 히스토리 복구·공유를 후속 범위로 제안합니다. |
| 외부 의존 조사 후보 | F-33 | 배민의 공식 지원 범위가 확인될 때만 검토하며 비공개 API나 스크래핑은 사용하지 않습니다. |
| 인터페이스 가설 | F-01, F-60 | MenuBar와 Interactive CLI는 기본 MVP를 전제하는 기능이 아니라, macOS 진입점 후보로 별도 검증합니다. |
| 출시 시점 결정 대기 | F-45~F-48 | 라운지를 주문 흐름 MVP와 동시에 제공할지, 후속 마일스톤으로 둘지 제품 책임자가 결정합니다. |

F-61은 `Mac에서 휴대폰으로 배민 링크를 전달할 수 있다`는 핵심 능력입니다. G-05에서 실제 사용할 방법을 정확히 하나 선택한 뒤 그 방법을 F-61의 구현 사양으로 반영합니다. 현재 표의 F-27과 F-28 분류는 두 방식을 추가 선택지로 제공한다는 전제의 제안입니다. G-05가 둘 중 하나를 기본 방식으로 선택하면 범위 확정 시 해당 F-ID도 MVP 핵심 구현으로 옮기고, 선택하지 않은 방식은 현재 분류를 유지합니다.

### 검증 민감 기능의 MVP 처리 계약

| 기능 | 지원할 수 있을 때 | 충분히 지원할 수 없을 때 |
|------|-------------------|---------------------------|
| F-53 생성자 이탈 뒤 방 유지 | 유지 조건과 승계 동작을 구현 | 생성자 이탈 시 유지되지 않는 조건, 상태 표시와 사용자의 다음 행동을 명시 |
| F-54 재연결·재동기화 | 지원하는 재연결 조건, 재시도와 복구 범위를 구현 | 복구할 수 없는 조건과 새로 참여하거나 다시 시작하는 흐름을 명시 |
| F-55 동시 수정 충돌 처리 | 일관된 충돌 처리 규칙을 구현 | 동시에 수정할 수 없는 항목이나 마지막 반영 규칙을 명시 |
| F-57 접근·전송 보호 | 실제 보장하는 접근 경계와 전송 보호를 구현 | 지원 네트워크, 신뢰 경계와 민감 정보 취급 제약을 명시 |
| F-62 늦은 참여자의 과거 채팅 | 전달할 메시지 범위와 순서를 구현 | 과거 채팅을 제공하지 않는 범위와 참여 시점 이후의 동작을 명시 |
| F-63 앱 실행·백그라운드 가용성 | 채택한 자동 실행·백그라운드 연결 동작과 상태 표시를 구현 | 발견·알림을 보장하는 실행 상태, 놓칠 수 있는 이벤트와 사용자의 다음 행동을 명시 |

### 라운지 출시 시점 결정

라운지는 제품 방향에 포함되어 있지만 이 문서에서 후속 범위로 단정하지 않습니다. 제품 책임자가 아래 두 선택지를 비교해 PRD 작성 전에 결정해야 합니다.

| 선택지 | 얻는 것 | 감수할 것 | 범위에 미치는 영향 |
|--------|---------|-----------|---------------------|
| 주문 흐름 MVP와 동시 제공 | 점심방과 잡담 공간을 분리한다는 전체 제품 방향을 처음부터 검증 | 상시 메시지 동기화, 이력, 읽지 않음·알림과 접근 정책까지 첫 출시 복잡도가 증가 | F-45~F-48을 MVP 구현 범위로 이동하고 G-02·G-04에서 라운지도 함께 검증 |
| 주문 흐름 이후 제공 | 첫 검증을 점심 모집과 주문 협의에 집중 | 라운지의 사용 가치와 방/잡담 분리 효과는 첫 출시에서 검증하지 못함 | F-45~F-48을 명시적인 후속 마일스톤으로 이동 |

결정할 때는 첫 사용자 집단이 라운지 없이도 LunchTime 안에서 반복적으로 협의할 가능성, 상시 P2P 연결 부담, 메시지 보존·알림 정책의 준비도를 함께 봅니다.

## MVP 성공 흐름 후보

1. 두 대 이상의 Mac이 지원되는 앱 실행·연결 상태에서 서로를 발견하며, 지원하지 않는 상태에서는 발견·알림 제약이 표시됩니다.
2. 사용자는 검증된 macOS 진입점에서 활성 점심방을 확인합니다.
3. 한 사용자가 제목과 주문 마감 시각으로 방을 만들고 다른 사용자가 참여합니다.
4. 늦게 참여한 사용자도 현재 방 상태를 전달받습니다.
5. 방별 채팅에서 가게 후보와 배민 링크를 공유합니다.
6. 사용자는 G-05에서 선택한 한 가지 기본 방식으로 배민 링크를 휴대폰에 전달하고 배민 앱에서 메뉴를 확인합니다.
7. 참여자별 메뉴를 등록하고 모아봅니다.
8. 주문 마감 시각이 되면 신규 참여만 닫히며 기존 참여자의 방과 채팅은 유지됩니다.
9. 실제 주문 후 참여자 누구나 완료 처리합니다.
10. 완료된 방이 각 참여자에게 기본 히스토리로 남습니다.

## 제품 가치와 성공 기준

아직 수치 목표를 임의로 정하지 않습니다. 현재 Slack·배민 중심 흐름의 기준선을 대표 사용자 집단에서 먼저 측정하고, 범위 확정 전에 각 지표의 목표값, 측정 기간과 책임자를 정합니다.

| 제품 가치 신호 | 기준선과 비교할 성공 방향 | 목표 설정 시점 |
|----------------|---------------------------|----------------|
| 활성 모집과 현재 상태를 파악하는 데 걸리는 시간 | 현재 방식보다 짧아짐 | 현재 여정 기준선 측정 후 |
| 실제 주문에서 누락되거나 다시 확인한 개인 메뉴 수 | 누락과 재확인이 줄어듦 | 현재 주문 표본의 기준선 측정 후 |
| 같은 모집을 Slack에도 중복 작성한 비율 | 핵심 협의가 LunchTime 안에서 끝나 중복 비율이 낮아짐 | 초기 사용 집단의 사용 행태 확인 후 |
| 생성→참여→메뉴→마감→완료 흐름 완주율 | 막힘 없이 전체 흐름을 완료하는 비율이 높아짐 | 프로토타입 또는 파일럿 기준선 확보 후 |
| 일정 기간 내 반복 사용과 히스토리 재사용률 | 일회성 사용이 아니라 반복 사용으로 이어짐 | 관찰 기간과 재사용 정의 합의 후 |
| P2P 연결·동기화 실패와 복구 성공률 | 실패가 허용 범위 안에 있고 복구 가능성이 높아짐 | G-01~G-03의 실제 환경 측정 후 |

수치 목표가 정해지기 전까지 이 표는 측정 후보이며 성공을 선언하는 기준이 아닙니다. PRD로 승격할 때는 채택한 지표마다 기준선, 목표값, 측정 방법과 관찰 기간을 기록합니다.

## 선행 검증

검증 게이트는 기능 목록과 별도 축으로 운영합니다. 각 게이트는 통과·실패만 기록하는 것이 아니라, 해당 기능에 반영할 구현 사양 또는 명시할 제품 제약을 산출해야 합니다.

| 게이트 | 연결 기능 | 반드시 남길 결과 |
|--------|-----------|------------------|
| G-01 사내망 발견·연결과 앱 가용성 | F-13, F-49~F-51, F-58, F-63 | 지원 네트워크와 앱 실행 상태, 발견·연결·알림 성공 기준, 실패 상태와 사용자 안내 |
| G-02 상태와 메시지 동기화 | F-19~F-21, F-50~F-55, F-58, F-62 | 현재 상태와 메시지의 전달 범위, 충돌·유실 처리, 지원하지 않는 조건의 정책 |
| G-03 생성자 이탈과 재접속 | F-53~F-54, F-59 | 생성자 종료, Mac 잠자기와 네트워크 전환별 보장 범위와 복구 흐름 |
| G-04 사용자 식별과 접근 | F-56~F-57 | 계정 서버 없는 식별 방식, 접근 경계, 전송 보호와 명시할 신뢰 가정 |
| G-05 배민 링크 모바일 전달 | F-25~F-28, F-61 | 실제 기기에서 사용할 기본 방법 하나, 지원 기기·전제 조건, 실패 시 대안 |
| G-06 macOS 진입점 | F-01, F-60 | MenuBar, 일반 앱 창과 CLI 후보의 사용 맥락을 비교한 기본 진입점 결정 |

### G-05. 배민 링크 모바일 전달의 완료 조건

- macOS 공유 시트·AirDrop, QR, 복사 후 기기 간 전달 등 실제 사용 가능한 후보를 대표 기기에서 비교합니다.
- 동작 수, 성공률, 지원 기기 조건과 실패 시 복구 방법을 확인합니다.
- 비교로 끝내지 않고 기본 방법을 정확히 하나 선택합니다.
- 선택한 방법의 화면·동작·지원 조건을 F-61의 핵심 구현 사양으로 넘깁니다.
- 한 가지도 허용 가능한 수준에 도달하지 못하면 배민 앱으로 이어지는 MVP 성공 흐름을 충족하지 못한 것으로 보고 범위를 확정하지 않습니다.

## 아직 보장하지 않는 것

검증과 동작 계약이 끝나기 전에는 다음을 제품 약속으로 표현하지 않습니다.

- 방 생성자가 사라져도 방이 항상 유지됩니다.
- 모든 메시지가 유실 없이 전달됩니다.
- 모든 네트워크 환경에서 동작합니다.
- 오프라인이었던 모든 기기가 자동으로 완전 복구됩니다.
- 늦게 참여한 사용자가 항상 전체 과거 채팅을 받습니다.
- 히스토리가 모든 사용자에게 항상 동일하게 보입니다.

## 범위 확정에 필요한 제품 책임자 검토

- MenuBar, 일반 앱 창과 CLI 중 어떤 macOS 진입점을 기본으로 삼을까요?
- 라운지를 주문 흐름 MVP와 동시에 제공할까요, 후속 마일스톤으로 둘까요?
- 앱 실행 중에만 발견·알림을 보장할까요, 로그인 시 자동 실행 또는 백그라운드 상주를 지원할까요?
- 기본 히스토리는 어떤 필드까지 저장해야 하나요?
- 선택적 투표를 첫 MVP에 포함해야 하나요?
- 히스토리 재모집은 즉시 방을 만들까요, 값이 채워진 생성 화면을 거칠까요?
- 주문 완료 되돌리기는 첫 MVP에 필요한가요?
