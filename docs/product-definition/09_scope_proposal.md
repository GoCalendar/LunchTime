# 09. 범위 제안

> 상태: 제품 책임자 회의 결정을 반영한 MVP 범위
>
> 입력: [06. 기능 후보 목록](./06_feature_inventory.md), [08. 목표 사용자 여정](./08_target_lunch_journey.md)
>
> 역할: 첫 MVP의 확정 범위, 후속 범위, 제외 기능과 구현 전에 동작 계약을 확정할 기술 항목을 구분한다.

이 문서는 PRD가 아니지만 제품 책임자가 결정한 MVP 경계와 그 근거를 기록합니다. 상세 요구사항은 PRD, 상태 전이·보존·권한·동기화와 실패 규칙은 Policies가 정본입니다.

## 범위 분류 원칙

- **확정 MVP**: 첫 배포에서 사용자에게 제공해야 하는 기능입니다.
- **확정 후속**: 제품 방향에는 남기되 MVP에서는 제공하지 않습니다.
- **Retired**: 현재 제품 모델과 맞지 않아 구현하지 않습니다. F-ID는 다시 사용하지 않습니다.
- **선행 검증·MVP 동작 계약**: MVP에 필요한 시스템 능력이지만 사내망과 실제 기기에서 검증한 뒤, 지원 동작 또는 실패 시 제약을 PRD·Policies에 명시해야 합니다.
- 선행 검증·MVP 동작 계약은 선택 범위가 아닙니다. 검증 결과가 기대보다 제한적이어도 조용히 제거하지 않고 지원 조건, 사용자에게 보일 상태와 복구 동작을 남깁니다.
- 아래 표는 F-01부터 F-74까지의 유일한 범위 분류입니다. 각 ID는 정확히 한 분류에만 속합니다.

## 확정한 제품 모델

### 하나의 Room은 하나의 주문 그룹이다

- Room은 하나의 가게와 배민 링크를 가집니다.
- 사용자는 후보 가게에 투표하지 않고 주문하고 싶은 Room에 참여합니다.
- 한 명만 참여한 Room도 정상적으로 주문할 수 있습니다.
- 한 사용자는 같은 운영일에 하나의 Room에만 참여합니다.
- MVP에서는 가게명을 Room의 식별 정보로 사용합니다. 별도 사용자 정의 방 제목·메모는 후속 범위입니다.
- 방 생성자는 자동 참여합니다. 주문 상태가 `진행 중`일 때만 참여를 철회할 수 있으며, 마지막 참여자의 철회는 Room을 취소합니다. 생성자가 철회하면 관리자 권한을 잃고 가게·입력 링크·마감 메타데이터는 잠깁니다.
- 현재 참여 중인 생성자는 진행 중 Room을 전체 취소할 수 있고, 주문 완료 상태라면 먼저 완료를 되돌립니다.
- 분산 충돌로 같은 사용자가 여러 Room에 참여한 상태가 되면 자동으로 하나를 고르지 않습니다. 사용자가 유지할 Room을 선택할 때까지 양쪽 참여를 `확인 필요`로 표시하고 두 Room의 주문 완료를 막은 뒤, 선택하지 않은 Room의 참여·활성 메뉴를 철회합니다.

### MenuBar와 별도 Window

- MenuBar는 LunchTime의 고정 진입점입니다.
- MenuBar에는 오늘의 Room, 참여자 정보, 주문 마감과 주문 상태 아이콘을 표시합니다.
- 14:30에는 사용자 종료와 동시에 MenuBar에서 당일 Room을 제거합니다. 종료 상세와 finalization 결과는 별도 Window에서 확인합니다.
- Room의 채팅, 개인 메뉴 입력, QR과 히스토리·라운지는 별도 Window에서 사용합니다.
- MenuBar의 라운지 바로가기와 라운지 읽지 않음 표시는 MVP에서 제외합니다.
- 라운지는 메인 Window의 최상위 탐색 영역에서 진입합니다.

### 배민 QR

- Room에 등록한 입력 링크 문자열을 그대로 QR로 표시합니다.
- 사용자는 Mac 화면의 QR을 휴대폰으로 스캔할 수 있습니다. 스캔 뒤 외부 배민 앱·페이지가 실제로 열리는지는 LunchTime이 보장하지 않습니다.
- MVP에서는 QR만 Mac→휴대폰 기본 전달 방식으로 제공합니다.
- 링크 복사, macOS 공유 시트와 AirDrop은 후속 범위입니다.
- 링크 형식은 검증·정규화하지 않고 경고도 표시하지 않습니다. QR 기술 검증은 입력 문자열을 그대로 인코딩할 수 있는 범위와 실제 기기 인식률을 확인합니다.

### 구조화된 개인 메뉴

- 참여자마다 하나의 활성 메뉴 제출을 가지며, 제출은 하나 이상의 메뉴 항목으로 구성됩니다.
- 각 메뉴 항목의 이름은 필수 단일행 문자열입니다.
- 각 메뉴 항목의 옵션은 순서를 보존하는 0개 이상의 문자열입니다.
- MVP에서는 옵션 가격을 별도 숫자로 해석하거나 검증하지 않습니다.
- 수정할 때 새 버전을 만들고 동기화 상태를 다시 계산합니다.

## 일일 라이프사이클

LunchTime의 회사 기준 시간대는 `Asia/Seoul`입니다.

| 구간 또는 이벤트 | 동작 |
|------------------|------|
| 11:00 이전 | 오늘의 Room·라운지에 쓰지 않으며 이전 히스토리만 열람합니다. |
| 11:00 | 오늘의 점심 세션을 활성화합니다. 전체 알림은 F-74 후속 기능입니다. |
| 11:00~주문 마감 시각 | Room 생성·참여·채팅과 주문 상태 변경을 허용합니다. 참여 요청은 마감 전에 승인 ACK를 받아야 하고, 메뉴는 주문 상태가 `진행 중`일 때 등록·수정합니다. |
| 주문 마감 시각 | ACK 없는 참여 요청을 실패 처리하고 신규 참여를 닫습니다. 마감 전에 승인된 참여 이벤트는 다른 Peer에 늦게 도착해도 유효합니다. 기존 참여자의 채팅·주문 상태 변경은 계속 허용합니다. |
| 주문 완료 | 참여자 누구나 처리할 수 있지만 Room을 종료하지 않습니다. 14:30 전에는 참여자 누구나 취소·정정할 수 있습니다. 완료 상태에서는 마감·메뉴·가게 정보를 잠급니다. |
| 14:30 | 모든 Room과 라운지를 사용자에게 즉시 닫고 열람 전용으로 전환하며 MenuBar에서 당일 Room을 제거합니다. |
| 14:30~최대 14:32 | 사용자 쓰기는 금지한 채 최대 120초 finalization으로 최종 스냅샷과 성공 히스토리를 계산합니다. |
| 이후 | 불완전 종료는 성공 히스토리를 만들지 않지만 확인된 상세는 열람할 수 있습니다. 수동 요청이나 새 의미 있는 트리거의 제한된 재동기화가 final snapshot과 히스토리를 다시 계산할 수 있습니다. |

주문 마감 시각은 당일 11:00 이상 14:30 미만으로 설정합니다. 주문 상태가 `진행 중`이고 기존 마감 전이면 현재 참여 중인 생성자만 바꿀 수 있으며, 한 번 마감에 도달하면 뒤로 미뤄 모집을 다시 열 수 없습니다. `주문 완료`는 되돌릴 수 있는 주문 축의 값입니다. 14:30에는 사용자 쓰기를 닫고 `최종 확정 중`으로 전환하며, 최대 120초 뒤 `종료`가 됩니다. 14:30에 잠든 Mac도 wake 시 같은 순서를 즉시 적용합니다.

## 서로 직교하는 상태 축

이 구조는 UI 섹션을 나누자는 뜻이 아니라 서로 독립적으로 바뀌는 도메인 상태를 하나의 단일 상태값으로 억지로 합치지 말자는 뜻입니다.

| 축 | 값 | 설명 |
|----|----|------|
| 일일 상태 | 대기 / 활성 / 최종 확정 중 / 종료 | 14:30에 사용자 쓰기를 막고 `최종 확정 중`으로 전환한 뒤 finalization이 끝나면 `종료`합니다. |
| 모집 상태 | 모집 중 / 모집 마감 / 모집 취소 | 마감은 신규 참여를 막고, 취소는 Room 전체를 열람 전용으로 만듭니다. |
| 주문 상태 | 진행 중 / 주문 완료 | 14:30 전에는 완료 취소·정정이 가능합니다. |
| 동기화 상태 | 이 기기에 저장됨 / 복제됨 / 동기화됨 / 확인 필요 | `동기화 중`은 상태가 아니라 현재 동작 중임을 나타내는 활동 표시입니다. |

메뉴 제출 인원과 참여 인원은 별도 상태가 아니라 위 상태에서 계산하는 진행 정보입니다.

## 메뉴 누락 방지 계약

메뉴 누락은 MVP의 최우선 안전 실패입니다. 채팅과 히스토리의 일시적 차이보다 참여·메뉴·마감·주문 상태의 완전성을 우선합니다.

### 참여와 메뉴 제출 성공 조건

- 모든 참여·메뉴 변경은 먼저 암호화된 로컬 중요 데이터 저장소에 기록합니다.
- 참여 요청은 마감 전에 승인 ACK를 받아야 `참여됨`으로 표시합니다. 승인 이벤트가 다른 Peer에 늦게 도착해도 유효하지만 마감까지 ACK가 없으면 실패합니다.
- 메뉴 최신 리비전은 Peer ACK 전까지 `이 기기에 저장됨`이며 주문 가능한 제출로 보지 않습니다.
- 로컬 단독 예외는 생성자가 처음부터 만들고 한 번도 다른 Peer에 공유하지 않은 단독 Room에만 적용합니다.
- 수정된 메뉴도 새 ACK를 받기 전까지 주문 가능한 최신 제출로 취급하지 않습니다.

### 주문 완료의 fail-closed 조건

주문 완료를 실행하는 기기는 다음 조건을 모두 만족해야 합니다.

1. 현재 참여자 전원의 최신 승인 ACK 리비전을 보유합니다.
2. 현재 참여자 전원의 최신 메뉴 ACK 리비전이 로컬에 있습니다.
3. 각 최신 리비전이 `복제됨` 또는 `동기화됨`이며 `이 기기에 저장됨`, `확인 필요` 상태가 없습니다.
4. 실제 주문용 메뉴 요약이 위 동기화 버전으로 생성됐습니다.

하나라도 충족하지 못하면 주문 완료를 차단하고 문제 대상을 보여줍니다. 과거에 보였거나 현재 오프라인인 Peer라는 이유만으로는 완료를 차단하지 않으며, MVP에는 강제 진행이나 경고 무시 동작을 제공하지 않습니다.

네트워크에서 완전히 고립돼 다른 Peer에게 알려지지 않은 참여를 전역적으로 증명할 수는 없습니다. 따라서 마감 전 승인 ACK가 없는 참여는 실패로 처리하고, 주문 완료는 현재 참여자의 최신 승인·메뉴 ACK 리비전만으로 판단합니다.

## 데이터 계층과 보존

| 데이터 계층 | 저장과 동기화 | 보존 |
|-------------|---------------|------|
| 중요 운영 데이터 | 참여, 메뉴, 마감, 주문 상태를 암호화해 로컬에 기록하고 버전 단위로 Peer에 복제 | 최대 14일 |
| 방·라운지 채팅 | 고유 메시지 ID로 중복을 제거하고 앱 메모리에서 best-effort 순서로 교환 | 앱 종료 시 해당 기기에서 즉시 삭제, 현재 운영일에 살아 있는 Peer가 보유한 메시지만 재수신 |
| 최근 Room 구조화 상세 | 종료된 주문의 참여·메뉴·상태를 암호화해 복제하고 모든 자동 신뢰 회사 WiFi Peer에게 열람 전용으로 표시, 채팅 제외 | 최대 14일 |
| 재모집용 최소 히스토리 | 가게명·입력 링크·최근 주문 완료 시각·로컬 이용 횟수를 보존하고 `trim(storeName) + raw inputLink` 완전 일치 키의 공유 필드만 병합 | 14일 이후에도 보존 |

앱 재실행 시 보유 Peer가 있으면 최대 14일의 구조화 운영 데이터와 현재 운영일의 채팅을 다시 받을 수 있습니다. 모든 Peer가 삭제했거나 보유 Peer를 만나지 못하면 채팅이 사라져도 허용합니다. 최소 히스토리의 기기별 차이, 삭제와 고아 데이터도 허용합니다.

14일 정리 시 삭제 표식과 오래된 충돌 정보가 다시 살아나지 않도록 tombstone과 garbage collection 규칙을 Policies에서 정합니다.

## P2P 복구 원칙

- 각 Peer가 자신이 보유한 버전 요약을 교환하고 상대에게 없는 레코드만 재전송합니다.
- 앱 시작, foreground 전환, 잠자기 복귀, 네트워크 경로 변경, 의미 있는 새 Peer 발견과 수동 요청은 각각 최대 3회이면서 총 30초 이내인 동기화 세션을 시작하고 둘 중 먼저 도달하면 종료합니다.
- 한 세션이 실패하면 자동 반복을 멈춥니다. 새 의미 있는 트리거나 사용자의 새 수동 요청만 별도 제한 세션을 시작합니다.
- 11:00~14:30에는 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 대상 Peer가 있을 때만 30초 간격으로 주기 동기화합니다. 대상이 없거나 14:30이 되면 멈추고, 제한을 소진한 실패 Peer는 정기 타이머만으로 재시도하지 않습니다.
- `동기화 중`은 네 가지 지속 상태와 별개인 활동 표시입니다.
- 동일 SSID뿐 아니라 두 사내 WiFi 사이의 발견과 직접 통신도 지원 목표입니다.
- 두 WiFi가 같은 내부망을 사용하더라도 multicast DNS가 경계를 통과한다는 뜻은 아니므로 같은 SSID A, 같은 SSID B, A→B, B→A를 각각 검증합니다.
- Peer 시계 차이를 확인하고 허용 범위를 벗어나면 참여 승인, 마감 변경과 주문 상태 변경 같은 시간 의존 사용자 쓰기를 차단합니다. 14:30 쓰기 종료는 안전하게 적용하되 종료 스냅샷의 유효성·완전성 판정은 `확인 필요`로 두며, 넉넉한 허용 오차는 기술 스파이크에서 확정합니다.

## 보안 원칙

- Peer 간 전송과 로컬 지속 데이터는 암호화합니다.
- 로컬에 지속되는 중요 운영 데이터와 히스토리는 암호화합니다.
- 장기 키는 macOS Keychain으로 관리합니다.
- 발견 패킷에는 가게, 메뉴와 채팅 같은 민감한 본문을 싣지 않습니다.
- 지원 회사 WiFi에서 발견한 Peer는 별도 승인 없이 자동 신뢰합니다.
- 인식된 Peer는 활성 Room의 메뉴와 채팅을 읽을 수 있고, 참여자만 Room 데이터를 쓸 수 있습니다.
- 이 모델은 악의적인 내부 Peer가 평문을 읽을 수 있는 위험을 수용합니다. 스파이크는 네트워크 경계와 암호화 세션 수립을 검증하며 별도 신뢰 승인 UI를 만들지 않습니다.

## 라운지 MVP

- 정식 명칭은 `라운지`입니다.
- 라운지는 점심방과 분리된 최상위 공간이며 MVP에 포함합니다.
- 11:00~14:30에 메시지를 작성하고 14:30 이후에는 열람만 할 수 있습니다.
- 메시지는 앱 메모리에만 두며 앱 종료 시 해당 기기에서 삭제합니다.
- 활성 운영일에 재실행하면 살아 있는 Peer가 메모리에 보유한 당일 메시지를 best-effort로 다시 받을 수 있습니다.
- MenuBar 라운지 버튼, 읽지 않음 표시와 세분화된 알림은 MVP에 포함하지 않습니다.

## MVP 성공 흐름

1. 사용자는 11:00 이후 MenuBar에서 오늘의 Room과 상태를 확인합니다.
2. 한 사용자가 가게명, 배민 링크와 주문 마감 시각을 입력해 Room을 만들고 자동 참여합니다.
3. 다른 사용자가 두 사내 WiFi 중 어느 쪽에 있더라도 Room을 발견하고 참여합니다.
4. 참여는 마감 전에 승인 ACK를 받아야 하며, ACK가 없으면 마감 시 실패합니다.
5. 사용자는 QR을 휴대폰으로 스캔하고 외부 배민 앱·페이지에서 메뉴 확인을 시도합니다. QR 이후 동작은 LunchTime의 성공 보장이 아닙니다.
6. 메뉴명과 옵션 템플릿으로 개인 메뉴를 제출하고 동기화 상태를 확인합니다.
7. 주문 마감 시각에는 신규 참여만 닫히고 기존 참여자의 협의는 계속됩니다.
8. 주문 완료 처리 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다.
9. 참여자 누구나 주문 완료 처리하거나 14:30 전에 취소·정정할 수 있습니다.
10. 14:30에는 Room과 라운지가 닫히고 MenuBar에서 당일 Room이 사라지며 최대 120초 finalization을 시작합니다.
11. 모든 자동 신뢰 회사 WiFi Peer는 종료 Room의 구조화 상세를 최대 14일 읽을 수 있습니다. 불완전 종료는 성공 히스토리를 만들지 않지만 이후 제한된 재동기화로 다시 계산할 수 있습니다.

## 기술 신뢰성 기준

D-27의 30초 anti-entropy 안전 목표를 제외한 정량 목표는 기술 스파이크에서 기준선을 측정한 뒤 PRD에 기록하되, 다음 항목은 반드시 측정합니다.

- 같은 SSID와 교차 SSID의 발견·연결 성공률
- wake·foreground·네트워크 복귀 후 동기화 완료 시간
- 중요 레코드의 누락 탐지·복구 성공률
- 메뉴 제출 ACK 성공률과 대기·충돌 비율
- 주문 완료 차단이 실제 누락·대기 항목을 놓치지 않는지
- 14일 정리 후 삭제 데이터의 재등장 여부
- 회사 WiFi 자동 신뢰 경계와 전송·저장 암호화 검증
- QR 스캔 성공률과 이후 외부 배민 동작 관찰값(제품 보장 조건은 아님)

## 구현 Issue 확정 전 확인

- 회사 WiFi 자동 신뢰 경계와 악의적인 내부 Peer를 수용하는 위협 모델을 보안 스파이크에서 검증합니다.
- 두 사내 WiFi의 교차 SSID 발견이 불가능하면 지원 제약과 대체 발견 흐름을 결정합니다.
- 14일 TTL과 `trim(storeName) + raw inputLink` 완전 일치 키의 인코딩·병합 동작을 기술 스파이크에서 검증합니다. 입력 링크에는 trim·URL 정규화·redirect 추적을 적용하지 않습니다.
- 트리거별 최대 3회이면서 총 30초 이내인 제한, 30초 주기 대조와 14:30~최대 14:32 finalization의 실제 복구 목표값을 측정합니다.
- 시간 의존 동작을 막을 시계 차이 허용 오차를 넉넉한 값으로 확정합니다.
- 제품 가치 기준선과 목표값은 실제 점심 흐름 관찰 후 별도로 정합니다.
