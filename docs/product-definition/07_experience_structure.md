# 07. 경험 및 정보 구조

> 상태: 검토가 필요한 설계 제안
>
> 입력: [06. 기능 후보 목록](./06_feature_inventory.md)
>
> 역할: 모든 기능 후보를 macOS 경험의 화면과 시스템 영역에 배치한다.

MenuBar와 Interactive CLI는 아직 인터페이스 가설입니다. 먼저 활성 방을 확인하고 주요 영역으로 이동할 수 있는 macOS 진입 Surface가 필요하며, 어떤 형태가 가장 적합한지는 검증을 거쳐 정합니다. MenuBar를 채택하더라도 채팅과 긴 메뉴 입력까지 작은 팝오버에 모두 넣기보다, 빠른 진입점과 현황판으로 사용하고 방·라운지·히스토리는 별도 창으로 여는 구성을 제안합니다.

## Surface 구성

| ID | Surface | 역할 | 연결 기능 |
|----|---------|------|-----------|
| S-01 | macOS 진입 Surface | 활성 방 요약, 알림, 새 방·라운지·히스토리 진입 | F-02~F-08, F-13 |
| S-02 | 새 방 Sheet | 제목, 필수 주문 마감 시각, 초기 가게·메모 입력 | F-09~F-12 |
| S-03 | Room Window | 참여자, 상태, 가게, 모바일 전달, 메뉴, 완료, 방별 채팅 | F-14~F-40, F-61~F-62 |
| S-04 | History Window | 완료 기록 탐색과 재모집 | F-41~F-44 |
| S-05 | Lounge Window | 점심방과 분리된 잡담 | F-45~F-48 |
| S-06 | Preferences / Connection | 사용자 식별, 알림, 실행 가용성, 연결·동기화 상태 | F-08, F-49~F-59, F-62~F-63 |
| H-01 | MenuBar Popover | S-01을 구현할 빠른 진입 인터페이스 후보 | F-01 |
| H-02 | Interactive CLI | 동일 Core를 이용하는 대체 인터페이스 후보 | F-60 |

## S-01. macOS 진입 Surface

### 목적

- LunchTime을 빠르게 엽니다.
- 현재 모집이 있는지 한눈에 확인합니다.
- 중요한 이벤트와 연결 상태를 최소한으로 보여줍니다.
- 검증 결과에 따라 MenuBar Popover, 일반 앱 창 등 실제 형태를 결정합니다.

### 정보

- 활성 방: 제목, 모집 열림/마감, 마감 시각, 참여자 수
- 읽지 않은 방·라운지 활동
- 새 방 만들기
- 라운지, 히스토리, 설정 진입
- P2P 연결 또는 동기화 문제 표시

### 미확정

- MenuBar Popover를 채택할지
- MenuBar를 채택할 경우 팝오버 안에서 채팅과 메뉴 입력까지 허용할지
- 활성 방 정렬 기준
- 여러 방 동시 참여를 허용할지

## S-02. 새 방 Sheet

### 목적

적은 입력으로 모집을 시작합니다.

### 정보

- 방 제목 또는 모집 메뉴
- 필수 입력: 주문 마감 시각
- 선택 입력: 초기 가게, 배민 링크, 메모
- 생성 결과와 주변 Peer 전파 상태

### 미확정

- 방 생성 시 생성자를 자동 참여 처리할지
- 히스토리 재모집 시 어떤 값을 미리 채울지

## S-03. Room Window

### 영역 제안

1. **상단 요약**: 모집 열림/마감, 마감 시각, 참여자, 주문 완료
2. **가게 결정**: 후보, 배민 링크, 검증된 한 가지 Mac→휴대폰 전달 방식, 선택적 투표
3. **개인 메뉴**: 내 메뉴 편집과 참여자별 메뉴 모아보기
4. **방별 채팅**: 구조화되지 않는 협의와 시스템 이벤트

### 중요한 원칙

- 구조화된 정보와 대화는 같은 방의 맥락을 공유합니다.
- 가게 추천과 메뉴 입력을 채팅 메시지에서 다시 찾아야 하지 않도록 별도 영역에 둡니다.
- 주문 마감 시각이 되면 모집 상태만 `모집 마감`으로 바뀌고 신규 참여만 닫힙니다.
- 기존 참여자는 마감 후에도 방과 채팅에 접근할 수 있습니다.
- 마감 후 개인 메뉴를 수정할 수 있는지는 Policies에서 결정할 질문으로 남깁니다.
- 늦게 참여한 사용자는 F-21로 현재 방 상태 스냅샷을 받고, F-62로 합의된 범위의 과거 채팅을 받습니다.
- 주문 완료는 전체 Room의 종료 상태입니다.

## Room 상태 단순화 제안

기존의 `투표 중`, `메뉴 등록 중`은 동시에 일어날 수 있으므로 전체 Room의 단일 상태로 두기 어렵습니다. 다음과 같이 독립된 축으로 나누는 방안을 제안합니다.

| 축 | 값 |
|----|----|
| 참여 상태 | 모집 열림 / 모집 마감 |
| 가게 결정 | 미정 / 선택됨 |
| 메뉴 진행 | 등록 인원 / 참여 인원 |
| 주문 상태 | 미완료 / 주문 완료 |

투표와 채팅은 상태가 아니라 방 안에서 사용할 수 있는 활동으로 봅니다. `모집 마감`은 신규 참여 가능 여부만 바꾸며 기존 참여자의 방과 채팅 접근을 끝내지 않습니다. 이 구조와 마감 후 메뉴 수정 규칙은 아직 Policies로 확정되지 않은 제안입니다.

## S-04. History Window

### 목적

- 완료된 주문을 확인합니다.
- 반복 이용하는 가게와 주문 맥락을 참고합니다.
- 채택될 경우 과거 기록에서 새 모집을 시작합니다.

### 미확정

- 모든 참여자에게 동일한 히스토리를 보장할지
- 저장할 가게, 링크, 메뉴, 참여자와 채팅의 범위
- 새 방으로 자동 복사할 정보와 참고로만 보여줄 정보

## S-05. Lounge Window

### 목적

점심방과 무관한 잡담을 점심 모집 흐름과 분리해 제공합니다.

### 미확정

- 라운지의 접근 범위
- 메시지 보존 기간
- 기본 알림과 음소거 정책
- Slack과 다른 사용 이유

## S-06. Preferences / Connection

### 목적

- Peer에서 보이는 이름과 식별 정보를 관리합니다.
- 방과 라운지 알림을 구분해 설정합니다.
- 현재 연결·동기화 상태와 문제를 숨기지 않고 보여줍니다.
- 로그인 시 자동 실행 또는 백그라운드 상주 설정과 현재 앱 가용성을 보여줍니다.
- 앱이 실행·연결되지 않아 발견이나 알림을 보장할 수 없는 상태를 안내합니다.
- 지원하는 네트워크 범위와 데이터 보존 방식을 안내합니다.

## 주요 이동 흐름

```text
검증된 macOS 진입 Surface
├── 새 방 만들기 → 새 방 Sheet → Room Window
├── 활성 방 선택 → Room Window
├── 라운지 → Lounge Window
├── 히스토리 → History Window → (후보) 새 방 Sheet
└── 설정/연결 → Preferences / Connection
```

MenuBar Popover가 검증을 통과하면 위 진입 Surface의 구현 후보가 됩니다. Interactive CLI는 동일 Core를 사용하는 별도 가설입니다.
