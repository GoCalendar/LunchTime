# 07. 경험 및 정보 구조

> 상태: 2026-07-24 회의 결정을 반영한 경험 구조
>
> 입력: [06. 기능 후보 목록](./06_feature_inventory.md), [LunchTime MVP PRD](../prd/01_lunchtime_mvp.md)
>
> 역할: 채택한 MVP 기능을 macOS Surface와 분산 시스템 영역에 배치한다.

MenuBar를 오늘의 현황을 빠르게 확인하는 기본 진입점으로 확정합니다. 채팅, 구조화된 메뉴 입력과 연결 문제 해결은 작은 팝오버에 넣지 않고 별도 창에서 수행합니다. Interactive CLI는 MVP에서 제외합니다.

## S-01. MenuBar Popover

### 목적

- LunchTime을 빠르게 엽니다.
- 오늘 생성된 Room과 주문 상태를 별도 창을 열기 전에 확인합니다.
- 메뉴 누락 가능성이 있는 동기화 경고를 숨기지 않습니다.

### MVP 정보와 동작

- Room별 가게명
- 참여자 이름, 공간이 좁을 때는 참여자 수
- 주문 마감 시각
- `모집 중`, `모집 마감`, `주문 완료` 상태 아이콘
- 별도 `확인 필요` 동기화 경고
- 선택한 Room Window 열기
- 새 Room 만들기와 전체 앱 창 열기

14:30이 되면 당일 Room을 MenuBar에서 즉시 제거합니다. finalization 진행·불완전 상태와 종료 상세는 별도 Window에서 확인합니다.

### 후속 범위

- MenuBar의 라운지 바로가기
- Room·라운지 읽지 않음 배지
- 메시지 알림과 정교한 읽지 않음 집계
- 팝오버 안에서 채팅하거나 메뉴를 편집하는 기능

## S-02. 새 Room Sheet

### 목적

가게 하나의 모집을 적은 입력으로 시작합니다.

### 정보

- 필수: 가게명
- 필수: 배민 링크
- 필수: 현재 시각 이후이며 14:30 미만인 주문 마감 시각
- 생성 결과의 로컬 기록과 Peer 전파 상태

생성자는 생성과 동시에 참여자가 됩니다. 한 명만 참여한 Room도 정상 상태입니다. 한 사용자는 같은 운영일에 하나의 Room에만 참여하며, 다른 Room으로 이동하거나 새 Room을 만들려면 주문 상태가 `진행 중`일 때 기존 참여를 철회합니다. 동시 참여 충돌이 생기면 사용자가 하나를 선택하기 전까지 양쪽 Room의 주문 완료를 막고, 선택하지 않은 Room의 참여·활성 메뉴를 철회합니다. 가게 후보 목록과 투표 설정은 제공하지 않습니다.

히스토리의 `다시 모집`으로 들어오면 가게명과 배민 링크를 미리 채우고 사용자가 주문 마감 시각을 확인한 뒤 생성합니다.

## S-03. Room Window

### 영역

1. **상단 요약**: 가게, 참여 상태, 주문 마감, 참여자, 주문 상태와 14시 30분 종료
2. **배민 QR**: Room의 배민 링크를 휴대폰으로 전달
3. **개인 메뉴**: 내 메뉴명과 반복 옵션 편집, 참여자별 주문 요약
4. **주문 전 점검**: 메뉴 미제출·`이 기기에 저장됨`·`확인 필요`·충돌과 Peer 최신성 확인
5. **주문 상태**: 14:30 전 주문 완료·되돌리기와 허용된 Room 취소
6. **방별 채팅**: 구조화되지 않는 협의와 시스템 이벤트
7. **연결 상태**: 마지막 대조, Peer 수, 수동 새로고침

### 중요한 원칙

- 한 Room에는 한 가게만 있으며 사용자는 해당 가게 주문에 참여합니다.
- 구조화된 정보와 대화는 같은 맥락을 공유하지만 채팅이 참여·메뉴·주문 상태를 대신하지 않습니다.
- 지원 회사 WiFi에서 인식된 Peer는 활성 Room의 메뉴와 채팅을 읽을 수 있고, 참여자만 메뉴·채팅·주문 상태를 쓸 수 있습니다.
- 참여 요청은 주문 마감 전에 승인 ACK를 받아야 합니다. 마감 전에 승인됐으면 늦게 전달돼도 유효하고, ACK가 없으면 마감 시 실패합니다.
- 주문 마감 시각에는 신규 참여만 닫힙니다. 기존 참여자는 주문 진행 중이면 14:30 전까지 자신의 메뉴를 수정하고 채팅을 계속 사용할 수 있습니다.
- 주문 상태가 `진행 중`이고 기존 마감 전이면 현재 참여 중인 생성자만 마감 시각을 바꿀 수 있습니다. 주문 완료 상태에서는 잠기고, 이미 도달한 마감을 뒤로 미뤄 모집을 다시 열 수 없습니다.
- 주문 완료는 14:30 전까지 되돌릴 수 있는 상태이며 Room 종료가 아닙니다.
- 주문 완료 상태에서는 참여, 가게 정보와 메뉴를 잠그고 채팅만 유지합니다. 완료를 되돌리면 원래 주문 마감 상태에 따라 편집 가능 여부를 다시 계산합니다.
- 참여자는 주문 진행 중에 철회할 수 있고 마지막 참여자의 철회는 Room을 취소합니다. 현재 참여 중인 생성자는 진행 중 Room을 전체 취소할 수 있고, 주문 완료 상태라면 먼저 완료를 되돌립니다.
- 생성자가 철회하면 재참여해도 복구되지 않는 관리자 권한을 잃고 가게명·입력 링크·마감 메타데이터를 잠급니다. 취소 Room은 앱 창의 오늘 Room 목록에서 14:30까지 열람 전용으로 접근할 수 있으며 Recent Room과 히스토리에는 남기지 않습니다.
- 14:30 자동 종료가 사용자 최종 상태이고 이후 모든 입력을 비활성화합니다. 시스템은 최대 120초 finalization만 수행합니다.
- 주문 완료 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다. 과거·오프라인 Peer의 미응답만으로는 차단하지 않습니다.
- 로컬 단독 예외는 생성자가 처음부터 만들고 한 번도 다른 Peer에 공유하지 않은 단독 Room에만 적용합니다.
- 메뉴가 없거나 최신 리비전이 `이 기기에 저장됨`, `확인 필요`이면 주문 완료를 막습니다.
- Room 채팅은 고유 메시지 ID로 중복 제거하고 Peer별 수신 순서를 best-effort로 정렬합니다. 완전 복구와 전역 순서는 보장하지 않습니다.

### 개인 메뉴 템플릿

| 필드 | 입력 |
|------|------|
| 메뉴명 | 필수 한 줄 텍스트 |
| 옵션 | 0개 이상 반복 가능한 텍스트 행 |

추가 금액은 옵션 문자열 일부로 보존하며 MVP에서 별도 계산하거나 검증하지 않습니다. 참여자는 메뉴 항목을 하나 이상 등록할 수 있습니다.

참여자별 요약에는 메뉴 내용과 함께 마지막 변경 시각, 로컬 저장과 Peer ACK 상태를 표시합니다.

### 배민 QR

- MVP의 공식 Mac→휴대폰 전달 동작은 `QR 보기` 하나입니다.
- QR은 Room에 입력된 링크 문자열을 검증·정규화하지 않고 그대로 기기 안에서 생성합니다.
- 링크 형식에 대한 경고나 차단을 제공하지 않습니다.
- AirDrop과 macOS 공유 시트는 MVP에서 제공하지 않습니다.

## Room 상태 구조

Room 상태는 서로 독립적인 네 축으로 계산합니다.

| 축 | 값 |
|----|----|
| 일일 상태 | 대기 / 활성 / 최종 확정 중 / 종료 |
| 모집 상태 | 모집 중 / 모집 마감 / 모집 취소 |
| 주문 상태 | 진행 중 / 주문 완료 |
| 동기화 상태 | 이 기기에 저장됨 / 복제됨 / 동기화됨 / 확인 필요 |

메뉴의 `미제출` 여부와 제출된 메뉴의 동기화 상태는 Room의 다섯 번째 상태 축이 아니라 참여자별 메뉴에서 계산하는 파생 정보입니다.

`동기화 중`은 위 지속 상태를 대체하지 않는 활동 표시입니다. `주문 완료`와 `모집 마감`은 14:30 이전의 상태이며, 어느 상태에 있든 14:30이 되면 사용자 `자동 종료`가 우선합니다.

## S-04. History Window

### 목적과 정보

- 반복 이용하는 가게명과 배민 링크를 다시 사용합니다.
- 최근 주문 완료 일시와 로컬 이용 횟수를 보여줍니다.
- `다시 모집`으로 값이 채워진 새 Room Sheet를 엽니다.

가게 히스토리는 14일 운영 이벤트와 구분되는 최소 로컬 레코드입니다. 같은 가게는 `trim(storeName) + raw inputLink`가 모두 완전히 같을 때만 합치며 입력 링크는 trim·URL 정규화·redirect 추적을 하지 않습니다. 채팅, 이전 참여자, 개인 메뉴와 전체 Room 이벤트를 포함하지 않습니다.

## S-05. Lounge Window

### 목적과 범위

- 정식 명칭은 `라운지`입니다.
- 점심방과 무관한 잡담을 별도 최상위 공간에서 나눕니다.
- MVP에 포함하고 11:00~14:30에 메시지를 작성합니다.
- 14:30 이후에는 현재 메모리에 남은 당일 메시지를 읽기만 합니다.
- 앱 종료 시 해당 기기의 채팅을 삭제합니다.
- 재실행 시 살아 있는 Peer의 당일 채팅을 다시 받을 수 있지만 복구를 보장하지 않습니다.

MenuBar 라운지 바로가기, 읽지 않음과 알림은 MVP에 포함하지 않습니다.

## S-06. Connection / Settings

### 목적

Connection / Settings에는 다음 Peer 목록만 제공합니다.

- Peer 닉네임
- 온라인 여부
- 마지막 확인 시각
- `이 기기에 저장됨`, `복제됨`, `동기화됨`, `확인 필요` 동기화 상태와 별도 활동 표시

같은 닉네임이 둘 이상이면 닉네임 뒤에 짧은 기기 ID를 붙입니다. 신뢰 승인·차단과 ACK 내역은 이 화면에 표시하지 않습니다. 지원 회사 WiFi에서 발견한 Peer는 자동 신뢰하며, 수동 재동기화는 Connection / Settings, Room과 Recent Room Detail의 문제 해결 동작으로 제공합니다.

## S-07. Recent Room Detail

- 모든 자동 신뢰 회사 WiFi Peer가 구조화 운영 데이터가 남아 있는 최대 14일의 종료 Room을 엽니다.
- 가게명·배민 링크, 최종 주문 상태, 참여자·개인 메뉴 스냅샷과 주요 시스템 이벤트를 열람 전용으로 보여줍니다.
- 채팅은 메모리 전용이므로 지속되는 최근 Room 상세에 포함하지 않습니다.
- 14일 뒤 상세가 만료돼도 S-04의 최소 가게 히스토리는 계속 남을 수 있습니다.
- 14:30 finalization이 불완전하면 성공 히스토리는 만들지 않되 확인된 상세와 `불완전` 상태를 보여줍니다.
- 수동 요청이나 새 의미 있는 트리거가 최대 3회이면서 총 30초 이내이고 둘 중 먼저 도달하면 끝나는 재동기화 세션을 시작해 final snapshot과 성공 히스토리를 다시 계산할 수 있습니다.
- 취소 Room은 Recent Room Detail과 재사용 히스토리에서 제외합니다.

## 주요 이동 흐름

```text
MenuBar Popover
├── 새 Room 만들기 → 새 Room Sheet → Room Window
├── 오늘의 Room 선택 → Room Window
└── 전체 앱 열기
    ├── Room Window
    ├── Lounge Window
    ├── History Window → 새 Room Sheet
    ├── Recent Rooms → Recent Room Detail
    └── Connection / Settings(Peer 목록)
```

## 복귀 동기화 흐름

```text
잠자기 해제 / 앱 전면 전환 / 네트워크 변경 / 새 Peer 연결
→ 최대 3회이면서 총 30초 이내, 둘 중 먼저 도달하면 끝나는 제한 세션 시작
→ 건강한 Peer와 Peer별 버전 요약 비교
→ 누락된 14일 이내 변경 요청
→ 중복 제거·충돌 판정
→ Room·메뉴·주문 상태 재계산
→ 마지막 동기화 시각과 확인 필요 상태 갱신
```

실패한 세션은 멈추고 새 의미 있는 트리거나 수동 요청만 새 제한 세션을 시작합니다. 11:00~14:30에 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 대상 Peer가 있을 때만 30초 간격으로 반복하며, 대상이 없거나 14:30이 되면 주기 대조를 멈춥니다. 제한을 소진한 실패 Peer는 정기 타이머만으로 재시도하지 않습니다. 잠든 Mac은 wake 시 즉시 사용자 종료를 적용한 뒤 최대 120초 finalization을 수행합니다.

## 시스템 영역

화면에 직접 속하지 않지만 MVP를 성립시키는 능력은 다음 시스템 영역에서 담당합니다.

| ID | 시스템 영역 | 역할 | 연결 기능 |
