# 05. 어떻게 하면 질문

> 상태: 검토가 필요한 초안
>
> 입력: [01. 제품 방향](./01_product_direction.md), [03. 문제와 유지할 강점](./03_pains_and_existing_strengths.md), [04. 가설과 검증 계획](./04_assumptions_and_validation.md)
>
> 역할: 실제 해결 방법을 떠올릴 수 있는 질문과 가설적 답을 만든다.

아래 답은 확정 해법이나 요구사항이 아닙니다. 다음 문서에서 기능 후보를 빠짐없이 도출하기 위한 가설적 답입니다.

## 상태와 대화

### Q-01. 어떻게 하면 대화의 유연성을 유지하면서 현재 주문 상태를 바로 알 수 있을까요?

- **가설적 답**: 방 상단에 참여자, 가게, 마감, 사람별 메뉴와 주문 상태를 고정 요약하고 그 아래에 방별 채팅을 둡니다.
- 상태 변경은 채팅에도 시스템 메시지로 남기는 방안을 검토합니다.

### Q-02. 어떻게 하면 여러 점심 모집이 동시에 있어도 대화와 상태가 섞이지 않게 할 수 있을까요?

- **가설적 답**: 모집 한 건을 독립된 Room으로 만들고 활성 방 목록에서 마감 시각과 현재 상태를 함께 보여줍니다.
- 여러 방 동시 참여 허용 여부는 미확정입니다.

### Q-03. 어떻게 하면 가게 결정이 단순 투표 도구로 축소되지 않게 할 수 있을까요?

- **가설적 답**: 후보별 배민 링크와 대화를 기본으로 제공하고, 의견이 갈릴 때만 선택적으로 투표를 시작합니다.
- 투표 결과를 자동 확정으로 볼지는 미확정입니다.

## 발견과 참여

### Q-04. 어떻게 하면 새 계정을 만들지 않고 동료의 모집방을 빠르게 발견하고 참여할 수 있을까요?

- **가설적 답**: 주변 피어가 공개한 활성 방을 자동 발견하고 한 번의 동작으로 참여합니다.
- 닉네임, 사용자 식별과 회사 구성원 판별 방식은 미확정입니다.

### Q-05. 어떻게 하면 앱을 자주 열지 않아도 새로운 모집과 중요한 상태 변경을 놓치지 않을 수 있을까요?

- **가설적 답**: 검증된 macOS 진입점에서 활성 방을 빠르게 확인하고 모집, 마감과 주문 완료 같은 핵심 이벤트만 알립니다. MenuBar는 그 진입점의 후보입니다.
- 방별 음소거와 라운지 알림 기본값은 Policies에서 정해야 합니다.
- 중앙 서버가 없으므로 앱이 실행·연결되지 않은 동안 알림을 보장할 수 있는지 검증하고, 보장하지 못하는 상태를 사용자에게 명확히 알려야 합니다.

## 배민과 개인 메뉴

### Q-06. 어떻게 하면 배민 링크를 Mac에서 휴대폰으로 가장 짧게 전달할 수 있을까요?

- **가설적 답**: 가게 정보에 macOS 공유 버튼을 두고 AirDrop을 우선 후보로 검토합니다.
- 링크 복사와 QR을 대안으로 두고 실제 기기 테스트 후 기본 방식을 정합니다.

### Q-07. 어떻게 하면 복잡한 배민 메뉴와 옵션을 빠뜨리지 않고 각자 등록할 수 있을까요?

- **가설적 답**: 사람별 자유 입력에서 줄바꿈을 그대로 보존합니다.
- 자신의 메뉴를 별도 영역에서 수정하고 전체 참여자의 메뉴를 한눈에 확인합니다.

### Q-08. 어떻게 하면 LunchTime이 배민을 무리하게 대체하지 않고 두 제품의 경계를 자연스럽게 연결할 수 있을까요?

- **가설적 답**: LunchTime은 모집, 협의, 메뉴 취합, 상태와 히스토리를 담당하고 메뉴 탐색, 주문과 결제는 배민 앱에 남깁니다.
- 비공개 API나 스크래핑에는 의존하지 않습니다.

## 마감과 완료

### Q-09. 어떻게 하면 마감 시각에 신규 참여만 정확히 닫으면서 기존 참여자의 협의는 이어갈 수 있을까요?

- **방향 확정**: 방 생성 시 주문 마감 시각을 설정하고 해당 시각에는 신규 참여만 닫습니다. 기존 참여자의 방과 채팅은 유지합니다.
- 마감 후 메뉴 수정 가능 여부는 Policies에서 정해야 합니다.

### Q-10. 어떻게 하면 방 생성자에게 의존하지 않고 실제 주문을 완료 처리할 수 있을까요?

- **가설적 답**: 모든 참여자에게 주문 완료 동작을 제공하고 수행자와 시각을 시스템 이벤트로 남깁니다.
- 오완료 방지를 위한 확인과 되돌리기 정책은 미확정입니다.

## P2P 연속성

### Q-11. 어떻게 하면 P2P 방 생성자가 자리를 비워도 다른 참여자가 계속 협의할 수 있을까요?

- **가설적 답**: 방 상태와 메시지를 참여 피어에 복제하고 남은 피어가 동기화를 이어갑니다.
- 구현 가능성과 정본 선정 방식은 기술 검증 전까지 확정하지 않습니다.

### Q-12. 어떻게 하면 늦게 참여하거나 재접속한 사용자가 현재 맥락을 놓치지 않을 수 있을까요?

- **가설적 답**: 현재 상태 스냅샷과 필요한 범위의 채팅 이력을 연결된 피어로부터 받습니다.
- 전달할 이력 범위와 충돌 해결 방식은 미확정입니다.

### Q-13. 어떻게 하면 별도 서버 없이 완료된 주문 히스토리를 믿고 재사용할 수 있을까요?

- **가설적 답**: 완료된 방은 우선 각 참여자의 기기에 로컬 저장합니다.
- 기기별 히스토리 차이를 허용할지, 피어 간 병합할지는 검증 후 결정합니다.

## 반복 이용과 라운지

### Q-14. 어떻게 하면 과거 가게로 새 모집을 열면서 오래된 정보를 잘못 복사하지 않을 수 있을까요?

- **가설적 답**: 우선 가게명과 배민 링크만 새 방에 채우고 마감과 참여자는 새로 설정합니다.
- 이전 메뉴는 자동 복사하지 않고 참고 정보로 보여주는 안을 검증합니다.

### Q-15. 어떻게 하면 라운지 채팅이 점심방을 방해하거나 알림 소음을 만들지 않게 할 수 있을까요?

- **가설적 답**: 라운지를 점심방과 분리된 최상위 공간으로 두고 알림 설정도 별도로 관리합니다.
- 라운지의 보존 기간과 기본 알림은 미확정입니다.
