# 05. 어떻게 하면 질문

> 상태: 2026-07-24 회의 결정을 반영한 해결 질문
>
> 입력: [01. 제품 방향](./01_product_direction.md), [03. 문제와 유지할 강점](./03_pains_and_existing_strengths.md), [04. 가설과 검증 계획](./04_assumptions_and_validation.md)
>
> 역할: 확정된 제품 방향을 사용자 경험과 검증 가능한 구현 문제로 전환한다.

아래에서 **방향 확정**은 제품 경험을, **검증할 해법**은 기술 스파이크에서 선택할 구현 방식을 뜻합니다.

## 상태와 대화

### Q-01. 어떻게 하면 대화의 유연성을 유지하면서 현재 주문 상태를 바로 알 수 있을까요?

- **방향 확정**: Room 상단에 한 가게, 참여자, 마감, 사람별 메뉴 제출 상태와 주문 상태를 고정 요약하고 방별 채팅을 함께 둡니다.
- 참여·메뉴·마감·주문 변경은 채팅 본문과 구분되는 시스템 이벤트로 보여줍니다.

### Q-02. 어떻게 하면 여러 점심 모집이 동시에 있어도 대화와 상태가 섞이지 않게 할 수 있을까요?

- **방향 확정**: 가게 하나의 모집을 독립된 Room 하나로 만들고 MenuBar에서 오늘의 Room, 참여자 수와 상태 아이콘을 보여줍니다.
- 한 사용자는 같은 운영일에 하나의 Room에만 참여합니다. 다른 Room으로 이동하거나 새 Room을 만들려면 주문 상태가 `진행 중`일 때 기존 참여를 철회하며 자신의 활성 메뉴도 함께 무효화합니다.
- 분산 충돌로 여러 Room 참여가 동시에 생기면 자동 선택하지 않고 사용자가 유지할 Room을 고를 때까지 양쪽 주문 완료를 막습니다. 선택하지 않은 Room의 참여와 활성 메뉴는 철회 이벤트로 함께 정리합니다.

### Q-03. 어떻게 하면 가게 투표 없이 주문할 사람을 모을 수 있을까요?

- **방향 확정**: Room을 만들 때 가게를 하나 정하고 사용자는 그 가게 주문에 `참여`합니다.
- 후보 목록, 투표와 최종 가게 선정 단계를 두지 않습니다.
- 한 명만 참여한 Room도 정상적인 주문 흐름입니다.

## 발견과 참여

### Q-04. 어떻게 하면 중앙 계정 서버 없이 동료의 Room을 발견하고 안전하게 참여할 수 있을까요?

- **방향 확정**: 사내 두 Wi-Fi가 연결되는 내부망에서 발견한 Peer를 자동 신뢰하고 오늘의 활성 Room을 발견합니다.
- 자동 신뢰된 Peer는 활성 Room의 메뉴·채팅을 읽고 참여자만 쓸 수 있습니다.
- **검증할 해법**: Bonjour/mDNS 등 발견 방식, 자동 신뢰 네트워크 경계와 암호화 세션을 실제 사내망에서 검증합니다. 악의적인 내부 Peer가 평문을 읽을 위험은 수용합니다.

### Q-05. 어떻게 하면 상세 창을 계속 열어두지 않고 오늘의 상태를 확인할 수 있을까요?

- **방향 확정**: MenuBar를 기본 진입점으로 사용하고, 오늘의 Room, 참여자와 진행·완료·동기화 주의 상태를 요약합니다.
- 14:30에는 당일 Room을 MenuBar에서 제거하고 종료 상세·finalization 상태는 별도 Window에서 봅니다.
- 채팅과 메뉴 입력은 별도 앱 창에서 수행합니다.
- 라운지 바로가기와 읽지 않음 표시는 후속 범위입니다.
- 잠자기·전원 꺼짐 중 실시간 알림은 보장하지 않으며 복귀 후 catch-up 상태를 보여줍니다.

## 배민과 개인 메뉴

### Q-06. 어떻게 하면 배민 링크를 Mac에서 휴대폰으로 짧고 일관되게 전달할 수 있을까요?

- **방향 확정**: MVP에서는 Room의 입력 링크 문자열을 검증·정규화하지 않고 그대로 로컬 QR로 만들어 Mac 화면에 표시합니다.
- AirDrop, 시스템 공유 시트와 자동 기기 전달은 MVP에서 제외합니다.
- 링크 형식 경고나 차단을 제공하지 않습니다.

### Q-07. 어떻게 하면 복잡한 메뉴와 옵션을 빠뜨리지 않고 각자 등록할 수 있을까요?

- **방향 확정**: `메뉴명`과 반복 가능한 `옵션` 행으로 입력 구조를 고정합니다. 추가 금액은 별도 계산 필드가 아니라 옵션 문자열의 일부로 보존합니다.
- 참여자별 메뉴를 주문용 목록으로 모으고 각 메뉴의 마지막 변경 시각과 동기화 상태를 함께 보여줍니다.
- 자유 형식 여러 줄 텍스트 한 덩어리는 MVP 입력 모델로 사용하지 않습니다.

### Q-08. 어떻게 하면 LunchTime이 배민을 무리하게 대체하지 않고 두 제품의 경계를 연결할 수 있을까요?

- **방향 확정**: LunchTime은 모집, 참여, 협의, 메뉴 취합, 주문 상태와 재사용 히스토리를 담당합니다.
- 메뉴 탐색, 실제 주문과 결제는 QR 이후의 배민 앱에서 수행합니다.
- 비공개 API와 스크래핑에 의존하지 않습니다.

## 마감과 완료

### Q-09. 어떻게 하면 방별 주문 마감과 하루의 최종 종료를 혼동하지 않을 수 있을까요?

- **방향 확정**: 방 생성 시 14:30 미만의 주문 마감 시각을 설정하고, 해당 시각에는 ACK 없는 참여 요청을 실패 처리하며 신규 참여를 닫습니다.
- 주문 상태가 진행 중이고 기존 마감 전이면 현재 참여 중인 생성자만 마감 시각을 바꿀 수 있습니다. 주문 완료 상태에서는 잠기고, 도달한 마감은 뒤로 미뤄 다시 열 수 없습니다.
- 기존 참여자는 주문 상태가 진행 중인 동안 14:30 전까지 채팅과 자신의 메뉴 수정을 계속할 수 있습니다.
- 14:30에는 모든 Room과 라운지의 쓰기를 닫아 현재 메모리를 열람 전용으로 전환하고, MenuBar에서 당일 Room을 제거한 뒤 최대 120초 finalization을 수행합니다.

### Q-10. 어떻게 하면 메뉴 누락 없이 주문 완료를 표시하고 주문 취소도 처리할 수 있을까요?

- **방향 확정**: 참여자 누구나 주문 완료를 표시하고 14:30 전에는 되돌릴 수 있습니다. 주문 완료는 최종 종료가 아닙니다.
- **fail-closed 계약**: 참여는 마감 전 승인 ACK가 필요하고, 주문 완료 기기는 모든 현재 참여자의 최신 승인·메뉴 ACK 리비전을 보유해야 합니다.
- 과거·오프라인 Peer의 미응답만으로는 완료를 막지 않습니다. 로컬 단독 예외는 생성자가 처음부터 만들고 한 번도 다른 Peer에 공유하지 않은 Room에만 적용합니다.
- 현재 참여자의 메뉴가 없거나 최신 리비전이 `이 기기에 저장됨`, `확인 필요`이면 주문 완료를 허용하지 않습니다.
- 주문 완료·되돌리기의 수행자와 시각을 변경 이벤트로 남깁니다.

## P2P 연속성

### Q-11. 어떻게 하면 생성자나 특정 호스트 없이 Room을 이어갈 수 있을까요?

- **방향 확정**: 각 Peer가 고유 변경 이벤트와 현재 상태를 로컬에 보유하고 연결된 Peer끼리 차이를 교환합니다.
- 한 Peer가 잠자기·종료돼도 다른 Peer가 데이터를 보유하면 Room을 계속 이용합니다.
- 마지막 참여자 철회는 Room을 취소합니다. 현재 참여 중인 생성자는 진행 중 Room을 전체 취소할 수 있고 주문 완료 상태라면 먼저 완료를 되돌립니다. 생성자 철회 뒤에는 관리자 권한이 사라져 가게·입력 링크·마감 메타데이터가 잠깁니다.
- **검증할 해법**: 변경 로그, CRDT 등 복제 자료구조와 충돌 규칙은 기술 스파이크로 정합니다.

### Q-12. 어떻게 하면 잠자기·재접속 후 놓친 정보를 복구할 수 있을까요?

- **방향 확정**: 깨우기, 앱 전면 전환, 네트워크 변경, 의미 있는 새 Peer와 수동 요청마다 최대 3회이면서 총 30초 이내인 동기화 세션을 시작하고 둘 중 먼저 도달하면 종료합니다.
- 11:00~14:30에 활성 Room·라운지 데이터 또는 미해결 동기화 데이터와 건강한 Peer가 있을 때만 30초 간격으로 anti-entropy 대조합니다. 대상이 없거나 14:30이면 멈추고, 제한을 소진한 실패 Peer는 정기 타이머만으로 재시도하지 않습니다.
- 지속 상태는 `이 기기에 저장됨/복제됨/동기화됨/확인 필요`이고 `동기화 중`은 별도 활동입니다. 실패한 세션은 새 의미 있는 트리거나 수동 요청 전까지 반복하지 않습니다.
- Connection / Settings에는 Peer 닉네임, 온라인 여부, 마지막 확인 시각과 동기화 상태만 표시합니다. 중복 닉네임에는 짧은 기기 ID를 붙이고 신뢰·ACK 관리는 노출하지 않습니다.
- 모든 Peer가 데이터를 삭제했다면 복구를 보장하지 않습니다.

### Q-13. 어떻게 하면 운영 데이터를 필요한 만큼만 남기면서 복구할 수 있을까요?

- **방향 확정**: Room·참여·메뉴·상태와 동기화 이벤트는 최대 14일 관리합니다.
- 채팅은 고유 메시지 ID로 중복 제거하고 best-effort 순서로 교환하며 프로세스 메모리에만 둡니다. 재실행 시 살아 있는 Peer의 당일 채팅은 다시 받을 수 있지만 완전 복구나 전역 순서는 보장하지 않습니다.
- 삭제 표식 만료 뒤 비핵심 고아 데이터가 남는 것은 허용하되, 지난 일일 세션을 활성 상태로 되살리지 않습니다.
- 불완전 finalization은 성공 히스토리를 만들지 않지만 확인된 상세는 남깁니다. 이후 제한된 재동기화가 final snapshot과 히스토리를 다시 계산할 수 있습니다.
- 최대 14일 종료 상세는 모든 자동 신뢰 회사 WiFi Peer가 읽을 수 있고, 취소 Room은 상세와 재사용 히스토리에서 제외합니다.

## 반복 이용과 라운지

### Q-14. 어떻게 하면 과거 가게로 새 Room을 열면서 오래된 주문 정보를 잘못 복사하지 않을 수 있을까요?

- **방향 확정**: 가게명, 입력 링크, 최근 주문 완료 시각과 로컬 이용 횟수만 최소 히스토리로 장기 보존합니다.
- 같은 가게는 `trim(storeName) + raw inputLink` 완전 일치로만 합치며 입력 링크는 trim·URL 정규화·redirect 추적을 하지 않습니다.
- `다시 모집`은 가게명과 링크가 채워진 생성 화면을 열고 사용자가 주문 마감 시각을 확인하게 하며 즉시 Room을 만들지 않습니다.
- 이전 참여자, 개인 메뉴와 채팅은 복사하지 않습니다.
- 신뢰 Peer와 최소 가게 히스토리를 병합할 수 있지만 모든 사용자의 목록이 같다고 보장하지 않습니다.

### Q-15. 어떻게 하면 라운지를 점심 주문 흐름과 섞지 않고 MVP에 포함할 수 있을까요?

- **방향 확정**: 정식 명칭은 `라운지`이며 별도 앱 창의 최상위 공간으로 제공합니다.
- 11:00~14:30에 대화하고 14:30 이후에는 현재 세션의 내용을 읽기만 합니다.
- 라운지 채팅도 앱 종료 시 삭제합니다.
- MenuBar 바로가기, 읽지 않음 배지와 메시지 알림은 후속 기능입니다.

## 보안

### Q-16. 어떻게 하면 같은 내부망에서 데이터 탈취와 위조를 줄일 수 있을까요?

- **방향 확정**: Peer 간 응용 데이터는 암호화 채널로 전송하고, 운영 이벤트와 히스토리 등 디스크 데이터도 암호화합니다.
- 키는 macOS Keychain으로 보호하고 지원 회사 WiFi에서 발견한 Peer를 자동 신뢰합니다.
- **검증할 해법**: 자동 신뢰 네트워크 경계, 암호화 세션과 악의적인 내부 Peer 위험은 보안 스파이크에서 검증합니다.

### Q-17. 어떻게 하면 기기 시계 차이 때문에 마감과 종료가 잘못 판정되지 않게 할 수 있을까요?

- **방향 확정**: Peer 간 시계 차이를 확인하고 허용 범위를 넘으면 참여 승인, 마감 변경과 주문 상태 변경처럼 시간에 의존하는 사용자 쓰기를 차단합니다. 14:30 쓰기 종료는 안전하게 적용하되 종료 스냅샷의 유효성·완전성 판정은 `확인 필요`로 둡니다.
- **검증할 해법**: 정상적인 사내 Mac의 편차를 수용하는 넉넉한 허용 오차는 기술 스파이크에서 정합니다.
